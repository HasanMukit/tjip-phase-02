API design best practices

Eleven habits that separate an API a client wants to keep using from one a client merely tolerates — naming, idempotency, versioning, pagination, query strings, predictable responses, security, simple references, async handoffs, range requests, OpenAPI docs — plus the rate-limiting plan most teams ship too late.

An API is the part of your system that's hardest to change once it's in the wild. Internal code can be refactored on a Friday afternoon; a public field name lives until every client has migrated off it, which in practice means forever. The choices that look stylistic at design time — singular or plural, header or query string, v1 in the URL or in a header — turn into commitments the moment the first client integrates. The cost of getting them right up front is small. The cost of getting them wrong is paid in deprecation timelines.

This page collects eleven habits that consistently separate APIs people enjoy integrating with from APIs people complain about, plus a bonus tip on rate limiting that bites every public API the moment it gets popular. They're orthogonal — you can adopt them in any order — but they reinforce each other: a clearly named, properly versioned, idempotent, paginated, well-secured, well-documented API with predictable response shapes and a real plan for long-running work is one whose contract a client can read once and trust.

Each tip points back to the page where the underlying mechanic is explained in depth. If a tip refers to retries, it's because making POST safe to retry is a topic of its own. If it refers to cursor pagination, the math and trade-offs are on the pagination page. This page is the playbook; the others are the proofs.

00Why "best practice" is a load-bearing word here

"Best practice" usually means "convention worth following because the alternative is mildly worse." On an API, it means something stronger: every one of these habits has a corresponding production failure mode the moment you skip it. Bad naming creates clients that hard-code typos. Non-idempotent POSTs double-charge customers when retries fire. No versioning strategy means breaking clients on every release. No pagination means a 5 GB response that crashes the mobile app. Sensitive data in query strings ends up in nginx logs forever. Deeply nested URLs lock you into a hierarchy that next quarter's product manager will want to break.

Every "best practice" below is a real outage someone else has already paid for. Treat them as the cheapest insurance you'll ever buy.

01Use clear, consistent naming

The URL is the most public part of the contract. Every client integration starts with a developer reading a path and forming an expectation about what it returns. A path that follows conventions is read once; a path that doesn't is read three times and still gets confused with another endpoint a year later.

Use nouns, not verbs

The verb is already in the HTTP method. Repeating it in the path is redundant and breaks the pattern that makes REST legible.

Don't

GET /getUsers POST /createUser POST /deleteUser/123 GET /fetchUserOrders?id=123 POST /updateUserEmail

The verb is duplicated, the case style is inconsistent, and POST /deleteUser hides the actual semantics behind the wrong method.

Do

GET /users POST /users DELETE /users/123 GET /users/123/orders PATCH /users/123

The path names a resource, the verb describes the action. A new client can guess GET /products, DELETE /products/45 without consulting the docs.

Singular vs plural — the rule that's actually a rule

The convention is simple and almost universally followed by APIs people don't complain about:

  • Plural for collections — /users is the set of all users. GET /users lists them. POST /users creates a new one.
  • Plural + ID for an instance — /users/123 is one specific user. GET /users/123 reads it; DELETE /users/123 removes it. The plural is kept because 123 is one element of users, not a different resource.
  • Singular only for genuine singletons — /me, /health, /config. There's exactly one. A plural would be a lie.
GET /users # list users (plural collection) GET /users/123 # one user (still plural — 123 is in the set) GET /users/123/orders # that user's orders (plural sub-collection) GET /users/123/orders/45 # one of their orders GET /me # the authenticated caller — a singleton, so singular

Mixing styles (/user for one, /users for many) doubles the surface a client has to remember and breaks every URL-completion tool that assumes the plural is canonical.

Be consistent — pick a case style and never switch

Pick one and apply it everywhere: kebab-case for paths (/payment-methods, not /paymentMethods or /payment_methods) and snake_case or camelCase for JSON fields (created_at vs createdAt). The which matters less than the consistency. Mixing styles forces the client to remember the spelling of every endpoint individually instead of one rule.

Names you don't get to fix
Public path names and field names are part of the contract. Once a paying customer integrates, renaming created_at to createdAt breaks them. The cheap fix at design time becomes a multi-quarter migration the moment you ship. Spend the extra five minutes picking a name you're willing to keep.

02Ensure reliability through idempotent endpoints

Networks fail. Retries are not optional — every modern client framework retries automatically on 5xx and connection errors, often without the developer realising. The question is whether your API survives the retries or compounds them into duplicate state.

The HTTP spec is explicit about which methods are idempotent by default. Calling them once and calling them five times leave the server in the same state:

Verb Idempotent by default? What that buys you
GET yes Safe to retry, cache, prefetch, hit speculatively.
PUT yes Replace-with-X applied twice is the same as once.
DELETE yes Deleting an already-deleted thing is a no-op (or a 404 — still safe).
POST no Two POST /orders calls create two orders unless you intervene.
PATCH depends JSON Merge Patch: yes. JSON Patch with add ops: no.

Honour the spec. If your PUT /users/123 creates a new user instead of replacing one, or your DELETE errors on the second call, retrying clients will see incorrect behaviour they have no way to predict. Idempotency isn't a property you opt into per endpoint; it's a contract the verb already promises.

For the operations that aren't idempotent by default — chiefly POST and side-effectful PATCH — make them idempotent on purpose. The two patterns are covered in detail on the main API design page:

Idempotency-Key header

Client generates a UUID, server stores the full response under it, retries replay the stored answer. General-purpose; works for any operation. Stripe popularised it.

Natural-key dedup

The request already carries something unique (order_id, (user_id, day, type)). A unique constraint plus an upsert dedupes at the storage layer with no extra header.

Idempotency is a precondition, not a feature
Every resilience pattern that involves retries — exponential backoff, retry budgets, circuit-breaker probes, hedged requests — assumes the underlying operation is idempotent. If it isn't, retries don't make the system more reliable; they make it more broken, faster.

03Plan for versioning from day one

Every API will eventually need a breaking change. A field renamed for clarity, a response shape simplified, a status code changed from 200 to 202 because the operation went async. The question isn't whether you'll need a v2 — it's whether you've already chosen how v1 and v2 will coexist.

Three common strategies, in rough order of how often they're picked:

URL versioning — the obvious one

GET /v1/users/123 GET /v2/users/123

Clients pick a version explicitly in the path. New version is a new tree. Easy to implement, easy to debug (the URL tells you everything), easy to deprecate (turn off the old subtree). The downside: every URL has a version segment forever. Most successful public APIs (Stripe, GitHub for early versions, Twilio) use this.

Header versioning

GET /users/123 Accept: application/vnd.acme.v2+json # or API-Version: 2

The path stays clean; the version moves into a header. URLs are stable forever. The downside: invisible from the URL — testing with curl needs an extra flag, CDN caching has to vary on the header, and an unset header has to default somewhere (which is its own minor footgun). GitHub's modern API uses this; Stripe uses a date-based variant (Stripe-Version: 2024-04-01).

Query-parameter versioning

GET /users/123?version=2

Possible, but rarely the right call. Query parameters are usually for filtering/sorting, and putting versioning there muddies that meaning. If you need it, header versioning is almost always the cleaner version of the same idea.

Communicate deprecation

When a version is going away, say so in the response, not just in a blog post. Two HTTP headers exist for exactly this:

Deprecation: true Sunset: Wed, 31 Dec 2026 23:59:59 GMT Link: <https://api.example.com/v2/users/123>; rel="successor-version"

Clients that monitor headers can surface the warning before the cutover; observability dashboards can count how many requests are still hitting the deprecated path. Cold deprecations (a blog post and silence on the wire) leave you guessing.

Keep at least one previous version alive

When a new version ships, the old one stays available — typically for at least one full release cycle, often a year for paid public APIs. The point isn't generosity; it's that every integrated client has its own deploy pipeline, and forcing them to migrate on your schedule is how you turn a routine release into a partner incident. Pair the overlap window with a stated Sunset date and visible deprecation traffic so the cutover lands when the dashboards say it can.

"What counts as a breaking change?"
Adding a field is non-breaking — old clients ignore it. Removing or renaming a field is breaking — old clients read the missing key as null. Changing a type (string to number, scalar to array) is always breaking. Tightening validation can be breaking even if the schema looks the same — a field that used to accept empty strings no longer doing so will surprise someone. Treat additive change as free; treat anything else as a v-bump.

04Always paginate list endpoints

A list endpoint is a small lie — GET /posts looks like one operation, but the result set might be ten rows or ten million. Returning everything in one response burns memory on both sides, blows past the bandwidth budget, and pushes time-to-first-byte into the seconds. Even if today's table is tiny, adopt the pagination contract from the first version. Adding pagination later is a breaking change; baking it in is free.

Two main styles, with very different cost models. Both are covered with figures and SQL on the pagination page; the summary here is enough to pick.

Offset / limit

GET /posts?limit=20&offset=40

Maps cleanly to "Page 3 of 50" UIs. Two failure modes: O(offset) cost at deep pages, and inconsistency when writes happen mid-walk (rows shift, duplicates and gaps appear).

Reach for it when the dataset is small, bounded, and the UX needs random-access page numbers.

Cursor

GET /posts?limit=20&cursor=eyJpZCI6MTIzfQ

"Give me what comes after this row." O(log N) per page, stable under inserts and deletes, natural fit for infinite-scroll. Costs: forward-only, needs a stable sort key.

Reach for it when the dataset grows unbounded or live writes happen while users paginate. The default for feeds, search, audit logs.

If your dataset can have a million rows and users routinely scroll past the first thousand, you want cursor — and even if it never gets that big, the contract costs nothing extra.

05Clear query strings — filter, sort, search, partial responses

Once you've named the resource correctly and paginated the list, the next question every client asks is "can I narrow it down?" That's what query strings exist for. Four common needs — filtering, sorting, search, and asking for a subset of fields — each have a convention worth following.

Filter and sort — separate parameters, separate purposes

GET /products?category=shoes&color=red&in_stock=true GET /products?sort=price&order=asc GET /orders?status=pending&created_after=2026-01-01

A few rules that compound nicely:

  • One field, one parameter. ?status=pending, not ?filter=status:pending. Each filter should be a top-level key, not a mini-query-language inside a value. Half of "GraphQL was a mistake" stories are really "we built a query language inside our query string".
  • Sort gets two knobs, not one. ?sort=price&order=asc is clearer than ?sort=-price. The clever-trick form (a leading - for descending) saves three characters and confuses every junior engineer who joins the team.
  • Use the same parameter names everywhere. If created_after means "ISO timestamp, exclusive lower bound" on one endpoint, it should mean exactly that on every other endpoint that takes it. The contract is the predictability, not the keyword.

Search — a different verb than filter

Filtering excludes by predicate ("only orders with status=pending"). Searching ranks by relevance ("find anything that looks like 'running shoes'"). They're different operations and deserve different conventions:

GET /products?q=running+shoes # search within /products GET /search?q=running+shoes # global search across resources GET /search?q=running+shoes&scope=products,brands # scoped global search
  • Use q as the shorthand for the query string — convention since at least early Google. Pick q or search and stick with it; don't mix.
  • Separate terms with + (URL-encoded space). Multi-word queries shouldn't need their own escaping rules.
  • For multi-resource search, expose a top-level /search endpoint and let the client narrow the scope. This is the one place where a verb in the URL is fine — search is an action, not a resource.

Partial responses — let clients ask for less

Sometimes a client only needs three fields out of thirty. REST endpoints return what they return, which is the over-fetching complaint that drives teams to GraphQL. The cheaper fix: a fields parameter that names the fields the client wants.

GET /users/123?fields=id,name,email GET /products?fields=id,name,price&limit=20
  • Comma-separated list of field names — the standard shape. Stripe, GitHub, JSON:API all converge on this with minor syntactic variations.
  • Skip-validate aggressively. Unknown field names should error, not silently drop — typos shouldn't return the full payload.
  • Pair with ?expand=. fields trims down the default shape; expand opts into nested objects (see "Keep cross-resource references simple" below). Together they give clients control in both directions without committing you to a query language.
Heads-up: query strings and caching
Some older HTTP proxies and CDNs strip or ignore query parameters when computing cache keys — meaning /products?category=shoes and /products?category=hats can collapse into one cached entry on a misconfigured intermediary. Modern CDNs (Cloudflare, Fastly, CloudFront) handle this correctly, but it's a footgun in older infra. The fix is configuration-level, not contract-level — but worth knowing the trap exists when "why is this user seeing the wrong list?" surfaces in production.

Combine cleanly with pagination

Filters, sort, fields, and pagination should layer without surprising interactions. The cursor encodes "where I left off in the filtered, sorted result"; new filter values invalidate the cursor (which is fine — the client just starts over).

GET /orders?status=pending&sort=created_at&order=desc&limit=20 GET /orders?status=pending&sort=created_at&order=desc&limit=20&fields=id,total,created_at&cursor=eyJ0c19c...

06Make responses and errors predictable

The shape of a successful response is half the contract; the shape of an error response is the other half. Clients write conditionals against both. Picking — and documenting — a consistent shape early means every endpoint feels like part of the same API instead of a federation of micro-services that each invented their own rules.

Content negotiation — let the client ask, but pick a default

The HTTP Accept header is exactly the mechanism for "I'd like the response in format X". A well-behaved API honours it:

GET /users/123 HTTP/1.1 Accept: application/json # server responds with JSON GET /users/123 HTTP/1.1 Accept: application/xml # server responds with XML if supported, or 406 Not Acceptable

Two rules that handle 95% of cases:

  • Default to JSON when the client doesn't specify a preference. JSON is the lingua franca of public APIs in 2026; clients without an explicit Accept header almost always want it.
  • Document every supported format. If you don't support XML, say so — and return 406 Not Acceptable when a client asks for it, not a silent JSON fallback that confuses parsers expecting one thing and getting another.

Errors should come back in the same format as the rest of the API. A JSON endpoint that returns an HTML stack trace on 500 is a contract violation in itself — clients can't parse it, monitoring tools can't aggregate it, and the on-call engineer won't even see it in the dashboard.

Error response shape — code, message, hint, link

HTTP status codes carry the high-level verdict; the body carries the detail. Both belong in the contract.

Three categories cover almost everything:

  • 2xx — success. The request worked. 200 OK for most reads and updates, 201 Created for new resources (with a Location header pointing at the new URL), 204 No Content when there's nothing to return.
  • 4xx — the client did something wrong. 400 Bad Request (malformed), 401 Unauthorized (no/bad credentials), 403 Forbidden (authenticated but not allowed), 404 Not Found, 409 Conflict, 422 Unprocessable Entity (validation failed), 429 Too Many Requests.
  • 5xx — the server failed. 500 for "we don't know", 502/503/504 for upstream / unavailable / timeout. The client did nothing wrong; the right behaviour is usually a retry (assuming idempotency).
Pick a small set and use them consistently
Eight status codes covers most APIs (200, 201, 204, 400, 401, 403, 404, 500); a streamlined set of three (200, 4xx, 5xx) is also defensible for very simple APIs. The point isn't which codes — it's that the same code means the same thing on every endpoint. Inventing 418 I'm a teapot for "user is rate-limited" is funny exactly once.

Inside the body, return enough for both humans and machines to act on the error. The pattern most public APIs converge on (and that RFC 7807 Problem Details standardised):

HTTP/1.1 422 Unprocessable Entity Content-Type: application/problem+json { "type": "https://api.example.com/errors/invalid-email", "title": "Invalid email address", "status": 422, "code": "validation.email_invalid", "detail": "The 'email' field must be a valid RFC 5322 address.", "hint": "Common cause: missing '@' or trailing whitespace.", "link": "https://docs.example.com/errors/invalid-email", "trace_id": "7f8a3c12-9e4b-4d1a-b2e7-1c0f8d9a4e22" }
  • code — a stable machine-readable identifier (validation.email_invalid). Clients write conditionals against this, not against title. Never reuse a code for a different meaning; never localise it.
  • title / detail — short and long human-readable messages. Localise detail if you support i18n.
  • hint — a one-line nudge toward the fix. Almost every developer-facing API benefits from this; it's the difference between "validation failed" and "validation failed; check that the email contains an '@'".
  • link — a URL to the error's page in the docs. Doubles as a stable identifier the support team can refer back to.
  • trace_id — the server's correlation ID for this request. When a customer reports "I got error X at 14:32", you can find the exact log line without grepping.

Use the same envelope on every endpoint. A 401 from the auth service and a 422 from the orders service should look like the same JSON shape — the client's error-handling code is written once, not per service.

Helpful to developers, useless to attackers
Errors should explain enough for a legitimate developer to fix the call, never enough for an attacker to map your internals. Don't return raw stack traces, SQL fragments, ORM class names, internal hostnames, file paths, or which row of a JOIN failed. Do return a stable code, a generic title, a trace_id the support team can correlate against logs, and — for 401/404 on protected resources — the same message whether the resource doesn't exist or the caller isn't allowed to see it. Distinguishing the two leaks the existence of resources to unauthorised callers.

07Don't make security an afterthought

Security retrofits are expensive, brittle, and almost always incomplete. Three habits, baked in from day one, prevent most of the common API breaches.

Sensitive data goes in headers, not query strings

Query strings are the worst place to put anything sensitive. They end up in:

  • Browser history — the user's machine remembers ?token=... forever.
  • Server access logs — nginx and most proxies log the full URL by default. Tokens and API keys land in plaintext on disk.
  • The Referer header — when the page links out, the next site sees the URL.
  • Analytics and APM tools — anything that captures URLs captures the secret in the URL.

Don't

GET /orders?api_key=sk_live_abc123 GET /users?password=hunter2

Do

GET /orders Authorization: Bearer sk_live_abc123 POST /login Content-Type: application/json {"password": "hunter2"}

Headers and request bodies aren't logged by default in most stacks. They're still encrypted on the wire (see below). The bar is "would I be comfortable if this string showed up in a Splunk dashboard?" — and for tokens, the answer is no.

TLS everywhere — no exceptions for "internal"

Every modern browser refuses plain HTTP for anything serious; treat your own services the same way. TLS gives you three things HTTP doesn't — encryption, integrity, authentication — and the cost is one round-trip per fresh connection (or zero with TLS 1.3 session resumption). The HTTPS handshake page walks the math; the practical version is: redirect HTTP to HTTPS, set Strict-Transport-Security, and refuse plaintext in the API gateway. Internal service-to-service traffic should also use TLS — "the internal network is trusted" stops being true the first time someone pivots through a compromised service.

Use OAuth 2.0 (and OIDC) — don't invent your own auth flow

For anything beyond a personal API key, the answer to "how should clients authenticate?" is OAuth 2.0. It's the standard authorisation framework for APIs in 2026 — supported by every major identity provider, every modern HTTP library, every API gateway. The grant types map to real client shapes (Authorization Code + PKCE for web/mobile, Client Credentials for service-to-service, Device Code for TVs and CLIs), and the token format (JWT or opaque) plugs into the verification step below. Layer OpenID Connect on top when you also need to identify the user rather than just authorise the call.

Home-grown auth schemes — custom signed cookies, hand-rolled HMAC headers, "just send the password on every request" — are where last quarter's CVE came from. The library exists; use it.

Verify keys and tokens on every request

Statelessness is one of REST's superpowers, but it has a corollary: the server can't trust anything about the caller from one request to the next unless the caller proves it again. So check the credential on every request, not just at session start.

  • Verify the signature on every JWT. A token that's "probably still valid because we issued it five minutes ago" is exactly the kind of assumption attackers exploit.
  • Check expiry, scope, and audience — a token signed correctly but with the wrong audience or expired five minutes ago is no good.
  • Authorize the action, not just the identity — knowing who is calling isn't the same as knowing they're allowed to do this thing to this resource. Authorization is per request, per resource.
  • Don't roll your own crypto. Use a library that's been hardened (jose, libsodium, your platform's standard crypto). The home-grown HMAC checker is where last quarter's CVE came from.
Where this lives in the request path
Most of this is the API gateway's job. Authentication runs first, rate limiting second, then the request gets routed to a backend that already trusts the gateway's verdict. Putting auth deeper means every backend re-implements it; putting it at the gateway means one place hardens it.

08Keep cross-resource references simple

The temptation when designing URLs is to encode every relationship in the path: /users/123/orders/45/items/9/refunds/2. It feels neat — every slash mirrors the database join. It also locks you into a hierarchy that the product will outgrow within a year, and forces every client to know the entire tree just to reach a leaf.

There's a security angle too: a URL that reads like a schema diagram is a schema diagram. /users/123/orders/45 tells anyone watching that user 123 has at least 45 orders, that orders are owned by users, and that internal IDs are sequential integers. Mirroring your database structure in your URLs leaks information you didn't have to leak; flat resources by ID (/orders/45) reveal less and let you renumber, partition, or reparent later without breaking every client URL.

One level of nesting is plenty

Don't — deep nesting

GET /users/123/orders/45/items/9/refunds/2 PATCH /companies/4/teams/2/members/8/permissions/3

Every URL leaks the entire path of ownership. Move an item between orders? URL changes. Reparent a team? Every saved URL breaks. Clients have to walk the tree to know IDs they shouldn't need.

Do — flat with explicit IDs

GET /refunds/2 GET /refunds?order_id=45 GET /permissions/3 GET /permissions?member_id=8

Each resource has one canonical URL by ID. Relationships are query parameters when listing, fields in the body when reading. Reparenting changes a column, not a URL.

Use IDs, not full embedded objects, by default

When one resource references another, return the ID and let the client fetch the related resource if it needs to:

// GET /orders/45 { "id": 45, "user_id": 123, "total": 5000, "items": [9, 10, 11] // IDs, not full item objects }

Reasons:

  • Response size stays predictable. An order doesn't accidentally drag 50 items, each with 30 fields, into a payload meant to be a list summary.
  • Cache layers work. /items/9 can be cached independently and served from a CDN; an embedded copy can't.
  • Updates don't fan out. If an item changes, you don't have to invalidate every order that embedded it.
  • Clients pick the depth they need. A list view fetches summaries; a detail view follows the IDs. This is the same problem GraphQL solves, but you get most of the benefit by not over-embedding in the first place.

Optionally support an ?expand=items parameter for clients that genuinely need the join in one round-trip. Stripe's expand, GitHub's sparse fieldsets, JSON:API's include are all variations of the same idea: flat by default, expandable on demand.

09Hand long operations off asynchronously

Some operations don't fit in a request/response cycle: video transcodes, bulk imports, report generation, ML inference jobs that take minutes. Holding the HTTP connection open while they run wastes server threads, blows past gateway timeouts (typically 30–60 seconds), and gives the client no way to recover if the connection drops mid-flight. The pattern that solves all three problems at once is the async 202 handoff.

The handshake — accept now, do later, expose progress

Three steps, three response codes:

  1. Client kicks off the work with a normal POST. The server validates the request, enqueues the job (or starts a background worker), and returns immediately:
    POST /reports HTTP/1.1 Content-Type: application/json {"format": "csv", "range": "2026-Q1", ...} → HTTP/1.1 202 Accepted Location: /jobs/abc-123 Retry-After: 30 {"job_id": "abc-123", "status": "queued"}
    The 202 means "I've taken responsibility for this; check back later." The Location header points at the status endpoint. Retry-After hints at how long to wait before polling.
  2. Client polls the status endpoint until the job finishes:
    GET /jobs/abc-123 → HTTP/1.1 200 OK {"job_id": "abc-123", "status": "running", "progress": 0.42}
    Status values are usually queued, running, succeeded, failed, cancelled. Optional progress, started_at, estimated_completion fields if the work is long enough to warrant a progress bar.
  3. On completion, the status endpoint redirects to the resulting resource — for create operations, the conventional response is 303 See Other:
    GET /jobs/abc-123 → HTTP/1.1 303 See Other Location: /reports/r-789 # Client follows: GET /reports/r-789 → HTTP/1.1 200 OK {report payload}
    303 specifically means "the result is somewhere else, fetch it with GET" — separate from 301 (permanent move) and 302 (generic redirect). Use it when an async operation creates a resource and you want to point clients at it.

Webhooks as the push complement to polling

Polling is fine for jobs measured in seconds; for jobs measured in minutes or hours, let the client register a webhook URL on the original POST. When the job finishes, the server fires a POST at the webhook URL with the result reference. Polling stays available as a fallback so clients without inbound webhook endpoints (mobile, CLI tools) aren't locked out — and so the client can recover state after a crash without waiting for the next webhook.

Why this beats "just hold the connection open"
A 30-second LB timeout is the floor — most production gateways are tighter. A 5-minute job over a held connection breaks at every layer (browser, CDN, gateway, load balancer, and the user's own laptop closing the lid). The 202 + status pattern survives all of that: the client can crash, reconnect, refresh the page, switch networks — and pick up where it left off as long as it remembers the job ID. It's the same idempotency lesson from §02, applied to time instead of retries.

10Support range requests for large resources

When a single resource is genuinely large — a video file, a 4K image, a multi-gigabyte export — the right pattern isn't pagination (that's for collections). It's HTTP range requests: the client asks for byte offsets, the server hands back the slice they asked for, and the response is treated as part of a stream the client is assembling.

How it works

The server advertises range support on the regular GET response:

GET /videos/keynote.mp4 → HTTP/1.1 200 OK Accept-Ranges: bytes Content-Length: 524288000 # ~500 MB

Once the client knows ranges are supported, it asks for slices with a Range header:

GET /videos/keynote.mp4 Range: bytes=0-1048575 # first 1 MiB → HTTP/1.1 206 Partial Content Content-Range: bytes 0-1048575/524288000 Content-Length: 1048576 <the first 1 MiB of bytes>

The status code is 206 Partial Content, distinct from 200 — it tells the client "this isn't the whole resource, I'm honouring the range you asked for." The Content-Range header echoes what was returned plus the total size, so the client knows when it's seen the last byte.

HEAD to probe before downloading

A HEAD request returns the same headers as GET but no body — which is exactly what a download manager wants when it's deciding how to chunk a file:

HEAD /videos/keynote.mp4 → HTTP/1.1 200 OK Accept-Ranges: bytes Content-Length: 524288000 ETag: "abc123"

The client now knows the total size and that ranges are supported, can plan its download (e.g. four parallel 125 MB ranges), and uses the ETag to detect if the file changes mid-download.

What ranges buy you
Resumable downloads — a connection drop at byte 200 MB out of 500 MB doesn't restart from zero; the client resumes with Range: bytes=209715200-. Parallel chunking — multiple ranges fetched concurrently can saturate bandwidth that a single connection couldn't. Streaming media — video players seek by asking for the byte range that maps to the timestamp the user clicked. None of this is exotic; every CDN supports it. The cost of not supporting ranges is making your large-file endpoint hostile to mobile networks and unusable for streaming.

11Document with OpenAPI

An API a client can't read is an API a client guesses at. Word-of-mouth integration (Slack threads, internal wikis, "ask the team that wrote it") is the most expensive form of documentation in existence; OpenAPI (formerly Swagger) is the cheapest. It's a YAML or JSON file describing every endpoint — paths, methods, parameters, request and response schemas, error envelopes, auth requirements — that an entire ecosystem of tools knows how to consume.

What you write once, you get many things back

  • Interactive docs — Swagger UI, Redoc, Stoplight, Scalar all render an OpenAPI spec into a browsable, "try it now" doc site. Onboarding a new client becomes "here's the URL," not "here's a tarball of PDFs."
  • Generated client SDKs — openapi-generator, Stripe's own toolchain, and most modern API gateways will spit out type-safe clients in TypeScript, Python, Go, Java, Swift, and Kotlin from the same spec. Client teams stop writing HTTP plumbing.
  • Server-side validation — middleware (express-openapi-validator, FastAPI's Pydantic models, Spring's springdoc) can check incoming requests against the spec at runtime. The schema becomes enforcement, not just documentation — closing the gap the REST critique opens up.
  • Mock servers — Prism, Stoplight, Postman can stand up a fake server straight from the spec so the frontend can build against the contract before the backend ships a single line.
  • Contract tests in CI — fail the build the moment a code change diverges from the spec the docs site advertises.

What to put in the spec

The minimum useful surface:

  • Every endpoint's path, method, summary, and description.
  • Path / query / header parameters, with type, example, and required-or-not.
  • Request body schema (referenced from a shared components/schemas block — define User once, reuse everywhere).
  • Every response status code your endpoint can return, each with its own schema (success bodies and the Problem Details error envelope from §06).
  • Auth requirements per endpoint (which scopes a token needs).
  • Examples — concrete request/response samples for every important shape. Examples are what readers learn from, not schemas.
Spec-first vs code-first — and why it matters
Two ways to keep the spec and the implementation in sync. Spec-first writes the YAML by hand, then generates server stubs and clients from it; the spec is the source of truth and the code follows. Code-first writes the implementation with annotations (FastAPI, NestJS, Spring) and generates the spec from the code; the code is the source of truth and the spec follows. Either works. What doesn't work is a hand-maintained spec next to a hand-written server — they will drift, and the docs site will lie. Pick one source of truth and let tooling derive the other.

+1Bonus — Plan for rate limiting

Rate limiting is the tip teams ship after the first abusive client lights up the database. By then, the customer-facing endpoint is on fire and there's no clean place to add the gate without breaking integrations. Plan for it before you need it; the cost while no one's abusing you is approximately zero.

Two purposes worth separating:

  • Fairness — one tenant or one API key can't starve everyone else. The cap is per-identity (user, key, IP), not global.
  • Defense — cap the cost of abuse, scraping, and DoS. The cap kicks in before the request hits anything expensive.

Both belong at the API gateway, which sees every request before it touches a backend. Common algorithms — token bucket, leaky bucket, sliding window — all answer the same question (allow or deny) with different memory and burst characteristics.

Communicate limits in the response

Rate limits are part of the contract; clients can't respect them if they can't see them. Standard headers:

HTTP/1.1 200 OK X-RateLimit-Limit: 1000 X-RateLimit-Remaining: 487 X-RateLimit-Reset: 1730000000 # When the limit is exceeded: HTTP/1.1 429 Too Many Requests Retry-After: 60

The Retry-After header on a 429 response is the single best protocol-level rate-limit primitive that exists. Well-behaved clients stop retrying until the server invites them back; servers actively shape their own incoming load. Use it.

Limit by what?
Per-key, per-user, per-IP, or per-endpoint? The right answer is usually several at once — a generous overall key budget, a tighter per-endpoint cap on expensive operations (search, exports), and an IP-based cap as a backstop against unauthenticated abuse. GraphQL adds a wrinkle: you have to limit by query complexity, not request count, because one query can cost as much as a thousand REST calls.
One more habit — split the API and the docs onto sibling subdomains
Public APIs converge on a stable layout: api.example.com serves the actual traffic; developer.example.com (or developers.example.com) hosts the docs portal, sandbox, and key-management UI. Keep them on different subdomains so they can scale, cache, and harden independently. Add web redirects for the obvious typos: a browser hitting api.example.com with no path should land on the developer portal; dev.example.com and developers.example.com should both resolve to the canonical developer.example.com. None of this is load-bearing on day one, but it's the layout you'll wish you'd picked when the second client integrates.

∞The one-line takeaway

A good API is one a client can read once and trust — naming that doesn't surprise, idempotency that survives retries, versioning that doesn't break v1, pagination that doesn't fall off the cliff, query strings that filter cleanly, responses and errors with the same shape on every endpoint, security that isn't bolted on, references that don't lock you into yesterday's hierarchy, async patterns that survive long jobs, range requests that survive long files, and an OpenAPI spec that keeps the docs honest.

None of these tips are exotic. Each one is roughly five extra minutes at design time, paid back the first time a client integrates without filing a ticket, the first time a retry storm doesn't double-charge a customer, the first time a v2 ships without breaking v1. Treat them as the cheapest insurance you'll ever buy on a piece of code that will live longer than the team that wrote it.
