HTTPS & the TLS handshake

How two parties on a hostile network agree on a shared secret without ever sending it — and why the lock icon in your browser is a real promise rather than a UI flourish.

An HTTP request is a postcard. It travels from your laptop through your home router, your ISP, three or four backbone routers, the destination's edge proxy, and a load balancer before it reaches a server — and at every hop, anyone with a packet capture tool can read every byte. HTTPS is the envelope. Sealing the envelope is the easy part. The hard part is agreeing on what seal to use without a trusted courier.

This page works through the problem in order: what HTTP fails to protect, the cryptographic primitives that make the protection possible, the math that lets two parties on a hostile wire derive the same secret while an eavesdropper learns nothing, and the one-round-trip handshake that does it all every time you click a link.

01What HTTP gets wrong

Plain HTTP fails on three independent axes. Each failure has a real-world attack class behind it.

Figure 01 — Three threats HTTP doesn't defend against

HTTP doesn't encrypt (anyone in the path can read), doesn't authenticate (anyone in the path can rewrite), and doesn't identify (the server claims to be bank.com with nothing to prove it). HTTPS adds one defence per failure: encryption, integrity, authentication.

HTTPS is HTTP carried over TLS (Transport Layer Security; SSL is the deprecated predecessor name still used colloquially — "SSL certificate" is technically a misnomer for "TLS certificate" but it's everywhere). TLS gives three guarantees, one for each problem above:

  • Encryption — only the client and the legitimate server can read the payload. Eavesdroppers see ciphertext.
  • Integrity — every TLS record carries a message authentication code; tampering changes the MAC and the receiver detects it. Either the bytes arrive intact or the connection breaks.
  • Authentication — the server proves it owns the domain by presenting a certificate signed by a trusted Certificate Authority. The browser ships with a list of CAs it trusts and rejects anything not signed by one of them.

The browser UI cues — the lock icon, no warning interstitial — exist precisely to surface these guarantees to the user.

02Building blocks — keys, certs, hybrid encryption

Three primitives compose into the handshake. Each is simple in isolation; the cleverness is in how they fit together.

Asymmetric (public-key) cryptography

Each party has two mathematically linked keys. Anything encrypted with one can only be decrypted with the other. By convention, you publish one (the public key) and keep the other (the private key) secret forever.

Figure 02 — Two operations on a key pair

Two operations, two directions. Encryption protects content. Signing proves origin. The TLS handshake uses both: the certificate's signature proves identity; the key exchange that follows establishes a shared secret for bulk encryption.

One critical caveat: asymmetric crypto is slow — orders of magnitude slower than symmetric ciphers like AES. So TLS uses it sparingly: only for setting up a shared secret, then switches to AES for the actual data. That's the hybrid in hybrid encryption.

The SSL/TLS certificate

A signed document that says "this public key belongs to bank.com". Issued by a Certificate Authority (Let's Encrypt, DigiCert, GlobalSign, etc.). The CA's own public key is shipped inside every browser and OS as part of a trust store; that's the root of the trust chain.

Figure 03 — The certificate trust chain

The certificate is a chain. The browser trusts ~150 root CAs out of the box; those CAs sign certs for individual domains; the domain proves ownership of its cert by signing the handshake with the matching private key. Break any link and the chain fails.

Mutual TLS (mTLS) is when the client also presents a cert. Used inside zero-trust infrastructure and some B2B APIs, but not for the public web — your browser doesn't have a cert.

03The naive approach & why it fails forward secrecy

A first instinct: "the server has a public key — let me encrypt my whole request stream with it." Simple in theory. Wrong in three ways.

  • Asymmetric crypto is too slow for stream-level use. AES at line rate; RSA at maybe a thousandth of that.
  • RSA encryption has a size cap (the modulus size). You'd have to chunk and re-encrypt, multiplying overhead.
  • No forward secrecy. If the server's long-term private key is ever stolen — say, a year from now — every recorded session from before is now decryptable. The attacker just had to capture ciphertext at the time and patiently wait for the key to leak.

The forward-secrecy failure is the deepest problem. Long-term keys leak. Servers get compromised. Subpoenas happen. A protocol that relies on the long-term private key staying secret forever is a protocol with a time bomb. The fix: derive a fresh, ephemeral key for every session, and throw it away as soon as the session ends. Even if the long-term key leaks later, the ephemeral keys are gone.

The way to derive that ephemeral key without ever sending it across the wire is Diffie-Hellman.

04Diffie-Hellman — the math behind the handshake

The puzzle: two parties on a hostile network, who have never communicated before, agree on a shared secret. An eavesdropper sees every byte of the conversation. The shared secret must end up known to both parties and to neither the eavesdropper.

Diffie-Hellman solves it with one piece of mathematical asymmetry: computing g^a mod p is easy, but recovering a from g^a mod p is computationally infeasible for large p. That asymmetry is called the discrete logarithm problem, and it's the entire foundation of the protocol.

Figure 04 — Diffie-Hellman key exchange

Both sides exchange only their public values A and B. Each combines their own private secret with the other's public value to land on the same number, g^(ab) mod p. Eve sees A and B — but recovering a from A = g^a mod p is the discrete log problem, which is computationally infeasible at large p.

The mechanics, step by step:

  1. Both sides agree on public parameters: a large prime p and a generator g. (These can be hard-coded or sent in the clear; they are not secret.)
  2. Alice picks a private random a. Computes A = ga mod p. Sends A.
  3. Bob picks a private random b. Computes B = gb mod p. Sends B.
  4. Alice receives B and computes Ba mod p = (gb)a mod p = gab mod p.
  5. Bob receives A and computes Ab mod p = (ga)b mod p = gab mod p.
  6. Both sides now hold gab mod p — the shared secret. AES is keyed from it.

The eavesdropper sees both A and B on the wire. To compute gab mod p, she'd need to recover a from A or b from B — which is the discrete logarithm problem. At a 2048-bit modulus, the best known algorithms take billions of years on adversarial hardware.

Crucially, in TLS 1.3 the DH keypairs a and b are ephemeral — generated for this one session and thrown away when the session ends. Even if the server's long-term certificate key is stolen years later, the ephemeral DH secrets are gone, so past recorded sessions stay safe. That's how forward secrecy is delivered in practice.

ECDHE — same idea, smaller numbers
Modern implementations use ECDHE (Elliptic-Curve Diffie-Hellman Ephemeral). The math runs on points of an elliptic curve instead of integers mod a prime, but the structure is identical: each side has a private scalar, exchanges a public point, and combines them to land on the same shared point. Equivalent security at much smaller key sizes (256 bits ECC ≈ 3072 bits RSA) and faster operations. Same mental model — just better numbers.

Where the certificate fits in

Diffie-Hellman alone is vulnerable to a man-in-the-middle: Mallory could pretend to be Bob, do DH with Alice, and separately do DH with Bob, sitting in the middle decrypting and re-encrypting everything. DH gives you a shared secret with somebody; you still need a way to know which somebody.

That's the certificate's job. The server signs its DH public value with the private key from its certificate. The client verifies the signature against the public key in the certificate, and verifies the certificate against a trusted CA. The chain proves "the entity I'm doing DH with really owns the certificate for bank.com", which closes the impersonation gap.

So:

  • Certificate → who you're talking to.
  • Diffie-Hellman → a shared secret nobody listening can derive.
  • Shared secret → seeds AES for the actual data.

05The TLS 1.3 handshake, end to end

Putting it all together. TLS 1.3 collapses what TLS 1.2 did in two round-trips into one — a meaningful latency win on every fresh HTTPS connection. The handshake is short.

Figure 05 — TLS 1.3 handshake (1-RTT)

One round-trip: client sends DH public + supported ciphers; server sends DH public + cert + signed handshake + Finished. Client verifies and replies Finished. Application data flows from the very next packet, all encrypted under the shared secret.

Step by step:

  1. ClientHello. The client sends supported ciphers (so the server can pick), its ephemeral DH public value A, and a random nonce.
  2. ServerHello + Certificate + Signature + Finished. The server picks a cipher, sends its ephemeral DH public value B, sends its certificate, and signs a transcript of the handshake so far with the certificate's private key. The Finished message contains a MAC over the handshake under the freshly derived shared secret — proving the server knows the secret.
  3. Client verifies. Cert chains to a trusted CA. Signature verifies against the cert's public key. The handshake transcript matches.
  4. Both derive the session key from the shared DH secret (with a key-derivation function — HKDF — for hygiene).
  5. Client Finished. Client sends its own MAC over the handshake. Both sides have now confirmed they hold the same key.
  6. Encrypted application data flows with AES (or ChaCha20) under that session key. From the very next packet.

Three things to notice. First, only one round-trip separates the click from encrypted application data. Second, the certificate never encrypts the bulk traffic — its signature only binds the DH exchange to a specific identity. Third, session resumption (TLS 1.3 0-RTT, pre-shared keys) can skip even that round-trip on a return visit; the cost is some weakening of the forward-secrecy story for the very first request.

06The one-line takeaway

The certificate proves who. Diffie-Hellman delivers a shared secret nobody listening can derive. AES does the bulk encryption. Forward secrecy means stealing the long-term key tomorrow can't decrypt the traffic you sent today.

Once that decomposition is internalised, every other piece of TLS terminology slots in. ECDHE? Same DH structure, smaller numbers. mTLS? Both sides have certificates. 0-RTT? Re-use a stored secret to skip the round-trip on a return visit. Cert pinning? Refuse to trust the global CA list and accept only one specific cert. The vocabulary explodes, but the core protocol is just three primitives composed in a way that survives a hostile network.
