Pagination

Offset/limit's two failure modes — the deep-page perf cliff and inconsistency under live writes — and how cursor pagination fixes both with an indexed seek and a stable handle on "where I left off".

A list endpoint is a small lie. GET /posts looks like one operation, but the result set might be ten rows or ten million. Returning everything in one response burns memory on both sides, blows past the bandwidth budget, and pushes time-to-first-byte into the seconds. Pagination is the contract that turns one impossible request into a sequence of bounded ones — and the choice of contract decides whether deep pages are cheap or catastrophic.

This page walks the two main pagination styles in the order most teams meet them: offset/limit (the obvious choice that breaks at scale), then cursor (the fix). The two failure modes of offset — the performance cliff and inconsistency under writes — share a root cause that cursor pagination directly addresses.

01Why paginate at all

Three forces push back against "just return everything":

  • Bandwidth. A million 1 KB rows is a 1 GB response. Mobile clients won't survive it; even on a fast desktop the browser will OOM.
  • Memory. The server has to materialise the result set somewhere. A list of a million ORM objects is enough to crash a process even if the network could handle the bytes.
  • Time-to-first-byte. The user staring at a spinner is paying the latency for the last row, not the first. Pagination lets the first 20 rows flush early and leaves the rest for "load more".

So the API takes the list a slice at a time. The interesting question is how the client tells the server which slice to return next.

02Offset/limit pagination

The classic shape, and the one most APIs reach for first because it maps naturally onto SQL:

GET /posts?limit=20&offset=40

Server runs:

SELECT * FROM posts ORDER BY created_at DESC LIMIT 20 OFFSET 40;

It feels natural — "give me page 3 of 20-per-page" maps to offset=40, limit=20. UIs love it because rendering "Page 3 of 50" is straightforward. Admin dashboards live on it. For small datasets it works fine.

It has two serious problems. The first one bites at scale. The second one bites under any live-write traffic, regardless of size.

03The two problems offset doesn't solve

Problem 1 — performance falls off a cliff at large offsets

OFFSET 1000000 does not mean "skip a million rows cheaply". It means the database has to scan and discard the first million rows before it can return yours. The cost is O(offset), not O(limit). A "load page 50,000" request costs the same as scanning 50,000 pages of data — even though the user only sees 20 rows.

Indexes help the ordering step (the database walks an index instead of sorting on the fly) but they don't avoid the discard-the-first-million step. The query plan still has to step over every offset row before it starts emitting yours.

Figure 01 — The offset cliff: cost grows linearly with depth

Offset cost grows linearly with depth — every "next page" pays for the rows it has to skip. Cursor cost is flat: an indexed seek to "where I left off" is the same operation whether you've paged once or a million times.

The cliff is invisible during local development (your test database has 100 rows; offset is free) and during early production (the offsets are still small). It shows up the day a user with a million-row inbox or a script crawling page-by-page hits the back of the table — and at that point the database is doing scan-and-discard for every request, which lights the CPU on fire.

Problem 2 — inconsistent results when data changes

Page 1 returns rows 1–20. The user reads them. Then someone inserts a new post at the top of the table. The user clicks "next page". Page 2 returns rows 21–40 — but the post that was at position 20 is now at position 21, so the user sees it twice. The reverse happens with deletions: rows skip out of view and the user never sees them at all.

Figure 02 — Why an insert duplicates a row across pages

Offset is computed against the table's current state every time. An insert at the head shifts every row down by one; the row that was at position 20 is now at position 21, so it appears at the top of page 2 too. Deletions cause the mirror-image bug — rows skip and never appear at all.

This sounds minor until you've built a feed. A user scrolling through "page 1, 2, 3" of a live-updating timeline sees duplicates and gaps. Worse: if the sort key changes (a post is edited and updated_at jumps), every subsequent page is scrambled, because the row's position in the ordering moved while the user was paginating.

Both problems share a root cause: offset is computed against the current state of the table at query time. There's no stable handle on "where I was." The fix is to give the client one.

04Cursor pagination

Instead of "skip 40 rows", say "give me what comes after this row". The "this" is a cursor — typically the sort key of the last row from the previous page, often base64-encoded so it looks opaque to clients.

GET /posts?limit=20 → { "results": [...], "next_cursor": "eyJpZCI6MTIzfQ" } GET /posts?limit=20&cursor=eyJpZCI6MTIzfQ → { "results": [...], "next_cursor": "..." }

Server runs (predicate depends on the sort key):

SELECT * FROM posts WHERE id < 123 ORDER BY id DESC LIMIT 20;

For a feed sorted by created_at with ties broken by id, the query is a tuple comparison so duplicates at the same timestamp don't slip:

SELECT * FROM posts WHERE (created_at, id) < ($cursor_ts, $cursor_id) ORDER BY created_at DESC, id DESC LIMIT 20;
Figure 03 — Cursor pagination as a moving handle

The cursor is the trailing edge of the previous page. The next query becomes "rows where the sort key is less than the cursor", which an index can answer with one seek. Inserts above the cursor don't affect the result; the user keeps walking forward through what they've already seen.

Why it's better:

  • O(log N) per page, not O(offset). The database does an indexed seek to the cursor position; the offset doesn't appear in the query at all.
  • Stable across inserts and deletes. New posts at the top don't shift you — the cursor is anchored to a specific row, not a position. You keep walking forward through the data you've already seen.
  • Natural fit for infinite-scroll UIs. Every "fetch more" call is just "give me what comes after the last thing I have."

What you give up:

  • Can't jump to "page 50". Only forward (and sometimes backward) sequentially. If the UI needs random-access pagination — "Page 50 of 200" with clickable page numbers — cursor doesn't fit.
  • Need a stable, monotonic sort key. Usually the primary key or a (created_at, id) tuple. If the sort changes mid-walk, the cursor's meaning breaks.
  • Cursor should be opaque. If you expose cursor=12345, clients will start crafting their own. Signing or encrypting the cursor keeps the contract under your control and avoids leaking schema details.

05When to pick which

Reach for offset/limit when…

  • The dataset is small and bounded (admin lists, settings, lookup tables).
  • The UX requires "Page N of M" with clickable page numbers.
  • The user routinely needs to jump to a specific page.
  • Writes are rare or paused during the user's session.

Reach for cursor when…

  • The dataset grows unbounded (feeds, search results, logs).
  • The UI is infinite-scroll or "load more".
  • Live writes are happening while users paginate.
  • Users routinely page deep into the dataset.

The rough rule: if your dataset can have a million rows and users routinely scroll past the first thousand, you want cursor. Offset is fine until the offsets get big — and then it's not, all at once.

Hybrid pagination
Some APIs offer both: cursor pagination by default for forward scrolling, with an optional page/page_size mode for admin tooling that needs jump-to-page. The two cost models are different and should be exposed as different endpoints (or different query parameters that can't be mixed) — not as a flag the client toggles invisibly.

06The one-line takeaway

Offset says "skip the first N rows". Cursor says "give me what comes after this row". The latter is what indexes are good at — and what stays consistent when writes are happening underneath.

Both styles look the same to a small-dataset eye, which is why offset is the natural first choice. The day the cliff arrives — the day a user with a long inbox or a script crawling deep pages hits the back of the table — the cost of switching to cursor is high, because clients have already baked page=N into their integration. Picking cursor up front, even if the UX shows page numbers, is the cheap insurance. The page numbers can be a thin client-side projection over a cursor underneath.
