API design

Three styles compared (REST, gRPC, GraphQL), the verbs and idempotency rules that make retries safe, and how to turn an unsafe POST into one a client can call ten times without consequence.

A system's API is the smallest interesting thing about it that's still load-bearing. The architecture diagram, the database choice, even the language — all of those can change underneath. But once a client is talking to your API, every endpoint, every field name, every status code is a promise. The contract is the part of the design that's hardest to change, so it's the part that deserves the most care.

This page walks the three dominant API styles, the HTTP verbs and idempotency rules that REST inherits from the protocol, the inefficiencies that get blamed on REST (and the reasons we keep using it anyway), and the two patterns for making a non-idempotent operation safe to retry.

01Why API design is the contract step

The interview framework places API design third — after requirements and entities, but before the high-level architecture. The order isn't arbitrary. Once the API is fixed, you've committed to:

  • What state has to be readable — every GET needs a backing store that can answer it cheaply.
  • What writes have to be durable — every POST/PUT is a synchronous promise; whatever it returns has to actually be persisted.
  • What can fan out async — any operation whose response doesn't include the downstream result is implicitly a fire-and-forget — write goes through, side-effects happen later.
  • The error surface — every status code and every error body shape becomes a thing clients write conditionals against.

Drawing a load balancer or picking a database before any of that is decided is premature. The API tells you what the system has to be able to do; the architecture is just whatever can satisfy those calls within the non-functional budget.

02Three API styles — REST, RPC, GraphQL

Three dominant styles in 2026. They overlap in capability but optimise for different things. Most large companies use all three: REST for the public API, gRPC for internal service-to-service calls, GraphQL for the BFF layer that aggregates services for a UI.

Figure 01 — Three styles, three mental models

Each style answers "what should the client send across the wire?" differently. REST sends a URL plus a verb. RPC sends a typed function call. GraphQL sends a query that names the exact fields it wants back. The downstream consequences — caching, tooling, schema rigidity, debuggability — all fall out of that choice.

REST

The dominant style on the public web. Resource-oriented — the URL identifies a thing (a user, a tweet, a list of orders) and the HTTP verb describes the action on it.

GET /users/123 # read a user POST /users # create a user PUT /users/123 # replace user 123 DELETE /users/123 # remove user 123 GET /users/123/orders # list user 123's orders

Four properties matter:

  • Stateless. Each request carries everything needed to process it (auth token, parameters). The server holds no per-client session in memory; this is what lets every instance handle every request, which is what makes horizontal scaling the default rather than the goal.
  • Cacheable via HTTP semantics. Cache-Control, ETag, Last-Modified, If-Modified-Since — the protocol gives you a vocabulary every CDN, browser, and intermediate proxy already speaks.
  • Text-based. Usually JSON. Human-readable; curl works, browser dev tools work, you can debug at 2 a.m. with your eyes.
  • Loose coupling. URLs are the contract, not generated stubs. Adding a field doesn't break old clients; renaming one breaks only the clients that read it.
A fifth property worth knowing — HATEOAS
Roy Fielding's original REST dissertation argues a response should embed hyperlinks to the actions a client can take next — "links": { "cancel": "/orders/45/cancel", "refund": "/orders/45/refunds" } — so the client navigates by following links instead of hard-coding URL templates. Worth knowing because it shows up in interviews and in some HATEOAS-y standards (HAL, JSON:API, Siren). Worth knowing it's not universal: most public REST APIs in the wild ship URI templates in their docs and call it a day. Treat it as a tool you can reach for, not a rule you have to follow.

RPC — gRPC, Protobuf, Thrift

Function-oriented instead of resource-oriented. You call a remote method as if it were a local function — userService.GetUser(123) — and the framework hides the network behind a generated client stub.

gRPC is Google's modern RPC framework, the de-facto standard for service-to-service calls inside a single system. Two pieces:

  • Protocol Buffers (protobuf) — schema-first IDL. You write a .proto file describing your messages and methods; a compiler generates type-safe client and server stubs in many languages (Go, Java, Python, Rust, etc.).
  • HTTP/2 + binary framing — multiplexed streams over a single TCP connection. Supports unary (one request, one response), server streaming, client streaming, and bidirectional streaming.

The wire format is binary and schema-aware. id=123 becomes ~3 bytes (a field tag plus a varint), not the 20+ bytes of {"id": 123} in JSON. Compactness and parse speed are the headline wins.

Figure 02 — JSON vs Protobuf wire cost

Protobuf trades human readability for ~3× smaller payloads and faster parsing. The trade-off only pays back when both ends are under your control and the volume justifies the rebuild of every diagnostic tool that used to "just work" with JSON.

Trade-offs:

  • Binary on the wire — you can't curl it and read the response. Need grpcurl or a generated client.
  • Browsers can't speak gRPC natively (HTTP/2 trailers aren't exposed); gRPC-Web exists as a workaround but adds a proxy.
  • The schema couples client and server harder than REST does — a renamed field can break old clients unless the team is disciplined about field-number stability.
  • Best-fit niche: internal service-to-service where you control both ends and want speed plus types.

GraphQL

A query language over the API. Single endpoint (typically POST /graphql); the body says exactly which fields to return.

query { user(id: 123) { name orders(last: 5) { id total items { name price } } } }

What it solves: over-fetching (REST's GET /users/123 returns 30 fields when the client wants 3) and under-fetching (the client wants user + orders + items and pays N round-trips in REST). GraphQL collapses both into one shaped response.

What it costs:

  • No URL-based caching. Every query is a POST to /graphql, so the CDN/browser-cache mechanics REST gets for free no longer apply. Caching shifts to the GraphQL server (DataLoader, Apollo cache, persisted queries).
  • N+1 queries on the resolver side. If you naively resolve orders.items per-order, you get one DB query per order. The standard fix (DataLoader / batch loaders) is mandatory, not optional.
  • Hard to rate-limit by request count. One query can be tiny; the next can ask for the universe. You end up rate-limiting by query complexity — every field has a cost, the query has a budget.
  • Schema and resolver layer have to be built. More server-side complexity than REST.

03The REST verbs & idempotency

HTTP defines a small set of methods; REST conventionally uses six of them. Two of the columns below decide whether retries and caching are even possible — they're worth committing to memory.

Verb Purpose Idempotent? Safe?
GET Read a resource yes yes (no side effects)
POST Create a resource (typically) no no
PUT Replace a resource (full overwrite) yes no
PATCH Partially update a resource depends no
DELETE Remove a resource yes no
HEAD Like GET but headers only yes yes
OPTIONS Discover allowed methods (CORS preflight) yes yes

Two terms to nail:

  • Safe — calling it doesn't change server state. GET, HEAD, OPTIONS are safe; the rest modify state. Safe methods can be cached, prefetched, and hit speculatively without worry.
  • Idempotent — calling once and calling five times leave the server in the same state. GET is idempotent (reads don't change anything). PUT is idempotent (replace-with-X applied twice is the same as once). DELETE is idempotent (deleting an already-deleted thing is a no-op). POST is not idempotent by default — calling POST /orders twice creates two orders.
Why idempotency matters
Networks fail mid-request. The client doesn't know whether the server got the request, processed it, and the response was lost — or whether the request never landed. If the operation is idempotent, the client can safely retry. If it isn't, retries can double-charge a credit card. Every resilience pattern that involves automatic retries assumes idempotency underneath.

PATCH's idempotency is subtle: a JSON Merge Patch ({"name": "Alice"} overwrites name) is idempotent. A JSON Patch op like {"op": "add", "path": "/tags/-", "value": "vip"} is not — appending the same tag twice produces two tags. The format determines the property.

Where OPTIONS shows up in real life — CORS

OPTIONS looks academic in the table above, but every browser-facing API hits it constantly because of CORS (Cross-Origin Resource Sharing). By default, browsers enforce the same-origin policy: JavaScript on app.example.com can't call api.other.com. CORS is the protocol the server uses to opt in to those cross-origin calls.

Two flavours of request, depending on what the call looks like:

  • Simple requests — GET, HEAD, or a "boring" POST (form-encoded, no custom headers). The browser sends the request directly with an Origin header; the server responds with Access-Control-Allow-Origin naming the origins it accepts. If they don't match, the browser drops the response before JS can read it.
  • Preflighted requests — anything with a custom header (Authorization, Content-Type: application/json, etc.) or a non-simple verb (PUT, DELETE, PATCH). Before the real call, the browser fires an OPTIONS request asking "is this allowed?". The server answers with Access-Control-Allow-Methods, Access-Control-Allow-Headers, and Access-Control-Max-Age (how long the browser may cache the answer). Only if the preflight passes does the real request go out.

The practical upshot for an API designer: every cross-origin POST /payments with a JSON body is really two round-trips — the preflight and the real call. Setting a generous Access-Control-Max-Age (e.g. 600 seconds) lets the browser cache the preflight verdict and reduces the tax to one round-trip per ten minutes. It's a free latency win that's almost always left on the table.

04REST's wire cost — and why we keep picking it

REST's "issues" mostly boil down to efficiency. The reasons to keep using it are mostly reach and flexibility. The trade-off is real, but the conclusion almost always lands on the same side for public APIs.

Where REST is inefficient

  • JSON is verbose — 3× the bytes of an equivalent protobuf message.
  • JSON is slow to parse — tokenising text beats reading length-prefixed binary fields.
  • Over- and under-fetching — endpoints return what they return; clients adapt or pay round-trips.
  • Multiple round-trips for related data — "user → orders → items" is N+2 calls; RTT dominates.
  • No native streaming — push needs WebSockets or SSE bolted on.
  • Schema is implicit — OpenAPI helps, but it's documentation, not enforcement.

Where REST still wins

  • Universality — every language and tool understands HTTP + JSON. New client in 30 seconds.
  • Caching for free — CDN, browser, proxy all already speak HTTP cache semantics.
  • Debuggability — curl works. Postman works. You can read the response with your eyes.
  • Loose coupling — adding a field doesn't break old clients.
  • Statelessness scales — no sticky sessions, no per-client memory; horizontal scaling is the default.
  • Conventions are well-known — a new engineer onboards in an afternoon.

REST is slower on the wire but faster to ship, easier to cache, and easier to evolve.

For a public API serving heterogeneous clients (browsers, mobile apps, partner integrations, scripts), the wire-speed loss is rarely the bottleneck and the operational wins are huge. The right time to reach for gRPC is when you control both ends, the volume justifies the tooling rebuild, and the latency budget is genuinely tight. The right time to reach for GraphQL is when one client (typically a UI) needs an aggregated, reshaped view across many backend services.

05Making POST safe to retry

The retry problem: a client POSTs /payments to charge $50. The connection times out before the response arrives. Did the charge land? The client can't tell. If it retries blindly and the original landed, the customer gets charged twice.

Two patterns solve this. They're not exclusive — large systems use both.

The Idempotency-Key header

Stripe popularised this; most payment APIs follow it now. The client generates a random unique key (a UUID is the common choice) and sends it on the request:

POST /payments HTTP/1.1 Idempotency-Key: 7f8a3c12-9e4b-4d1a-b2e7-1c0f8d9a4e22 Content-Type: application/json {"amount": 5000, "currency": "USD", ...}

The server, on first receipt, looks up the key in a fast durable store. If it's not there, the server processes the request and stores the full response (status code + body) under the key with a TTL. If the key is found, the server returns the stored response immediately, without re-processing. Retries become cheap replays.

Figure 03 — Idempotency-key request flow

First attempt: cache miss, do the work, store the response. Retry with the same key: cache hit, replay the stored response. The customer is charged exactly once even if the client retries ten times.

Things to get right:

  • The key is client-generated, not server-generated — that's the whole point. The client knows "this is the same logical request" before the server does.
  • Store the entire response, not just "success/fail". A client retrying after a network blip wants the same body the first attempt produced (including the order ID, etc.).
  • Keep the storage durable — Redis with persistence, or an indexed DB row. Losing the key store loses idempotency.
  • Pick a TTL that exceeds the client's retry budget. If the client retries for up to 10 minutes, the TTL should be at least that. Stripe defaults to 24 hours.
  • Validate that a repeat key with a different request body is rejected — otherwise an attacker can replay your idempotency key with a malicious body and get a "success" response back.

Natural-key dedup at the storage layer

Sometimes the request already carries something unique — a client-generated order_id, a transaction_reference, a (user_id, day, type) tuple. The server can use a unique constraint plus an upsert:

INSERT INTO orders (id, user_id, total, ...) VALUES (...) ON CONFLICT (id) DO NOTHING RETURNING *;

If the row already exists, the insert is a no-op and the server returns the existing row. The dedup happens in the database, no extra header required.

When to pick which:

  • Idempotency-Key is general — works for any operation, even if the body has nothing unique about it. Costs: the client has to participate, plus a key-store dependency.
  • Natural-key dedup is automatic — no client cooperation needed — but only works when the request structurally identifies itself.
  • Big systems often use both: header-level dedup as the protection most clients see, plus storage-level constraints as the safety net underneath.

06The one-line takeaway

The API is the contract — pick the style that matches your reach, follow the verb semantics, and treat idempotency as a precondition for safe retries, not an afterthought.

REST will keep winning the public-API surface for as long as browsers exist, gRPC will keep winning service-to-service for as long as latency budgets are tight, and GraphQL will keep winning the BFF layer for as long as one UI needs to compose many backends. The verbs and idempotency rules underneath all three are inherited from HTTP itself; they don't change when you change styles. Get them right and every retry, every cache, every CDN starts working in your favour.
