Latency

The time one request takes from start to finish — and why the hierarchy of how long different operations take stretches across nine orders of magnitude.

Latency is the stopwatch. You press start when a request leaves, stop when the response comes back, and read the number. That's it — one number, per request. Most of what it measures isn't computation; it's waiting. And the waiting lives in a hierarchy of places whose costs differ by factors of a million.

This page is about that single number: what it actually contains, the well-known hierarchy of how long different operations take, and a subtle trap (access latency vs bandwidth) that hides inside the canonical list.

01What latency is

Latency is the duration between starting a request and getting the response. It's a per-request measurement, reported in milliseconds, microseconds, or occasionally seconds. One number per request. If you sent 10,000 requests you'd have 10,000 latency samples — and a distribution worth looking at closely (that's a separate topic, covered on the combined page).

Latency is not the same as throughput. A service can answer a single request instantly and still be slow at handling many — or vice versa. The two numbers can move independently and often trade off against each other; see the Latency ⇄ Throughput page for how they interact.

Latency is the end-to-end clock time for one request. Most of it is usually waiting — in queues, on the network, on disk — not actual computation. Optimizing latency means finding out which stage dominates and attacking that one.

Latency is dominated by whoever is slowest in the chain. Shaving 2 ms off a 12 ms compute step matters; shaving 2 ms off a 2 ms network hop is usually not worth the effort. Before optimizing, measure — profile the stages and find the fat one.

03The 1,000× hierarchy

Jeff Dean's famous list (still the reference) boils performance down to a short table of numbers every engineer should have memorized. The surprising insight is that the hierarchy is roughly logarithmic — each tier is about 1,000× slower than the one above it. A cache miss that falls through to disk is a million times slower than a hit.

04Multiply everything by a billion

Raw nanoseconds are hard to feel. A classic trick is to scale every number up by 109 — treat one nanosecond as one second — and the hierarchy suddenly becomes human:

  • L1 cache hit (~1 ns) → 1 second (a heartbeat).
  • Main memory reference (~80 ns) → 80 seconds (a short errand).
  • NVMe SSD random read (~30 µs) → 8.3 hours (a workday).
  • Same-datacenter round-trip (~500 µs) → 5.8 days (a vacation).
  • HDD seek (~10 ms) → 16.5 weeks (a university semester).
  • Cross-continent packet (~150 ms) → 4.8 years (a bachelor's degree).

Seen this way, the cost of hitting disk or crossing an ocean isn't just "slower" — it's categorically a different kind of activity. This is why cache locality, keeping data in memory, and minimizing network hops dominate performance work. Even a single extra cross-region call can cost more than the entire CPU budget of a request.

The practical lesson
Before profiling, guess. Which tier does this operation live on? If a request touches disk or a remote service, that's almost certainly where the time is going — not in your carefully optimized loop. Optimize the slowest tier first; anything else is micro-optimization on a rounding error.

05Turning numbers into technology choices

The hierarchy is also a decision tree for picking the right tool. Working backward from your latency budget:

Need < 1 ms

Data must be in memory. In-memory stores (Redis, Memcached), local caches, or replicated state. Anything involving disk or network will blow the budget by itself.

Need < 10 ms

Local SSD is fine; same-datacenter network calls are fine. Typical zone for a single DB query plus a little business logic. A single cross-region hop already eats your budget.

Need < 100 ms

Room for a handful of same-region network calls, a DB commit (~1 ms), and some compute. This is the "one page load" budget for most web apps.

Need < 1 s

Cross-region calls are feasible. Complex multi-service flows, external API calls, or heavier queries. Beyond this, users perceive the system as "slow" and you need async/streaming patterns.

06Access latency vs bandwidth

There's a subtle trap hiding in the hierarchy above. Every storage tier has two latency numbers — a "single access" number and a "read 1 MB" number — and people conflate them constantly. They measure different physics, and confusing them leads to the wrong design.

Two numbers, two physics

Access latency

How long you wait before getting the first byte. Set by the device's physics — pipeline depth for RAM, flash-controller overhead for SSDs, head seek plus rotational delay for HDDs, round-trip signaling for networks.

Bandwidth

Once data starts flowing, how fast it streams. Set by bus width, number of channels, and raw parallelism. Grows with every hardware generation. RAM: ~100 GB/s. NVMe: ~5–7 GB/s. HDD: ~150 MB/s. 10 Gbps NIC: ~1 GB/s.

A "read 1 MB" number is mostly a bandwidth measurement. A "random read" or "seek" number is mostly a latency measurement. The ratio between them tells you the shape of the device — whether it's optimized for finding things fast or for moving lots of data once found.

What each tier actually does

Tier Single access Read 1 MB Shape
RAM ~80 ns (cache-line miss) ~10 µs (~100 GB/s) 125× for 16K× data
NVMe SSD ~30 µs (random 4 KB) ~150 µs (~5–7 GB/s) 5× for 256× data
HDD ~10 ms (seek + rotation) ~7 ms (~150 MB/s) 1 MB is cheaper than the seek

The HDD row is the oddest — and most instructive. Finding the data costs more than reading it. For spinning disks, the 10 ms seek is pure overhead before a single byte moves; once the head is parked, streaming 1 MB is only ~7 ms on top. This is why databases obsessed over sequential layouts on disk: if you pay the seek once, you might as well slurp a big chunk. One random 8-byte read and one sequential 1 MB read cost roughly the same wall-clock time.

The practical lesson

"How slow is storage?" is actually two questions. You have to ask both:

  1. How often am I paying the per-access cost? Random/scattered access patterns mean latency dominates. A key-value lookup hits this.
  2. How much data am I moving per access? Sequential/bulk workloads mean bandwidth dominates. A full-table scan or a large blob fetch hits this.

This is why SELECT * FROM big_table hates random I/O but loves sequential scans. Why a 4 KB key-value lookup on NVMe (~30 µs) is nearly the same cost as a 1 MB blob fetch (~150 µs) — 5× the time for 250× the data. And why every database's storage engine — B-trees, LSMs, columnar formats — is essentially a strategy for turning random reads into sequential ones.

The two-number rule
Whenever you see a single "storage is X fast" number, ask which one it is. Latency and bandwidth are independent axes, and a design that's fine for one access pattern may be catastrophic for the other. Capacity-plan both.

07The one-line takeaway

Latency is the time one request takes. Most of that time is waiting — in a hierarchy of tiers whose costs differ by a factor of a million.

The hierarchy is the mental model. When you're staring at a slow endpoint, the first question isn't "is my code efficient?" — it's "which tier is this request actually spending its time in?" Almost always, the fat stage is crossing a tier boundary (disk, network, cross-region) and everything else is noise.
