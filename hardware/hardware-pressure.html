Hardware pressure

"How would you handle 1M req/s?" has no answer until you say which kind of request. Each kind pushes on a different part of the machine.

Someone walks up to a whiteboard and asks "design something that handles 1 million requests per second." It feels like a single number, a single problem. It isn't. A million tiny health checks, a million credit-card charges, and a million 4K video streams all read "1M RPS" on a graph — and they need wildly different machines underneath. The unit hides everything that matters.

This page is about what "1M requests" actually does to a server. The same load pushes on different hardware depending on what the request is doing — and once you can see which component is being squeezed, the design starts to write itself.

01The trick question

"Can your system handle 1M req/s?" is a trick question because it conflates the rate with the work. Two services with identical RPS numbers can be doing completely different jobs:

Service A — health probes

1M req/s of GET /healthz. Each request is < 100 bytes, returns "OK", touches no database. The CPU barely shrugs; the network card is the only thing breaking a sweat.

Service B — video transcode

1M req/s of "transcode this 1080p clip to 720p." Each request burns seconds of CPU on a video codec, reads gigabytes from storage, writes more back. One box couldn't even start.

Same number on the dashboard. Same words at the meeting. Different planet underneath. There is no answer to "can you handle 1M req/s" until you describe what one request actually does — what it reads, what it writes, what it computes, where the bytes have to travel.

The real question
Replace "how many requests" with "what does one request cost, on which component." Cost in CPU cycles, cost in bytes pushed, cost in disk seeks, cost in network round-trips. Multiply by RPS. Now you have a number a piece of hardware can answer.

02Four things a request can pressure

A modern server is, roughly, four resources welded together. Any request stresses some mix of them — but rarely all four equally. Knowing which one is being squeezed is half of system design.

Figure 01 — The four pressure points

CPU spends cycles on parsing, math, crypto, business logic. RAM holds the working set, the cache, and the in-flight state of every connection. Disk persists data and is the slowest tier most requests touch. Network carries every byte the request reads or writes — and at high RPS, packets-per-second can saturate the NIC before bandwidth does.

CPU — cycles per second

The CPU is the cycle-burner. Every parsed header, every JSON deserialization, every SHA-256, every gzip, every floating-point multiply costs cycles. A modern core does ~3 × 10⁹ cycles/sec; a 32-core box does ~10¹¹. Once those cycles are spent, no clever scheduling gets them back. CPU pressure looks like high %user or %sys in top, and the fix is either fewer cycles per request or more cores.

RAM — capacity and bandwidth

RAM is two resources in one suit: capacity (how much fits) and bandwidth (how fast you can stream it). At a million in-flight requests, even 1 KB of state per request is a gigabyte. Caches, hash tables, connection buffers, sort scratch space, deserialized objects — they all live here. Run out of capacity and you spill to disk (death). Run out of bandwidth and the CPU starves while it waits for cache lines.

Disk — IOPS and throughput

Disk is what survives a power cut. NVMe SSDs do hundreds of thousands of IOPS and a few GB/s of bandwidth; spinning disks do ~150 IOPS and ~150 MB/s. Almost every request that "writes to the database" is really queueing a disk write somewhere. At 1M req/s, even 10% disk-touching requests is 100K IOPS — that's a real provisioning problem, not a footnote.

Network — bandwidth and packets

Network has two limits people forget are separate: bandwidth (Gbps) and packets per second. A 10 Gbps NIC can move ~1.25 GB/s of payload, but only after it punches through ~1–2 million packets/sec of OS overhead. A flood of tiny requests can saturate packets long before bytes. And every request that calls another service spends RTTs on top of its own work.

03Five requests, five pressure profiles

Here's the same "1M req/s" shaped five different ways. Read the chart like a small bar chart per row: each row is one kind of request, the four columns are the four hardware components, and the height of each bar is how hard that request leans on that component. The orange bar is the bottleneck — the one that fills up first — and the label on the right names which resource of that component runs out (a network bottleneck can be packets/sec or bytes/sec, and they're not the same thing).

Figure 02 — The same RPS, five different shapes

Same 1M req/s, five different request types, five different bottlenecks. Two of the rows (health check and video segment) both bottleneck on Network — but on different network resources: tiny requests run out of packets-per-second long before bytes-per-second, while video chunks run out of bytes-per-second long before packets-per-second. Same column, same height, different fix. The component bar tells you where to look; the right-hand label tells you what specifically ran out.

1. Health check — network-bound (packets)

A 50-byte GET that returns "OK" does almost nothing on the CPU and zero work on disk. At 1M req/s, what saturates is packets per second — the NIC and the kernel network stack handling 1M tiny TCP exchanges. Bytes are trivial (50 MB/s), but each one is its own packet, and the per-packet overhead is the limit. The fix is fewer, fatter requests (HTTP/2, batched probes) — not more cores.

2. Password hash — CPU-bound

A login request with bcrypt at cost factor 12 spends ~250 ms of pure CPU per request. One core does ~4 logins/sec. To do 1M req/s you'd need 250,000 cores — which is why nobody does 1M logins/sec, and why login pages are the textbook place to put a CDN, a queue, or a rate limit. The pressure isn't the network or the database; it's that a deliberately-slow function multiplied by RPS is a CPU mountain.

3. Cache lookup — RAM-bound

A Redis GET on a hot key reads a ~1 KB value from RAM and ships it back. CPU is light; disk is untouched; bandwidth is fine. What gets squeezed is memory bandwidth on the cache box — every value travels DRAM → CPU → NIC, and at 1M req/s that's 1 GB/s of memory traffic per server before you count metadata. Add 100K connections and the working set of TCP state itself starts to evict your hot cache lines. RAM capacity matters; RAM bandwidth matters more than people think.

4. Order write — disk-bound

An INSERT into Postgres has to durably hit disk before it can return — that's what "committed" means. Each commit is at least one fsync to the write-ahead log. NVMe fsyncs run ~10–50 µs each; you can pull tens of thousands per second per disk, not millions. At 1M req/s of writes, the disk runs out before anything else does. The fix isn't bigger CPUs — it's batching commits, sharding writes across many disks, or relaxing durability per request.

5. Video segment — network-bound (bytes)

A 2 MB HLS chunk is small CPU (already encoded), small disk (probably page-cached), small RAM, but 2 MB out the door. At 1M req/s, that's 2 TB/s of egress — physically impossible from one rack. Real video systems don't serve from one box; they spread the bandwidth across a CDN, geographically close to viewers, where each edge box only handles a sliver. The whole architecture exists to spread the network bar.

There is no correct answer
Five requests, five different limiting hardware components, five different fixes. "How do you handle 1M req/s?" has no answer that doesn't start with "tell me about one request." The shape of the request decides everything that follows.

04Reading the pressure shape

Once you know how to look, every request has a recognizable shape. A useful exercise: before you write code, sketch which of the four bars will be tallest. You're usually right, and when you're wrong it's because there's a hidden cost — a network call inside a "compute" function, a fsync inside an "in-memory" cache, an O(n²) inside a "lookup."

Workload smell Likely bottleneck Tell-tale metric
Crypto, compression, regex, ML inference CPU %user near 100%
Big in-memory tables, JSON parsing, sorts RAM bandwidth cache-miss rate, IPC drop
Durable writes, log appends, fsync Disk IOPS iostat util%, fsync latency
Large blobs, file downloads, video Network bandwidth NIC at line rate
Lots of tiny requests, RPC chatter Network packets/sec softirq high, NIC pps cap

The mistake every junior engineer makes once: assuming "more requests" always means "more CPU." It often doesn't. A web server pegged at 5% CPU but at line-rate on the NIC is just as stuck as one at 100% CPU — the wall is just somewhere else. Looking at one number lies; looking at all four tells you which lever to pull.

05From shape to capacity plan

The pressure profile is also a budget. Once you know which component dominates, you can do crude arithmetic and get a number that's right to within a factor of ten. That's the whole game of back-of-the-envelope estimation; this is its raw material.

If CPU dominates

Cycles per request × RPS = total cycles needed. Divide by per-core cycles/sec. That's your core count. Add headroom. Now you know how many boxes — or whether the algorithm itself has to change.

If RAM dominates

Bytes per request × concurrent requests = working set. Plus your caches. Plus the OS. If it doesn't fit, you're spilling to disk and your RAM-bound workload becomes a disk-bound one — silently and catastrophically.

If disk dominates

IOPS per request × RPS = required IOPS. Match against your disk's spec sheet. If it doesn't fit on one disk, you shard, batch writes, or relax durability. There is no software trick that beats the seek time of a spinning disk.

If network dominates

Bytes per request × RPS = bits per second. Compare against NIC line rate and upstream bandwidth caps. Pps-bound? Make requests bigger, batched, or routed through fewer hops. Bytes-bound? CDN, edge caching, compression.

Once one of these numbers is bigger than the box you can buy, you're not optimizing — you're scaling out, which is a different game (sharding, partitioning, fanning across racks). But the first move is always the same: figure out which bar is the tallest, and aim everything at that one.

06The one-line takeaway

"1M req/s" is a meaningless number until you know what one request costs and which component pays. Find the tallest bar; everything else is decoration.

System design questions sound like quantitative puzzles ("scale this to N users") but they're really qualitative ones — what is the work, where does it land, what fills up first. Two engineers handed the same RPS target will design different systems if they imagine different requests. Always force the question down to one request, on one box, against one bottleneck. The architecture falls out from there.
