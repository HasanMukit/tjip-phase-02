The memory hierarchy

Registers, L1, L2, L3, RAM, SSD, HDD, network. Each tier is faster, smaller, and more expensive than the one below. Every performance decision lives somewhere in this pyramid.

If you could buy a single kind of memory that was instant, infinite, free, and survived a power cut, computers would have one tier. We can't, so they have eight. Each tier is faster than the one below it by a factor of ten to a thousand, smaller by a similar ratio, and more expensive per byte by even more. That whole stack — the trade-off you can't escape — is the memory hierarchy. Almost every performance decision is really a question of which tier you're paying for.

This page walks the pyramid: registers at the top, tape at the bottom, and the trade-offs that put each tier where it sits.

01Why a hierarchy at all

Every storage technology trades three things against each other:

Speed

How fast you can read or write a byte — measured in latency (ns, µs, ms) and bandwidth (GB/s). Set by physics: gate switching speed, distance from the CPU, mechanical motion, the speed of light over a cable.

Size

How many bytes fit. Set by density and floor space — registers are tiny because they live inside the CPU; tape is enormous because it spools cheaply.

Cost

Dollars per byte. Set by manufacturing complexity. SRAM is fast and brutal to fabricate; magnetic tape is dense and almost free per gigabyte.

Persistence

Does it survive losing power? Volatile tiers (registers, cache, RAM) lose state on reboot; non-volatile tiers (SSD, HDD, tape) keep it. The boundary matters more than people think.

No single technology wins on all axes. SRAM is fast but expensive. DRAM is cheaper but slower. Flash is non-volatile but slower still. Spinning disk is dirt cheap per byte but glacial. So real machines stack them: a tiny scoop of the fastest stuff right next to the CPU, a bucket of medium-fast stuff a little further out, and a swimming pool of slow-but-cheap stuff for everything else.

The hierarchy exists because we can't have one memory that's fast, big, and cheap. We have to lie to the CPU — pretend that the slow tier is fast, by keeping a copy of the hot bits one tier up.

02The eight tiers, top to bottom

From the chip outward. Sizes and timings are 2026 ballpark for a typical server CPU; treat them as orders of magnitude, not specs.

Figure 01 — The pyramid

A real pyramid. Each tier is roughly 10–1,000× slower than the one above and ~10–100× larger. The CPU sits at the top and pulls bytes downward only when forced to — every miss into a lower tier is a tax.

Registers — the CPU's hands

A few hundred bytes living inside the CPU itself. Sub-nanosecond access — they don't have a "latency" so much as they're the inputs to the next instruction. Every value the CPU operates on must be in a register; loads and stores exist precisely to move things in and out. Compilers spend enormous effort minimizing register pressure.

L1 cache — the snack drawer

32–64 KB per core, ~1 ns access. Split into separate instruction and data halves (L1i / L1d). This is what makes loops fast — touch the same data again and it's already here. Miss it and you fall to L2.

L2 cache — the pantry

256 KB to 1 MB per core, ~3 ns. A bigger backstop for L1 misses. Still per-core on modern Intel and AMD chips. Big enough to hold a hot inner data structure but not your whole working set.

L3 cache — the shared fridge

8–64 MB, ~10 ns, shared by all cores on the same socket. This is where cores hand data to each other (and where they fight over it). A cross-core read of "your" L1 line goes through L3 — that's what makes shared-mutable concurrency expensive.

Main memory (DRAM) — the kitchen

Tens to hundreds of GB, ~80 ns, ~100 GB/s of bandwidth on modern DDR5. Big enough to hold most working sets; slow enough that a cache miss is a noticeable stall. The CPU literally idles for ~250 cycles waiting on a DRAM read — which is why caches exist.

NVMe SSD — the basement

1–8 TB, ~30 µs random read, several GB/s sequential. Persistent. Fast enough that databases live here happily; slow enough relative to RAM that databases also try to keep their hot pages cached in DRAM. The tier where "in memory" stops being true.

HDD & network storage — the warehouse

Tens of TB, ~10 ms per random seek, ~150 MB/s sequential. The mechanical floor. S3 and similar object stores live in this neighbourhood latency-wise even though they're SSD-backed, because the network round-trip dominates. Cheap, durable, slow.

Tape & cold archive — the off-site vault

Petabytes, seconds to minutes of access latency, fractions of a cent per GB-month. You don't read this tier in a request path; you read it for compliance, recovery, or analytics on years-old data. The end of the line.

03L1, L2, L3 — three caches, three jobs

The three CPU caches aren't just "more of the same memory." They have different sizes for different jobs, and the differences are deliberate.

Figure 02 — Inside one CPU socket

L1 and L2 are private — each core has its own. L3 is shared across the socket and is where cores synchronize. A cache line written by one core has to travel down to L3 (or worse) before another core can read it; that's the hidden cost of shared-mutable state.

L1 — small, fast, split

L1 is tiny on purpose. Smaller circuits switch faster and decode faster; that's how it stays at ~1 ns. It's also split: separate L1i (instructions) and L1d (data), so the CPU can fetch the next instruction and the data it operates on in the same cycle without contention. Almost every cycle of a tight loop runs entirely out of L1.

L2 — bigger, single, still private

L2 is the L1 backstop: bigger (so more fits) but slightly slower (because bigger circuits take longer to address). Unified — instructions and data share it. Still per-core, so no contention with neighboring cores. If your inner loop's working set fits in L2 you'll be fast; if it doesn't, every L1 miss falls to ~3 ns.

L3 — large, shared, the rendezvous

L3 is the meeting point. All cores on the socket can see it, which makes inter-core communication cheap-ish (~10 ns) but introduces coherence: when one core writes to an address, every other core's L1/L2 copies of that line must be invalidated or updated. That protocol — MESI, MOESI, or a variant — is invisible until you write multi-threaded code, at which point it becomes the dominant cost. False sharing is what happens when this goes wrong.

Why three levels and not four?
Each level has to be enough faster than the next to be worth maintaining a separate copy. Three has been the sweet spot for ~15 years; some chips experiment with a smaller "L0" micro-op cache or a victim L4 in DRAM, but the three-tier on-chip arrangement has stuck because it matches workload sizes well: tight loops in L1, hot data structures in L2, working sets in L3.

04What's the "fastest" storage?

Trick question — the answer depends on how you define "storage." Each candidate has a case:

Candidate Latency Capacity "Storage"?
Registers < 0.3 ns ~few hundred bytes arguably not — they're CPU operands
L1 cache ~1 ns 32–64 KB yes — first thing called "memory"
DRAM ~80 ns tens to hundreds of GB yes — what most people mean by "memory"
NVMe SSD ~30 µs terabytes yes — first persistent tier

If "storage" means "things you address with a pointer and treat as memory," registers are the fastest. If it means "the first tier you'd recognize as memory, with byte addresses you can store data into," L1 cache is the fastest. If it means "memory the way most programmers talk about it," that's DRAM. And if it means "the fastest persistent storage" — keeping data through a power cut — the answer flips entirely to NVMe SSD, which is six orders of magnitude slower than L1 but the first tier that survives.

The honest one-line answer to "what's the fastest storage in a computer?" is "registers, by definition; L1 cache, in practice." The interview-friendly answer is "registers." Either way, the lesson is that "fastest" is a question about the unit of measurement.

05Speed vs size vs cost

Why doesn't everyone just have 1 TB of L1 cache? Because the trade-off is brutal in both directions. Here's the rough shape, in 2026 dollars per gigabyte:

Figure 03 — Cost falls as you go down

Each tier is roughly 10–100× cheaper per byte than the one above and roughly 10–1,000× slower. The pyramid isn't an aesthetic choice — it's the cheapest way to get "enough fast bytes" without paying SRAM prices for everything.

Why fast is expensive

L1's SRAM uses six transistors per bit and has to be physically close to the CPU's execution units — every micrometer of distance adds picoseconds of signal travel. DRAM uses one transistor and one capacitor per bit (cheaper, denser) but has to be refreshed and lives further from the CPU. Flash uses a charged floating gate (no power needed to retain) but switching it is slow. Magnetic platters store bits as patches of polarity on a moving surface — basically free per byte, terrible to read.

Why big is slow

This one isn't intuitive but matters: any memory grows slower as it grows bigger. The address decoder takes longer to select a row, the bus has more capacitive load, the wires get longer. That's why we don't just build a 1 GB L1 — even ignoring cost, it would be slower than L2 because of pure circuit physics.

Why every architecture looks like this

Smartphones, laptops, servers, GPUs, microcontrollers — they all have some flavour of the same hierarchy. The numbers shift but the shape doesn't, because the underlying physics doesn't. A GPU has registers, shared memory (its L1), L2, and HBM (its DRAM). A microcontroller might collapse to two tiers (registers + flash) but the pattern is there.

06Ephemeral vs persistent

There's a hard line drawn through the pyramid: above it, everything dies when power is cut. Below it, everything survives. Knowing which side of the line a piece of state lives on is one of the most consequential design decisions in any system.

Figure 04 — The volatility line

Everything above the line is a copy. The truth — the persistent state of a system — is whatever lives below the volatility line at the moment the power dies. Designs that confuse this are the ones that lose data on outages.

What "ephemeral" actually means

An in-memory cache, a session table in RAM, a Redis instance without AOF, a Kubernetes pod's local disk in some setups — all ephemeral. Useful, fast, and conditionally truthful. They're conceptually derived from persistent state and rebuildable if lost. If your system can't rebuild them, they aren't ephemeral; they're a data-loss bug waiting for an outage.

What "persistent" actually buys

SSDs and HDDs hold their bits without power because they store them as physical state — trapped charge in a flash cell, magnetic polarity on a disk surface. That's why writes to them have to reach the medium before they're durable. fsync exists because the OS, by default, lies to you: a write returns "done" when it hits a buffer, not when it hits the platter. Every database that calls itself durable is really a careful conversation with that lie.

The expensive lesson
Every "we lost data when the box rebooted" incident is a misplaced piece of state that lived above the volatility line when the team thought it lived below. Always ask: if power dies right now, what does this system have?

07The one-line takeaway

The memory hierarchy is the trade-off between speed, size, and cost made physical. Every performance decision is really a question of which tier you're spending on.

The hierarchy isn't a list of components — it's a vocabulary. "This is hot, keep it in L2." "That's the working set; fit it in DRAM or accept the page-cache miss." "This must survive a crash; it has to land on disk before we ack." The same conversation, at every layer of every system you'll ever build. Once the pyramid is in your head, all those decisions become the same shape.
