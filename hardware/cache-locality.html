Cache locality & branch prediction

Big-O lies on real hardware. Two algorithms with the same complexity can differ by 10× because one walks memory in order and tells the CPU's predictor what to expect, and the other doesn't.

There's a video that's been baffling new programmers for fifteen years: take a million random integers, run an if (x > 128) sum in a loop, and time it. Then sort the array first and run the exact same loop. The sorted version is several times faster. Same algorithm. Same data. Same machine. Sorting added work and the program got faster. The explanation isn't a trick — it's two of the most important ideas in modern hardware: cache locality (the CPU is fast because it bets on which bytes you'll touch next) and branch prediction (it's fast because it bets on which way if statements will go).

This page is about those two bets, why they pay off so massively, and what it means for the way we write code.

01Why two same-O algorithms differ by 10×

Big-O analysis pretends every memory access is the same cost and every if takes the same time. On real hardware that's wildly untrue. A cache hit is ~1 ns; a DRAM access is ~80 ns — that's an 80× gap inside what a textbook calls "constant time." A branch that the CPU correctly predicts costs zero extra cycles; a misprediction costs 10–20 cycles of wasted work the CPU has to throw away. Multiply those gaps by a tight loop running a billion times and you get the difference between a function that finishes in 0.3 seconds and one that finishes in 3.

Big-O describes the algorithm. The hardware decides the constant. On modern CPUs, the constant can dominate.

The two big knobs hiding in the constant are the same two knobs every modern CPU was built around: caches (so it doesn't have to wait for DRAM) and pipelines (so it can be working on the next instruction before this one finishes). Both work by guessing — what data you'll touch next, and which way each branch will go. When the guesses are right, the machine flies. When they're wrong, it stalls.

02Cache lines, not bytes

The first thing to internalize: the CPU doesn't read one byte at a time, ever. The smallest unit of memory traffic between any two tiers of the hierarchy is a cache line — almost universally 64 bytes. If you read arr[0], the CPU pulls in arr[0..15] (assuming 4-byte ints) at the same cost. The next 15 reads are essentially free because they're already in L1.

Figure 01 — One read pulls in 64 bytes

A "memory access" is really a cache-line access. One DRAM hit pays for 64 bytes; the program that uses all 64 wins, and the program that uses 4 of them and jumps somewhere else loses 15 free reads.

That single fact rewrites how you think about loops:

  • Iterating an array forwards — every 16th read is a real DRAM hit; the other 15 are L1. You're using the line you paid for.
  • Iterating an array with stride 16 — every read jumps to a new line. Every read is a DRAM hit. Same algorithm, ~10× slower.
  • Walking a linked list — each node is somewhere different in memory. Every node->next dereference is potentially a fresh cache miss.
  • Hash-table probe over scattered buckets — same problem as the linked list. The "O(1) lookup" can hit DRAM 5 times per call if the buckets aren't co-located.
Mechanical sympathy
Programs that respect how the hardware moves data are dramatically faster than ones that fight it. Same algorithm; same machine; better layout.

03Spatial and temporal locality

Caches work because real programs touch memory in two predictable ways. CPU designers gave them names so we could talk about exploiting them.

Spatial locality

If you touched address X, you'll probably touch X+1, X+2, X+3 soon. The CPU exploits this by pulling in a whole 64-byte line per access. Sequential array iteration is the perfect case.

Temporal locality

If you touched address X, you'll probably touch X again soon. The CPU exploits this by keeping recently-used lines in L1/L2 instead of evicting them. Loops that read the same lookup table on every iteration are the perfect case.

Most "this is fast" code is fast because it has one or both. Most "why is this slow?" code is slow because it has neither — it jumps to fresh, unrelated addresses, never benefits from a prefetched line, and never reuses a cached one.

Layout choices that change locality

The same logical data can be laid out in memory in ways that have wildly different cache behaviour. The two famous patterns:

Pattern Layout Best for
Array of Structs (AoS) {x,y,z,name},{x,y,z,name},... touching all fields of one record at a time
Struct of Arrays (SoA) x[..],y[..],z[..],name[..] touching one field across all records

If your loop reads name for every record but ignores x, y, z, AoS wastes 3/4 of every cache line on data you never use. SoA packs all the names contiguously so each line you pay for is full of names. Game engines and columnar databases obsess over this for exactly that reason.

04Branch prediction and pipelining

The other half of "fast on real hardware" is the CPU's pipeline. A modern core doesn't run instructions one at a time; it has a deep pipeline (15–20 stages) of instructions in flight — fetched, decoded, scheduled, executed, retired — all overlapping. To keep that pipeline full, it has to know what instruction to fetch next. And whenever it hits an if, it doesn't know — not yet. The condition won't be evaluated for another 10+ cycles.

So the CPU guesses. It speculates which branch will be taken, fetches and decodes ahead on that path, and starts executing those instructions on the bet. If the guess was right, those instructions are committed and the pipeline stays full. If the guess was wrong, the pipeline is flushed — every speculative instruction is thrown away — and the CPU restarts from the correct path. That flush costs ~10–20 cycles. Cheap if it's rare; ruinous if it's not.

Figure 02 — Predicted right vs predicted wrong

A correctly predicted branch is essentially free — the speculative instructions become real ones. A misprediction wipes out everything in flight. Real CPUs maintain branch-prediction tables that can hit 95%+ accuracy when the data has any regularity.

What predictors actually do

Modern predictors maintain a small table indexed by branch address, recording whether each branch was recently taken or not, plus more elaborate history of preceding branches. They're remarkably good — >95% accuracy on typical code — because real branches are usually biased: a loop's exit branch is taken 999 times out of 1000, an error-handling branch is taken almost never, and a sorted-data branch flips at exactly one point and never again.

The hostile case is a branch that flips randomly — neither side wins, no history helps, the predictor effectively coin-flips and is wrong half the time. That's the case the next section is built around.

05The famous sorted-array experiment

This is the experiment that anchors the whole topic. Take an array of a million random ints in [0, 256). Run a tight loop that adds up only the ones above 128:

long sum = 0; for (int i = 0; i < n; i++) { if (data[i] >= 128) sum += data[i]; }

Run it on shuffled data. Then sort the array and run the identical loop. The sorted version finishes 3–6× faster. Why?

It's not the cache. Both versions touch the same bytes, in the same order, with perfect spatial locality. Both should be cache-hot. The difference isn't memory — it's the branch.

Figure 03 — The branch each version sees
n log n work up front; the loop saved more than that. Net: doing more total work made the program faster. — this is the trick

Orange = "branch taken" (value ≥ 128). On shuffled data, the predictor sees nearly random outcomes and is wrong about half the time. On sorted data, the branch goes "false, false, false, ..., true, true, true" — one transition in the whole array. The predictor is essentially never wrong, the pipeline never flushes, and the loop runs at peak speed.

This is the canonical demonstration of the topic — it's the famous Stack Overflow question "Why is processing a sorted array faster than processing an unsorted array?" The video at the top of this class works through the same experiment with profilers and assembly to make the cause visible. The headline is: doing extra work (sorting) added so much branch predictability that the inner loop ran several times faster, which more than paid for the sort.

The branchless escape hatch
You can also fix the slow version without sorting — by writing the comparison branchlessly, e.g. sum += (data[i] >= 128) * data[i] or with cmov. With no branch to predict, the predictor has nothing to be wrong about. Both fixes — sort the data, or remove the branch — point at the same culprit: an unpredictable branch was the bottleneck.

06Why "inefficient" sorts win on small arrays

The same two effects — cache locality and branch predictability — explain a result that sounds backwards: on small arrays, an O(n²) sort like insertion sort beats an O(n log n) sort like quicksort or mergesort. Every serious sorting library (Python's Timsort, Java's Arrays.sort, Rust's sort_unstable, libc++ std::sort) detects "small subarray" and switches to insertion sort, typically below ~16 elements. The asymptotically-worse algorithm wins.

Cache & pipeline favor insertion sort

Insertion sort walks the array linearly, comparing adjacent elements. Perfect spatial locality — every read is in L1. The inner branch ("is the new element smaller?") gets very biased once the array is partially sorted, so the predictor is correct nearly always.

Quicksort pays overhead per call

Quicksort recurses, which means stack frames, function calls, and pivot-selection logic on every recursion. For n=8 those fixed costs dwarf the algorithmic savings. The "log n fewer comparisons" matter only when n is large enough for them to outweigh the per-call tax.

This generalizes far beyond sorting. The same pattern shows up everywhere on real hardware:

  • Linear scan beats binary search for arrays under ~64 elements. Linear has perfect spatial locality and a predictable branch; binary search jumps around (no spatial locality) and the comparison branch is unpredictable.
  • Linear scan beats hash table for arrays under ~10 elements. The hash and the indirection cost more than just looking at all of them.
  • std::vector beats std::list almost always, even for "list-friendly" workloads, because contiguous memory crushes the cache penalty of pointer-chasing.
  • Bubble sort can beat selection sort on partially-sorted data because its branch becomes massively biased once most of the array is in order.

Big-O tells you which algorithm wins as n → ∞. For the n you actually have, the constants matter — and the constants are mostly cache and branches.

The "inefficient" sort isn't really inefficient

It's only inefficient on the model that pretends every operation costs the same. On the actual machine, where memory is hierarchical and branches are speculated, "more comparisons in a tight, sequential, predictable loop" is sometimes the cheapest thing the CPU can possibly do. The textbook complexity is comparing the algorithms in a lab where the hardware is fictional. In the real lab — your laptop — the constants are the story.

07The one-line takeaway

Caches reward sequential, repeated access. Pipelines reward predictable branches. Code that respects both is sometimes 10× faster than equally "efficient" code that doesn't.

Two practical habits fall out of this. First, when something is unexpectedly slow, check for cache misses and branch mispredictions before you hunt for an asymptotically better algorithm — they're the two things big-O is silently lying about. Second, when you're choosing data structures, weigh the layout, not just the operations: an "O(1) lookup" hash table that scatters its entries across DRAM can lose to an "O(n) scan" of a packed array, every single time, for the n you actually have.

The deeper point is that modern CPUs are fast because they bet — on what data you'll touch and which branch you'll take. The job of fast code isn't to outsmart those bets; it's to set them up so the CPU is right.
