The JavaScript event loop

One thread, one stack, two queues. The whole machinery JavaScript uses to feel concurrent without ever being parallel — and the trap that breaks the moment any single task takes too long.

JavaScript runtimes — the browser, Node.js, Deno, Bun — all share a strange property: they run user code on exactly one thread. No locks, no race conditions, no parallelism. And yet they handle thousands of concurrent connections, animations, network calls, and timers without breaking a sweat. The thing that makes this work is the event loop: a tiny scheduler that picks one task at a time off a queue, runs it to completion on the single thread, and goes back for the next. It's the canonical example of concurrency without parallelism — and it has its own peculiar gotchas, the kind you only understand once you've seen the loop diagrammed.

This page walks the model: what one thread really means, what the queues are, what one tick does, and where people stub their toes (microtasks vs macrotasks, the lying setTimeout(0), and the cardinal sin of blocking the loop).

01Single-threaded by design

JavaScript was created in 1995 to make web pages interactive. The decision to make it single-threaded wasn't a limitation — it was a feature. Multi-threaded code with shared mutable state is famously hard to get right; race conditions, deadlocks, and torn reads are common bugs even for experienced engineers. JavaScript's authors picked a different deal: only one piece of code runs at a time, and any "concurrent" feeling has to come from how cleverly that one piece of code is broken up.

The trade-off works because most of what a browser does is wait — wait for a click, wait for a network response, wait for a frame to render. While the program is waiting, the thread is idle. The event loop is the machinery that says: "while you're waiting, I'll come grab the next thing on the queue and run it." A single thread is plenty if it's never idle and never blocked.

"Single-threaded JavaScript" is more nuanced
The JavaScript user code runs on one thread. Behind the scenes, the runtime delegates I/O, timers, file reads, and DNS lookups to other threads or to the OS — Node uses libuv, browsers use a constellation of internal threads. Those threads don't run your code; they just call back into the event loop with results. From the JS engineer's seat, it's still one thread.

02Stack, queues, and the loop

The full model has four pieces: the call stack (where the running function lives), the macrotask queue (incoming work like timers and I/O callbacks), the microtask queue (a higher-priority lane for promise reactions), and the loop itself that moves work between them.

Figure 01 — The four pieces

The call stack runs the current task. The microtask queue holds promise reactions. The macrotask queue holds timer callbacks and I/O completions. The loop empties the stack, drains all the microtasks, then takes one macrotask, and starts again. That ordering is the cause of nearly every "but I scheduled it first…" surprise in JavaScript.

03One tick of the loop

One iteration of the event loop — one "tick" — does these steps in order:

  1. Run one macrotask from the macrotask queue. Pop it off, push its function onto the call stack, and run until the stack is empty (the function has returned).
  2. Drain the microtask queue. Run every microtask, in order. If a microtask schedules another microtask, run that one too — keep going until the queue is empty.
  3. Render (browser only). If it's time for a frame, the browser does layout, paint, and compositor work. Any requestAnimationFrame callbacks fire here.
  4. Go back to step 1. Pick up the next macrotask. Repeat forever.

Two consequences fall out of this. First, microtasks have priority over macrotasks — a chain of Promise.then(...).then(...).then(...) all runs before the next setTimeout fires, even if the timer has been waiting longer. Second, a long-running synchronous task blocks everything. If a macrotask takes 5 seconds to run, the browser doesn't render, no clicks are processed, no promises resolve, no timers fire. Whoever is staring at the page sees a frozen screen for 5 seconds.

04Microtasks vs macrotasks

The split between the two queues is the most-asked question about the event loop, and it's the source of the trickiest behaviour to predict. The TL;DR:

Type Microtask Macrotask
Source Promise.then, queueMicrotask, MutationObserver setTimeout, setInterval, I/O, DOM events
When After current task, before next macrotask One per loop tick (after microtasks drain)
Drain rule All of them, including ones added during draining Exactly one per tick

The classic puzzle that catches people:

console.log('1'); setTimeout(() => console.log('2'), 0); Promise.resolve().then(() => console.log('3')); console.log('4');

The output is 1, 4, 3, 2 — not 1, 4, 2, 3, even though the setTimeout was scheduled before the promise. The synchronous code runs to completion (1, 4). The current task ends. The runtime then drains microtasks (3). Only after the microtask queue is empty does it pull the next macrotask (2). A microtask that finishes "after" a 0 ms timer always runs first.

Why two queues?
Microtasks exist so that promise chains and observers can complete an entire reaction before the runtime returns control to the outside world (next timer, next event, next render). It guarantees that by the time a paint happens, every .then() set up by the most recent code has resolved. The cost: a runaway microtask chain (resolve().then(() => resolve().then(...))) can starve macrotasks indefinitely — the loop never gets to step 3 (render) because step 2 (drain microtasks) never finishes.

05Why setTimeout(0) isn't 0

setTimeout(fn, 0) doesn't run fn immediately. It schedules fn as a macrotask and adds it to the queue. fn can only run after (a) the current task finishes, (b) all microtasks drain, and (c) the macrotask queue ahead of it has cleared. Even on a quiet runtime, browsers historically clamp the minimum to ~4 ms (a HTML spec rule) when timers are nested deeper than five levels, so setTimeout(fn, 0) often actually fires after 4 ms or more.

The mental model that makes this make sense: timers are about "no sooner than," not "exactly at." The 0 argument means "the runtime is allowed to run this as soon as it can find a slot." It does not promise immediacy. If you want to defer something to "as soon as the current code is done but before the next render," queueMicrotask(fn) is the tool — that's a microtask, not a macrotask, and it runs in the same tick before the loop hands control back to the browser.

06Never block the loop

The single most important rule of writing JavaScript is also the simplest: do not run synchronous code that takes a long time. The loop is single-threaded; everything is paused while your code runs. Five seconds of synchronous work is five seconds where the user can't click, no animation frames render, no XHR responses are processed, no promises resolve. The page is dead until your function returns.

Figure 02 — A blocked loop is a dead loop

During a long synchronous task, the loop can't tick. Every queued macrotask waits, microtasks pile up, and the page is frozen from the user's perspective. Even one slow JSON.parse on a big payload, or one unbounded loop in a render function, is enough to drop frames.

What "blocking" looks like in practice

  • Parsing a 50 MB JSON document with JSON.parse on the main thread.
  • A tight loop computing a hash over a large file synchronously.
  • A regex with catastrophic backtracking on user input.
  • Calling a synchronous third-party library that does heavy work without yielding.

The fixes

Break work into chunks

Process the first 1,000 items synchronously, then queueMicrotask or setTimeout(_, 0) the rest. Each chunk gives the loop a chance to tick — handle UI events, render frames, fire timers. Total time is similar; responsiveness is dramatically better.

Move it off-thread

For genuinely heavy CPU work, use a Web Worker (browser) or worker_threads (Node). That's a separate OS thread with its own event loop and its own JS context — true parallelism. Communication is by message-passing (no shared state), so the model stays clean.

A frozen page is almost always one synchronous loop that didn't yield. Fix it by yielding, or by moving the loop to another thread.

07The one-line takeaway

The event loop is JavaScript's whole answer to concurrency. One thread, one stack, two queues, and a rule: drain microtasks between every macrotask, and never let any task run for too long. When you keep that rule, a single core handles thousands of in-flight things gracefully. When you break it, the page goes dark.

This is the canonical example of concurrency without parallelism — many things in flight, exactly one running at a time. Languages with similar models (Python's asyncio, Erlang's actor system on a single scheduler, async Rust on a single-threaded executor) all have a version of the same loop. The vocabulary differs; the constraints don't.
