Process vs thread

A process is the OS's unit of isolation. A thread is the OS's unit of execution. Most "should I use threads or processes?" questions are really about which kind of safety you can give up.

When a program runs, the operating system has to track two different things about it: what memory it's allowed to touch, and where in the code it currently is. The first is a process — a sealed box of memory, file descriptors, and permissions. The second is a thread — a position in code, with a stack and a register set, executing instructions. Every running program has at least one of each. The interesting question is what happens when you add more.

This page is about the difference between adding another process and adding another thread — what they each cost, what they each protect you from, and why most "use threads or processes?" arguments are really arguments about isolation versus speed.

01Why ask

"Process" and "thread" get used interchangeably in casual speech and they shouldn't be. They're the OS's two different answers to two different questions:

"Whose memory is this?"

The OS answers with a process. Each process has its own private address space — a piece of virtual memory the kernel guarantees no one else can read or write. Crash one process, the others keep running. That isolation is what a process buys you.

"What is the CPU doing right now?"

The OS answers with a thread. A thread is a sequence of instructions being executed, with its own stack and register state. A process needs at least one thread (otherwise nothing's running); it can have many.

Threads live inside processes. A process with one thread is a single sequential program. A process with eight threads is eight cursors moving through code at the same time, all reading and writing the same memory. That's where the speed comes from — and that's where the danger comes from.

02What a process is

A process is everything the OS needs to keep one running program separated from every other running program. When you launch chrome and postgres on the same machine, the OS is the thing keeping their bytes apart.

Figure 01 — Anatomy of a process

A process owns its memory and its OS-level resources. The kernel uses page tables to enforce that one process can't read another's address space — try and you get SIGSEGV. Inside the process, at least one thread is what actually runs the code.

Three properties of a process are load-bearing:

  • Isolation. Memory and resource permissions are enforced by the OS. A bug in one process can't corrupt another's memory.
  • A separate "view" of the world. Each process has its own file descriptor table, its own current working directory, its own environment variables. Two processes both opening "file 5" are looking at different files.
  • One execution context required. A process is conceptually a container; it needs at least one thread to actually do anything.

Two processes that need to talk to each other have to use inter-process communication (IPC) — pipes, sockets, shared memory regions, message queues. The data has to leave one address space and re-enter another. That's slower than two threads sharing a pointer, and it's the whole point.

03What a thread is

A thread is a single thread of execution — one cursor moving through one program's code. It owns very little, but what it owns is precious: a stack (its function calls and local variables), a set of registers (the values the CPU is currently working on), a program counter — a special register that's literally a "you are here" pointer to the next instruction — and some scheduling metadata. Everything else — the heap, globals, open files — it shares with every other thread in the same process.

Figure 02 — Threads inside one process

Every thread has its own stack (function calls, locals) and its own registers (where it is in code). Everything else — the heap, globals, open files — lives in the process and is shared. Pass a pointer between threads and they're both pointing at the same byte.

The shared heap is the source of every advantage and every problem of threading. Two threads can communicate by writing to the same variable — no IPC, no copying, just memory. That's astonishingly fast. It's also the source of race conditions: if both threads write at the same moment with no coordination, the result is undefined. Locks, atomics, channels, and concurrent data structures all exist to manage this hazard. The hazard isn't an edge case; it's the central engineering tax of threads.

Why "threads share, processes don't"
Threads share the heap because they live in the same address space. Processes don't share the heap because their address spaces are different — process A's pointer 0x7fff1234 and process B's pointer 0x7fff1234 resolve to two different physical pages. The OS deliberately built it that way. Threads are an opt-in to give up that protection for speed.

Where the state lives when paused

The figure above shows each thread's "private kit" as a tidy box. Where does that box actually live in the machine? The answer is split, and the split is where the cost of threading comes from.

While a thread is running, its private kit lives where it has to: in the core's register file. Registers are the fastest storage on the machine — sub-nanosecond access — but they're tiny (a few hundred bytes) and there's exactly one set per core. The register file holds one thread's state at a time; it has no room for two.

That raises an obvious question. If one core has one register file, but a system has dozens of OS threads, what happens to the ones that aren't currently running? Their state has to be saved somewhere the registers can be reloaded from later. The OS keeps that snapshot in the Process Control Block (PCB) — a per-thread struct in RAM that holds the thread's saved registers, program counter, stack pointer, and scheduling state until the thread runs again. Every paused thread has a PCB sitting in kernel memory waiting for it.

Figure 03 — Save / restore: the irreducible direct cost

A context switch swaps Thread A out and Thread B in by saving A's registers down into PCB-A and loading PCB-B up into the registers. Each side of the swap is a CPU↔RAM round-trip — registers at ~0.3 ns, RAM at ~80 ns, two orders of magnitude apart. That gap is the irreducible direct cost of a context switch. Cold caches and TLB misses pile on after the switch — see context switching for the full bill.

The takeaway: a thread's "private kit" is conceptually private but physically split. While running, it's in registers. While paused, it's in a PCB in RAM. The OS spends real cycles moving state between the two on every scheduling decision — and the further apart the two locations are on the memory hierarchy, the more those cycles cost.

Race conditions don't go away with parallelism
Hyper-threading and multi-core make threads more physically simultaneous, not less — two threads can write to the same shared variable on the same nanosecond. Locks and atomics aren't there to slow the CPU down; they're there because the contention is over data, and data races exist whether the threads are interleaved on one core or running truly in parallel on two. The hazard is in the heap, not in the schedule.

04Core vs thread (the terminology collision)

One catch worth flagging before going further: the word "thread" is overloaded. The OS thread we just described is the software unit — a cursor through code that the kernel schedules. But your CPU also has things it calls threads, and they're a completely different idea. Conflating the two is one of the most common sources of confusion in performance and capacity-planning conversations.

Cores are the hardware

A modern CPU socket contains multiple physical cores. Each core is a complete execution engine: its own arithmetic units (ALU, FPU), its own L1 and L2 caches, its own branch predictor, its own register file, its own instruction front-end. One core can run one stream of instructions at full speed. Cores are silicon — adding more requires buying a different chip.

Hardware threads — same core, second register file

Most modern server CPUs (every Intel since the early 2000s, every AMD Zen) implement simultaneous multi-threading (SMT) — Intel brands it Hyper-Threading. Each physical core is augmented with a second (sometimes fourth) hardware thread: an extra register file and program counter, but no extra ALUs, caches, or front-end. The core can hold two execution contexts at once and pick which one to issue from on every clock cycle.

Why? Because real instructions stall. A cache miss, a branch mispredict, a dependency wait — the core would otherwise sit idle for those cycles. With two hardware threads, when one is stalled the core issues from the other. The execution units stay busy; the core's effective throughput rises ~20–40% on most workloads, at the cost of a small bump in die area.

There's a sharper way to read SMT: it's a hardware shortcut around the OS context switch. When the kernel switches between two OS threads on a single-register-file core, both threads have to round-trip through RAM via the PCB (Figure 03). With two hardware threads, both contexts are already on-chip — the core flips between them in a single cycle, no save, no load, no kernel involvement. SMT in one sentence: duplicate the register file so the second context never has to leave the chip.

Figure 04 — One physical core, two hardware threads

A single physical core with 2-way SMT. Each hardware thread has its own program counter and register file — that's enough to hold an entire execution context. Everything else — execution units, L1, L2, branch predictor, TLB — is shared. The core picks instructions from whichever HT can make progress that cycle.

What the OS sees

The OS sees hardware threads as logical CPUs — the slots it can schedule OS threads onto. A box with 8 physical cores and 2-way SMT presents 16 logical CPUs to Linux. lscpu makes the topology explicit:

$ lscpu CPU(s): 16 # logical CPUs (HT slots) Thread(s) per core: 2 # SMT degree Core(s) per socket: 8 # physical cores per socket Socket(s): 1

When you launch an OS thread, the kernel scheduler picks one of those 16 logical CPUs and runs your code there. Migration moves the thread from one logical CPU to another (often a different physical core). The OS thread is the unit you control; the logical CPU is where it ends up running.

Why the distinction matters

Two HT siblings on the same physical core are not two independent cores. They share execution resources. For pure CPU-bound work that fully uses the execution units, two threads on the same core get only ~1.2–1.4× the throughput of one — not 2× — because they're competing for the same ALUs. This shows up in three places:

  • Capacity planning. A "16-thread" CPU can run 16 OS threads concurrently, but its peak throughput is closer to 8 cores' worth, not 16. Marketing numbers count HTs; performance models should count physical cores.
  • Latency-critical pinning. If you pin a hot thread to a physical core, also reserve (or pin away) its HT sibling — otherwise a noisy neighbour will compete for the core's execution units and your tail latency will spike. See core pinning.
  • Security. The shared L1 between HT siblings is the channel side-channel attacks like L1TF and MDS exploit. Sensitive workloads (cryptography, multi-tenant clouds) sometimes disable HT entirely for that reason.

One word, three things

To finish untangling: when someone says "thread," they probably mean one of three things, and asking which one resolves nine out of ten arguments before they start.

Term What it actually is Bound by
Physical core Independent execution engine — its own ALU, L1, branch predictor Silicon: how many cores the chip has
Hardware thread (SMT/HT) Per-core execution slot — its own register file + PC, shared everything else Chip topology — usually 1× or 2× physical cores
OS thread (software thread) Execution cursor scheduled by the kernel onto a logical CPU RAM and scheduling overhead — practically tens of thousands per box

The OS thread is software; the hardware thread and core are physical. Adding OS threads is cheap. Adding hardware threads or cores requires buying different silicon.

A useful sentence to internalize
"We have 16 threads" almost always means 16 hardware-thread slots (8 cores × 2-way SMT). "We're running 16 threads" almost always means 16 OS threads. The first is a fact about the chip; the second is a fact about the program. They should never be in the same sentence without disambiguation.

05The five differences that matter

Dimension Process Thread
Address space Private, isolated Shared with siblings
Communication IPC: pipes, sockets, shm, queues Shared variables (need locks)
Crash blast radius One process; others survive Whole process; all threads die
Creation cost ~0.5–5 ms (fork on Linux) ~10–100 µs
Context-switch cost Higher — TLB flush, full state swap Lower — same address space, no TLB flush

The shape of every choice falls out of these five rows. Threads are cheaper, faster, and tightly-coupled — perfect for sharing data and worst at containing failures. Processes are more expensive, more isolated, and harder to coordinate — perfect for safety boundaries and worst at high-frequency communication.

06Creation cost & the COW trick

"Processes are expensive to create" used to be a hard rule. On modern Linux it's softer, because of a clever trick called copy-on-write.

When a process calls fork(), the child gets a copy of the parent's address space. Naively, that means duplicating gigabytes of memory — slow and wasteful. Linux instead marks every page as shared and read-only; the child and parent point to the same physical pages. Only when one of them writes to a page does the kernel actually duplicate it. If the child immediately calls exec() to replace its memory with a new program (the typical "fork-exec" pattern), the duplicates were never made.

Threads skip even that overhead. Creating a thread is essentially: allocate an 8 KB stack, set up a register frame, add an entry to the scheduler queue. Tens of microseconds, not milliseconds. Modern languages add their own layer on top — green threads, fibers, goroutines, Java virtual threads — that compress thread cost further into the nanoseconds-to-microseconds range by multiplexing many language-level threads onto fewer OS threads.

Lightweight threads ≠ free
Goroutines, virtual threads, and fibers cost less than OS threads but they're not free. They still need stacks, scheduling, and synchronization. The win isn't "threads cost zero" — it's "threads cost little enough that you can have a million of them and let the runtime sort it out."

07Picking one

The decision tree, in order of priority:

Pick processes when…

You need isolation. A crash in component A must not bring down component B. Browser tabs (Chrome's per-tab process), database connection workers (Postgres), serverless function invocations, and anything running untrusted code all default to processes for this reason.

Pick threads when…

You need cheap shared state. Workers all reading the same in-memory cache, a thread pool processing items off the same queue, a request handler offloading work to a parallel computation. Threads' headline feature is that the data doesn't have to move.

Pick processes when…

Your runtime has a Global Interpreter Lock. Python's GIL serializes Python bytecode across threads, so for CPU-bound work the only way to use multiple cores is multiple processes (multiprocessing). Threads still help for I/O-bound work because the GIL releases on syscalls.

Pick threads when…

The work is I/O-bound and the runtime supports it cheaply. Java virtual threads, Go goroutines, Erlang processes (which are misleadingly named — they're more like green threads). Tens of thousands of them can wait on I/O with negligible overhead.

The historical context helps: web servers used to spawn one process per request (CGI), then one thread per request, then a fixed thread pool, then async / event-loop models running on a single thread. Each step traded isolation for throughput because the per-unit cost was the bottleneck. The pendulum swings; the trade-off doesn't.

Threads share by default and isolate by effort. Processes isolate by default and share by effort. The "right" choice is whichever default matches the failure mode you're most afraid of.

08The one-line takeaway

A process is a memory boundary; a thread is an execution cursor. You add a process when you want safety. You add a thread when you want speed. Most production systems use both — processes for the architecture (one per service, one per tab, one per worker pool) and threads for the parallelism inside each one.

One last subtlety: more threads or more processes does not automatically mean more parallel execution. That requires multiple cores and a kind of work that can be split. Concurrency (the structure of having multiple things in flight) and parallelism (the execution of multiple things simultaneously) are different ideas, and they're worth their own page — see concurrency vs parallelism.
