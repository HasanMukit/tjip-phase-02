Concurrency vs parallelism

Concurrency is having many things in progress. Parallelism is doing many things at once. They look the same on a busy graph and they aren't the same idea.

Two of the most important words in software engineering are routinely treated as synonyms by people who should know better. Concurrency and parallelism describe different things — one is about how the program is structured, the other about how the hardware runs it — and conflating them leads to wrong design choices. A program can be highly concurrent on a single core that does exactly one thing at a time. Another program can be deeply parallel and yet poorly structured for it. Knowing which property you're after is half the work of building fast systems.

This page is about the distinction, the famous analogy that makes it stick, and the four combinations you actually see in practice.

01A mix-up worth fixing

The shortest version of the misunderstanding: people think "concurrent" means "running at the same time." It doesn't. "Running at the same time" is parallelism. Concurrency is something subtler — it's about how the program is composed so that multiple tasks can be in flight, not about whether they happen to execute simultaneously on the hardware. A single-threaded JavaScript program juggling thousands of HTTP requests is concurrent. None of those requests are running in parallel — there's only one thread — but they're all in progress.

The clearest version of the distinction was popularized by Rob Pike in his 2012 talk Concurrency Is Not Parallelism. His phrasing has aged well:

Concurrency is the composition of independently executing things. Parallelism is the simultaneous execution of (possibly related) things. — Rob Pike

Concurrency is a property of the program: it's about how you structured the work into independent pieces that don't have to be in lockstep. Parallelism is a property of the execution: it's about whether the hardware actually runs those pieces at the same instant. The first enables the second, but neither implies the other.

02Two precise definitions

Concurrency

Multiple tasks in progress, making progress over the same span of time. The program is structured so that tasks can be started, paused, and resumed independently. A single-core machine running a multi-threaded program is concurrent: the OS interleaves the threads, so all of them are "in flight" even though only one is running at any instant.

Parallelism

Multiple tasks executing simultaneously, at the same physical instant, on different hardware. Requires more than one core (or more than one machine, GPU lane, etc.). A program that uses 8 cores at once to process an array is parallel; a program that has 8 threads on a single core is not.

The clean rule that falls out: concurrency is about structure; parallelism is about execution. Structure can exist without simultaneous execution. Simultaneous execution can be wasted on poorly structured code. The two ideas are orthogonal — you can have either, both, or neither.

03The jugglers analogy

Pike's analogy makes the distinction physical. Imagine a circus act with balls and jugglers:

Figure 01 — The four combinations as jugglers

A juggler is a CPU core. Balls are tasks. Sequential: one ball, one juggler, finish before starting the next. Concurrent only: one juggler interleaves many balls — every ball is in flight, but only one is in their hands at any instant. Parallel only: many jugglers, each handling one ball — multiple things at once but no composition. Concurrent and parallel: many jugglers, each interleaving many balls, all coordinated.

Two takeaways from the analogy. First, "concurrent" is what one juggler with many balls is doing — a structural achievement of the juggler, not a fact about how many objects are in the air at the same instant. Second, "parallel" is what two jugglers with one ball each are doing — a fact about simultaneity, with or without the structural cleverness.

04The kitchen analogy

The juggler analogy makes the shape of the distinction obvious. The kitchen analogy makes the payoff obvious — why an engineer would bother structuring a program concurrently in the first place. Imagine one chef preparing a three-course meal: 20 minutes to boil rice, 5 minutes to chop vegetables, 10 minutes to simmer a sauce.

A sequential chef does each task to completion before starting the next. Boil the rice for 20 minutes, then chop for 5, then simmer the sauce for 10. Total wall-clock time: 35 minutes. The chef's hands are busy the whole time, but most of those 20 minutes of "boiling" was just standing there watching a pot.

A concurrent chef notices that boiling rice does not require the chef. So the chef starts the rice (zero active time — just toss it in the water), uses the next 5 minutes to chop vegetables, then spends 10 minutes building the sauce. The rice finishes around the same time. Total wall-clock time: 20 minutes. Same chef. Same hands. Same single-core human. The win came from structuring the work so the waiting overlaps with the doing.

Figure 02 — Sequential vs concurrent chef on the same three dishes

The concurrent chef finishes 15 minutes faster on the same hardware (one chef, two hands). The win is structural: rice is I/O-bound — it cooks itself once started — so the chef's active time can be spent on other dishes. Add a second chef and you get parallelism on top.

The lesson generalizes cleanly. Replace "chef" with "CPU core," "boil rice" with "wait for a database query," and "chop vegetables" with "process the next request." A single-threaded server with one core can serve thousands of slow requests in the time a sequential server would serve a handful — not because the core is faster, but because the program lets it work on something else while the slow thing finishes on its own. That is the entire reason event loops, async/await, and goroutines exist.

05Four combinations

All four corners of the table show up in real systems. Each one is the right answer for some workload.

Sequential — neither

One thread, one task at a time, no overlap. Most simple scripts and command-line tools live here. Slow if there are many tasks; perfectly fine if there's one. The conceptual baseline.

Real-life example — a deploy script. Picture a typical bash deploy: npm test && npm run build && rsync ./dist server:/var/www && ssh server systemctl reload nginx. Each step strictly depends on the one before it — there is no point in starting rsync before build finishes, because the directory it copies doesn't exist yet. The wall-clock time is the literal sum of every step. Trying to "make this concurrent" would change nothing because there is no waiting overlap to exploit; the dependency graph is a straight line. This is the right shape for migrations, one-shot data transformations, and most CLI utilities. Sequential isn't a failure mode — it's the correct answer when the work is genuinely ordered.

Concurrent only — one core, many tasks in flight

A single-threaded JavaScript runtime handling 10,000 simultaneous WebSocket connections. A Python asyncio server. Erlang's actor system on one core. Each task is in progress; none are running simultaneously, but the program could use parallelism if it had more cores. The structural win is real: the system doesn't block on one slow request just because nothing else is running. See the JavaScript event loop for the canonical example.

Real-life example — a Node.js API on one core. Imagine an Express server backing a mobile app, running on a single vCPU container. A request comes in, the handler issues a Postgres query that takes 50 ms, then renders the JSON response. If the program were sequential, 5,000 requests at 50 ms each would take 250 seconds end to end and the server would handle exactly 20 requests per second. Instead, Node's event loop hands the query off to the OS, registers a callback, and immediately starts processing the next request. While Postgres is doing the actual database work — which the application's CPU is not involved in at all — the event loop is shepherding thousands of other in-flight requests through their own steps. The CPU stays close to 100% utilized doing useful work, and throughput climbs to thousands of requests per second on the same single core. Adding cores wouldn't help here because the bottleneck was never CPU; it was the program's willingness to wait. The fix was structural, not hardware.

Parallel only — many cores, no shared composition

Embarrassingly parallel work: render frames, hash files, train a neural-net layer's matrix multiply across GPU lanes. The tasks don't talk to each other and don't need to be interleaved — you just shard them across cores and collect results. Concurrency is barely involved because the tasks are independent.

Real-life example — batch video encoding with ffmpeg. A media team has 200 raw video files that need to be transcoded to H.264 overnight. Each encode is purely CPU-bound, takes about 4 minutes per file, and is completely independent of every other file — no shared state, no ordering requirement, no need to communicate. Done sequentially on one core, that's 200 × 4 minutes = 13 hours. Run via GNU parallel on a 16-core machine — parallel ffmpeg -i {} out/{.}.mp4 ::: *.mov — and the same workload finishes in roughly 50 minutes (16× speedup, minus a small amount of disk contention). There is no clever concurrent structure in the application code; each ffmpeg process is a plain sequential program. The win comes entirely from the hardware running 16 of those plain programs at the same instant. This is what people mean by "embarrassingly parallel" — the speedup falls out of nothing more than replication.

Concurrent and parallel — both

The default model of most modern servers. A Go program with goroutines on a multi-core box: concurrency in the structure (many goroutines making progress, possibly waiting on I/O) and parallelism in the execution (the runtime spreads them across N cores). A Java application with a thread pool. A Node.js server with worker threads. The composition lets the program manage many in-flight tasks; the cores let those tasks really run at once.

Real-life example — a Go API gateway on an 8-core box. Picture a payment-routing service: every incoming HTTP request fans out to three or four upstream services (fraud check, balance lookup, ledger write, notification), waits for them in parallel, combines the responses, and replies. The handler spawns one goroutine per request — under load there might be 20,000 goroutines alive at any moment, most of them parked waiting on a network response. That is the concurrent structure: tens of thousands of in-flight tasks, each making progress independently. Underneath, the Go runtime maintains a small pool of OS threads — by default one per CPU core — and multiplexes those 20,000 goroutines onto the 8 available threads. So at any single instant up to 8 goroutines are literally executing simultaneously (parallelism), while up to 20,000 are logically in flight (concurrency). CPU-bound work like JSON parsing spreads naturally across cores; I/O-bound waiting overlaps cheaply. Add a 16-core box and throughput roughly doubles, because the structure was already there to absorb the extra cores. This is the shape almost every backend service should aim for.

The trap people fall into
Adding cores to a poorly-structured program does not magically parallelize it. If the program is one giant lock and one shared mutable object, all the cores will line up waiting for the same thing. Parallelism without good concurrent structure is just expensive sequential code. "Concurrency enables parallelism, but doesn't guarantee it."

06A worked example — 20 requests, 4 cores

Numbers make the distinction concrete. Here are 20 HTTP requests landing on a single machine, traced through two scenarios on the same hardware: a single-threaded async runtime (concurrency only, one core) and a four-core thread-pool / goroutine scheduler (concurrency + parallelism). Pick a realistic per-request profile:

  • 2 ms of CPU — parse the HTTP request, route it, build the database query
  • 50 ms of I/O wait — Postgres runs the query; the application's CPU is doing nothing during this time
  • 3 ms of CPU — deserialize the result, serialize JSON, write the response

Per request: 5 ms of CPU work bracketing 50 ms of network wait, totalling 55 ms in pure isolation. Assume all 20 requests are queued and ready at t = 0.

Sequential baseline. One request fully finishes before the next starts: 20 × 55 ms = 1100 ms wall-clock. Of that, only 100 ms is actual CPU work, so the core sits idle ~91 percent of the time. This is the dead-simple "no concurrency, no parallelism" baseline we're trying to beat.

Concurrent only — 1 core, async event loop

The runtime starts every request as soon as it can: 2 ms of CPU to register the database call, then yield to the event loop and pick up the next request. Twenty 2-ms "starts" run back-to-back from t = 0 to t = 40. The first request's I/O completes at t = 52, leaving a 12 ms idle gap where the core has nothing to do — every request is waiting on Postgres. Then the "finish" work drains: 20 × 3 ms = 60 ms of back-to-back CPU bursts from t = 52 to t = 112. Total wall-clock: 112 ms — a 9.8× speedup over sequential, on the same single core.

Concurrent and parallel — 4 cores, thread pool

Same 20 requests, same per-request profile, but now the runtime spreads them across 4 cores — 5 requests per core. Each core does its 5 starts (10 ms), waits 42 ms while all four cores' Postgres queries run in parallel, then drains its 5 finishes (15 ms). Total wall-clock: 67 ms — a further 1.7× speedup over the single-core concurrent version, and 16.4× over the sequential baseline.

Figure 03 — Wall-clock to process all 20 requests

Concurrency alone bought the big win — almost 10× — because it converted I/O wait time from "blocking the core" into "happening in the background." Adding 4× the cores on top added another 1.7× because the workload is overwhelmingly I/O-bound; most of the wall-clock time is still Postgres, not the CPU.

The pattern is the lesson. Concurrency unlocked the I/O wait — that's where most of the dead time was, and one core was enough to absorb it. Parallelism added a smaller multiplier on top because there wasn't much CPU contention left to relieve. For a CPU-bound workload (image resizing, hashing, ML inference) the picture flips: concurrency on one core does almost nothing, and parallelism does almost everything. The right ordering is always: structure first, then scale. Add concurrency to attack the wait time; add cores to attack the compute time. Doing it in the wrong order — buying a 64-core box for a single-threaded blocking server — buys you 63 idle cores.

07Why the distinction matters

The reason this matters in design conversations: the two ideas suggest different fixes. If your bottleneck is "the system can only handle 100 in-flight requests at a time," that's a concurrency problem — restructure the program to start more in flight. If your bottleneck is "the system handles 10,000 in-flight requests but is using 12% of one core," that's a parallelism problem — let the runtime spread them across cores.

Three common situations and which idea is being invoked:

  • "Why is my single-threaded server so fast?" Concurrency, not parallelism. The event loop is keeping thousands of I/O-bound requests in flight on one core because the program is structured well. Adding cores would help only if the work became CPU-bound.
  • "Why does my multi-threaded program not get faster on more cores?" Bad concurrency structure — too much shared state, too many locks, false sharing — so adding cores adds contention, not throughput. Restructure first; scale second.
  • "Why does adding async/await everywhere not double my speed?" async/await adds concurrency (more in flight) but not parallelism (not more cores). On a single-threaded runtime, throughput per core is the same; you're just better at not blocking.

The vocabulary lets you separate "we need a different program structure" from "we need more hardware." Often both are needed, but in different proportions.

08The one-line takeaway

Concurrency is the program's idea. Parallelism is the hardware's idea. Concurrency without parallelism is real and useful. Parallelism without concurrency is real and limited.

The mistake every junior engineer makes once: assuming "concurrent" implies "fast" and "parallel" implies "many cores." Both wrong. Concurrent code on one core is fast for I/O-bound work and meaningless for CPU-bound work. Parallel code is fast only if the program structure lets it spread without contention. The two questions — "does the program have many things in flight?" and "can it execute multiple things at once?" — have to be asked separately.
