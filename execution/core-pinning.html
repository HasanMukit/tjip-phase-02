Core pinning

By default, the kernel moves your threads between cores whenever it sees fit. For latency-critical hot paths, that migration is the bug. Core pinning is the override.

The Linux scheduler is a marvel of fairness. It balances threads across cores, migrates them when one core is hotter than another, and prefers the configuration that maximizes overall throughput across hundreds of competing workloads. That's perfect for a desktop running Slack, a browser, and a music player. It's actively hostile to a high-frequency-trading thread that wants to keep its L1 cache warm and its tail latency under 10 microseconds. Core pinning — also called CPU affinity — is the override: a way to tell the kernel "this thread runs on this core, period." The trade-off is that you give up some of the scheduler's intelligence in exchange for the cache and pipeline state that comes with staying put.

This page is about that trade. What the kernel does by default, why it costs you, what pinning fixes, and the cases where pinning is the wrong tool.

01What the kernel does by default

When you start a thread on Linux, the scheduler doesn't bind it to any particular core. Instead, it picks a core whenever the thread becomes runnable, and it's allowed to migrate the thread to a different core later if doing so improves load balance. The migration happens silently — your code doesn't know it's been moved.

This default is good for a general-purpose system: cores stay roughly evenly loaded, no single core gets pinned at 100% while others sit idle, and overall throughput is high. But it has a cost no one usually measures: every time a thread moves to a new core, it loses everything that was warm on the old core.

Figure 01 — A migration loses cache state

When the scheduler moves thread A from core 0 to core 4, the L1 and L2 caches on core 4 don't have any of A's data. The thread now has to refill them — every memory access misses cache for ~50–500 µs of execution. The branch predictor's history is also local to core 0 and is lost. The migration finished in microseconds; the slowdown lingers.

The kernel doesn't deliberately punish your thread — it's just optimizing for a different metric. It cares about aggregate throughput across all threads; you may care about this one thread's tail latency. Those goals don't always agree.

02The hidden cost of migration

The thread switch itself is cheap (~1–3 µs — see context switching). The migration cost is what comes after, and it has three flavours:

  • Cold L1/L2 caches. Per-core caches are private. The new core has none of the thread's data. The first hundred microseconds after migration are dominated by cache misses — every read pays the L3 or DRAM tax.
  • Cold branch predictor. The branch-prediction tables are also per-core. After migration, the predictor sees this thread's branches as if for the first time, and is wrong more often until it relearns.
  • Cross-NUMA memory. On multi-socket boxes, RAM is split across NUMA nodes; access to your "local" RAM is faster than access to a different socket's RAM. Migrate to a core on a different socket and your DRAM accesses now cost 1.5–2× more, often without anyone telling you.

For most workloads this is invisible — the thread is doing I/O, the migration is rare, the cold-cache window is amortized away. For latency-sensitive code, it's the difference between a 50 µs operation and a 500 µs one. The p99 line on your dashboard moves up; engineers chase it for weeks; eventually someone runs perf stat -e migrations and finds the scheduler quietly moving the hot thread every few hundred milliseconds.

03What pinning actually is

Core pinning, formally called CPU affinity, is a per-thread bit-mask: the set of cores the kernel is allowed to run that thread on. By default, every thread's mask is "all cores." Pinning narrows the mask — sometimes to a single core, sometimes to a small subset.

Figure 02 — Default vs pinned

Pinning narrows the affinity mask. With one core eligible, the scheduler still time-slices that core between this thread and others — but when this thread runs, it always runs on C2, with C2's caches still holding its data.

Pinning a thread doesn't give it the core exclusively — other threads can still be scheduled there too. It just guarantees the thread won't migrate. To get exclusive use of a core, you typically combine pinning with kernel boot parameters (isolcpus) that exclude certain cores from the general scheduler entirely.

Pinning vs isolation
Pinning (sched_setaffinity) restricts where a thread runs but doesn't reserve the core. Isolation (isolcpus / cpuset) tells the kernel "don't put any general work on these cores." For genuinely tight latency requirements you do both — pin your hot thread to an isolated core, so it's the only thing that ever runs there.

04NUMA and why pinning isn't only about caches

On a single-socket machine, "which core" is mostly a question of L1/L2 warmth. On a multi-socket machine, it's also a question of NUMA — Non-Uniform Memory Access — and the cost ratio is higher.

A two-socket box has its DRAM split: half the RAM is attached to socket 0, the other half to socket 1. A core on socket 0 reading its local DRAM takes ~80 ns. The same core reading DRAM on socket 1 — over the inter-socket link — takes ~120–160 ns. That's a 1.5–2× tax on every uncached read, applied silently.

If a thread allocates memory while running on socket 0 (Linux's "first touch" policy puts those pages on socket 0's local DRAM) and then gets migrated to socket 1, every subsequent access pays the cross-socket tax until those pages are re-touched and (potentially) migrated. For a long-lived process on a multi-socket NUMA box, pinning to a specific socket — and ideally a specific core — is what keeps the memory and the thread on the same NUMA node.

05How to do it on Linux

The mechanics. Three layers, from quickest to most surgical.

From the shell — taskset

# launch a process pinned to core 2 taskset -c 2 ./my_server # pin an existing PID's threads to cores 4–7 taskset -pc 4-7 12345

Quick, no code changes, scope is whole-process. Useful for benchmarks, ops fixes, and anything where you don't have the source. The downside: every thread of the process inherits the same mask, so you can't pin one hot thread while letting bookkeeping threads roam.

From inside the program — sched_setaffinity

Each language exposes the syscall differently — pthread_setaffinity_np in C, os.sched_setaffinity in Python, runtime.LockOSThread + thread-pinning libs in Go, etc. — but they all wrap the same Linux syscall. Per-thread pinning is the precise tool: pin the latency-critical thread, leave everything else to the scheduler.

Boot-time isolation — isolcpus and nohz_full

Boot the kernel with isolcpus=4-7 and the scheduler will leave cores 4–7 alone — no general work, no kernel housekeeping, no timer ticks if you also set nohz_full=4-7. Then pin your latency-critical threads to those cores using one of the tools above. This is the standard setup for DPDK, real-time audio, and exchange-grade trading systems.

06When pinning is wrong

Pinning is a power tool and easy to misuse. Three scenarios where it makes things worse:

General-purpose servers

If your workload is "many requests, each fast, all roughly the same shape," the kernel's load balancing is doing a great job. Pinning everything restricts the scheduler's ability to fill in idle cores when one thread is blocked. Throughput drops for no benefit.

Mismatched mask

Pinning a thread to a core that's also busy with other pinned threads creates a queue. You forced contention you didn't have to. Pinning works only if the cores you pinned to actually have spare capacity for the thread.

Hyperthread siblings

Pinning two CPU-bound threads to two hyperthread siblings of the same physical core makes them fight over the same execution units. Pin to physical cores, not logical IDs, unless you understand the topology.

Premature optimization

If your tail latency is dominated by network I/O, GC pauses, or downstream services, pinning saves you nothing because cache thrash isn't the problem. Profile first; pin later.

The right reasons to pin are narrow: a hot thread whose tail latency matters, on a system where you've already proven the scheduler is moving it, and where you've measured the migration cost. The wrong reason is "it sounds like it'll be faster." Often it isn't.

07The one-line takeaway

Pinning trades the scheduler's intelligence for cache continuity. It pays off when one thread's tail latency matters more than the system's overall load balance.

The bigger picture is that the OS scheduler is making decisions on your behalf, every millisecond, without telling you. Most of the time those decisions are good. For the rare workloads where they're not — high-frequency trading, real-time signal processing, DPDK packet pipelines, microsecond-sensitive RPC paths — pinning is how you take the keys back. The trade is "less smart, more predictable." Predictable is what tail latency rewards.
