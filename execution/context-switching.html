Context switching

Every time the OS pauses one thread and resumes another, it pays a tax in registers, TLB, and cache. Add enough threads and the tax exceeds the work — your server gets slower the more workers you give it.

A modern laptop runs hundreds of threads on a handful of cores and feels like it's doing all of them at the same time. It isn't. Each core runs one thread, then pauses it, picks up another, runs that one for a while, pauses, picks up a third — fast enough that humans can't see the seams. The mechanism is context switching. It's how the OS pretends to have more cores than it does, and it's also one of the most expensive operations the OS performs. Push a server too hard with too many threads and the cost of switching between them eats more time than the threads accomplish. The graph of throughput-vs-thread-count peaks, then collapses.

This page is about that mechanism: what one switch actually does, what it costs (in cycles you can measure and ones you can't), and why "more threads" is not a free lever you can keep pulling.

01The illusion of "many at once"

One core can run exactly one thread at a time. (Hyperthreading lets two threads share a core, but they're still serializing on most resources.) Yet a Linux box with 8 cores happily runs 1,200 threads. The trick is time-slicing: the OS scheduler picks a thread, runs it for a few milliseconds, then a timer fires, the OS regains control, and the scheduler picks a different thread. Repeat thousands of times per second. Each thread gets its turn often enough that, from the outside, it looks like progress everywhere.

Figure 01 — One core, three threads

Three threads share one core. Each gets a slice of CPU time. Between every slice the OS does the context switch — saves the outgoing thread's state, loads the incoming thread's state. Those orange slivers are wasted from the program's point of view: the CPU is busy but no application code is running.

The illusion costs something. Every transition between threads is a small piece of overhead — work the OS does that doesn't move any user code forward. When threads are few and switches are infrequent, that overhead is invisible. When threads are many and switches are frequent, it stops being invisible.

02What happens during a switch

The OS performs a defined sequence of steps to swap one thread for another. None of them are slow individually; together they add up.

  1. Save the outgoing thread's state. All CPU registers, the program counter, the stack pointer, floating-point state — written into a kernel structure for that thread. Hundreds of bytes per thread.
  2. Update kernel bookkeeping. Move the outgoing thread to the run queue (or to a wait queue if it blocked); pick the next thread to run; update accounting, scheduling stats, and run-time counters.
  3. Switch address space if it's a different process. Load the new process's page tables. This invalidates the TLB (the CPU's cache of address translations) — every memory access for the next thread has to re-resolve, often with a slower page-table walk.
  4. Restore the incoming thread's state. Load its registers, its PC, its stack pointer. The CPU is now in the new thread's world.
  5. Resume execution. The CPU goes back to running user code — but in caches that are now cold: the new thread's data probably isn't in L1 or L2 yet, so the first reads after a context switch tend to miss several layers of cache.

Steps 1, 2, and 4 are the direct cost — measurable in cycles and microseconds. Step 3 (TLB flush) and step 5 (cold caches) are the indirect cost — they don't show up at the moment of the switch, but they slow down everything the new thread does for a while afterward.

03The direct and indirect cost

Numbers vary by hardware and workload, but the orders of magnitude are stable:

Cost Time Notes
Direct switch (same process) ~1–3 µs register save/restore + scheduler bookkeeping
Direct switch (different process) ~3–10 µs + page-table swap, partial TLB flush
TLB miss penalty ~10–100 ns each multiplied by every fresh page touched after the switch
Cold-cache penalty ~50–500 µs of slow execution until the new thread's working set is back in L1/L2

The headline number — "~1–10 µs per context switch" — is the easy part to measure. The hard part is the cold-cache tail: a switch happens in 3 µs, but for the next 100 µs the thread runs at a fraction of full speed because every memory access misses caches. That's the cost that sneaks up on you, because it's invisible in perf stat -e context-switches but very visible in cycles per instruction.

Why the cache cost is the real cost
Saving and restoring registers is fast — they're literally on the CPU. The expensive part of a context switch is what you lose in flight: a thread that was warm in L1, holding onto useful TLB entries and a primed branch predictor, gets evicted while another thread runs. By the time the first thread is rescheduled, all of that state is gone and has to be rebuilt. The CPU isn't idle while it rebuilds — it's just running slowly.

04Voluntary vs involuntary

Not all context switches are forced. Linux distinguishes two kinds:

Voluntary

The thread blocks itself — calls a syscall that has to wait (read from a socket with no data, lock that's held, sleep, condition variable). The kernel marks it as not-runnable and picks someone else. The thread "agreed" to give up the CPU.

Involuntary

The thread is happily running and the kernel preempts it anyway — typically because its time slice expired or a higher-priority thread became runnable. The CPU is forcibly wrenched away. The thread didn't agree to give it up.

Both kinds appear in /proc/<pid>/status as voluntary_ctxt_switches and nonvoluntary_ctxt_switches. The ratio is diagnostic. Lots of voluntary switches means the program is I/O-bound — threads are waiting on something a lot. Lots of involuntary switches means the program is CPU-bound and thread-heavy: the scheduler is having to forcibly time-slice between hot threads, and that's where context-switch overhead bites hardest.

05Why more threads can mean less throughput

This is the surprising shape that catches engineers building servers. Naively, you'd expect throughput to keep rising as you add threads — more workers, more work done in parallel. In reality, the curve peaks at some thread count and then falls.

Figure 02 — The throughput-vs-threads cliff

A typical throughput curve as you add threads. Up to some point, more threads do more work — they fill in waits, exploit additional cores, smooth out queues. After the peak, throughput falls because the cost of switching between them exceeds the value they add. The peak is much higher for I/O-bound workloads (lots of waiting, switches are productive) than for CPU-bound ones (every switch is pure overhead).

Why does the cliff happen? Three things compound past the peak:

  • Switching tax. Each tick, more threads share the CPU, so each one's slice is shorter. Per second, the OS does more switches; the cumulative direct cost rises.
  • Cache thrash. Each thread's working set evicts the others' from L1/L2. By the time a thread is back, its caches are cold; it runs at a fraction of the speed it ran before.
  • Lock contention. More threads contending for the same shared resources (mutexes, queues, connection pools). Threads spend more time waiting for locks, which forces voluntary switches and starts the cycle again.

The classic case study is the C10K problem from the late 1990s — handling 10,000 concurrent connections on a single server. The "thread per connection" model collapsed under exactly this curve: 10,000 threads meant 10,000 stacks (massive RAM cost), constant context switching, and dreadful cache locality. The fix was a different model entirely — event loops on a few threads (Nginx, Node.js), or M:N green threads that multiplex many language-level "threads" onto a small OS-thread pool (Go, Erlang, modern Java virtual threads). Both replaced "many OS threads" with "many in-flight tasks on few threads," which is exactly the concurrency-vs-parallelism distinction in production form.

06Designing around the cost

The pragmatic playbook for keeping switches under control:

Match thread count to cores

For CPU-bound work, the right thread count is roughly the core count (or 2× for hyperthreaded systems). More threads beyond that just trade work for switching overhead. Most thread-pool defaults follow this rule.

Use async/event-loop for I/O-bound work

I/O-bound threads spend nearly all their time blocked, so a thread per request is mostly stacks-of-RAM and switches-for-nothing. Event loops handle 10K+ connections on a single thread because no thread ever waits — the loop tracks pending I/O cheaply.

Lean on lightweight threads

Go's goroutines, Erlang processes, Java virtual threads, async Rust tasks — they're scheduled by a userspace runtime that can switch in 100s of ns instead of µs. The OS sees a small thread pool; the language sees millions of "threads."

Pin hot threads to cores

For latency-critical work, prevent the kernel from migrating a thread between cores. A pinned thread keeps its L1/L2 caches across slices. See core pinning.

Past the peak of the curve, every thread you add slows the server down. "More workers" is a guess; profile the switch rate, the cache misses, and the lock contention before you guess again.

07The one-line takeaway

Context switching is what makes "many threads" possible and what makes "too many threads" disastrous. Each switch costs microseconds directly and many more indirectly through cold caches and TLB misses. There's a peak thread count above which throughput falls; that peak is workload-dependent but always real.

The deeper point worth taking away: the OS scheduler is doing a hard job — splitting one CPU among many threads — and it's not free. The systems that scale most gracefully are the ones that cooperate with the scheduler: keep thread counts modest, do as much work as possible per slice, and use higher-level concurrency primitives (event loops, lightweight threads, async) so that the OS rarely has to do an expensive switch in the first place.
