Distributed systems in the wild

Seven short scenarios that exercise consistent hashing and the CAP theorem. Read the problem, understand the terms, think about what you'd do — then reveal the solution.

Reading a theorem is one thing. Watching it decide, at 3 a.m., whether a million users get an error or a stale balance is another. Each case study below gives you the scenario, the diagnosis, and the vocabulary you'll need — and only reveals the fix when you click for it. Try forming your own answer first; the solution makes more sense when you've taken a swing at it.

01Cache fleet resize — the mod N catastrophe

Scenario
You run a 10-node Memcached fleet for HTTP session storage, routed with hash(session_id) mod 10. Traffic is doubling. You add 5 nodes, roll out the new config, and ~15 minutes later the origin database falls over. Why?

What's happening. The fleet went from N = 10 to N = 15. Under a modulo-N routing rule, the destination for almost every key depends on the current value of N. Change N and the destination changes for most keys — roughly 1 − 10/15 ≈ 67% of them. All those keys now look up a cache entry on the wrong node, find nothing, and fall through to the origin database at the same moment. The database wasn't designed for a 67% burst of uncached requests, so its connection pool saturates and everything upstream stalls. This is not a real traffic spike; the routing scheme created its own spike.

Terms to know
Memcached
A distributed in-memory key-value cache. Each server holds a slice of the keys; a client-side routing rule decides which server owns which key.
hash(key) mod N
The simplest way to map a key to one of N servers: hash the key to a big integer, take the remainder when divided by N. Fast, but the mapping breaks the instant N changes.
Cache miss
A lookup for a key the cache doesn't have. The system then has to fetch the real value from the slower source of truth (here, the database).
Origin / source of truth
The authoritative, usually slower, store behind a cache. A cache exists to shield this store from load.
Thundering herd
A situation where many requests suddenly hit a shared downstream resource at once — usually because a cache was flushed, a service restarted, or a lot of keys expired simultaneously.
Connection pool
A fixed-size set of open database connections an application reuses. When all connections are busy, new queries queue or time out, making the database appear "down."
Reveal the solution Click to show the fix and the design principle behind it.

Solution. Swap mod N for consistent hashing. Imagine a giant clock face numbered 0 to a huge integer. Hash each server's name to a position on the clock. Hash each key to a position too. The rule: starting at the key's position, walk clockwise until you hit a server — that server owns the key.

Why this is kinder to the cache when you resize: a new server only "steals" the keys in the small arc between itself and its counter-clockwise neighbor. Every other server keeps its keys exactly where they were. Going from 10 to 15 servers moves roughly 5 / 15 ≈ 33% of keys — not 67%. Cache miss storm avoided.

One tweak makes it robust: a single position per server can get unlucky (one new server lands in a dense patch of keys, another in a sparse one). Let each physical server plant 150 virtual positions around the ring so territories spread out and load averages. Memcached's ketama library, Cassandra, and Riak all do this by default — flip the flag.

Immediate fix

Change your cache client from mod N to consistent-hashing mode (usually one config flag, e.g. hash: ketama). Next resize will touch ~1/N of keys instead of the majority.

Defence in depth

Add request coalescing between cache and database. If 500 requests miss for the same key simultaneously, only the first one queries the DB — the other 499 wait for its result. Even a bad resize can't stampede the origin.

02The celebrity hot shard

Scenario
Your social app partitions user timelines by hash(user_id) across a Cassandra-like ring. One user gains 40M followers overnight. Reads of that user's partition now hit a single physical server at 90% CPU while the rest of the fleet idles at 10%.

What's happening. Consistent hashing spreads keys evenly across servers, not load. The hash of one user_id always maps to the same ring position and therefore the same server. When that single key receives 1000× more traffic than average, the host server carries 1000× more load than its peers — no matter how many vnodes you use or how many new servers you add. The routing is working exactly as designed; it's the workload that's skewed.

Terms to know
Partition
In a sharded database, a partition is the unit of data assigned to one node. All rows whose partition key hashes to the same position live together.
Hot key / hot shard
A single key or partition receiving disproportionate traffic. The rest of the cluster can be healthy while the host of the hot key melts.
Skew
Uneven distribution of work. In key-value stores skew can mean uneven data (lots of rows on one node) or uneven load (lots of requests to one node). They're related but not the same.
Virtual node (vnode)
A marker a physical server plants at many positions on the hash ring (typically 150–256). They spread a server's territory out so load balances across the ring — but they can't split one key.
Fan-out read
A read pattern where the client asks several partitions and merges the results. Solves the "one key lives in one place" limit at the cost of more coordinator work.
Reveal the solution Two patterns depending on whether the workload is write-heavy or read-heavy.

Solution. One key is one ring position, and one ring position is one server — so as long as the celebrity lives under a single key, no amount of re-sharding helps. The fix is to stop treating it as a single key. Split the celebrity's data across many keys, put them on many servers, and merge on read. Which pattern you pick depends on whether you're drowning in writes or drowning in reads.

Key salting (write-heavy)

Instead of writing under celebrity_id, pick a random number 0–31 (the "salt") and write under celebrity_id#7, celebrity_id#23, etc. Each salt hashes to a different ring position, so writes now spread across up to 32 servers instead of one. On read, the client queries all 32 partitions in parallel and stitches the results together — more work per read, but the celebrity is no longer a single hot spot.

Read-path caching (read-heavy)

For a 40M-follower celebrity, reads probably outnumber writes 1,000×. Put a small in-memory cache (Redis, or in-process) in front of that one partition. The first read fills the cache; the next 999 answer from RAM and never touch the shard. Pair it with request coalescing so a cache miss doesn't trigger a mini-stampede.

The deeper lesson: consistent hashing solves "where does this key live?" but skew is a workload problem, not a routing problem. If one key gets most of the traffic, you need to change the key shape or put something in front of it — the hash ring alone cannot help.

03A fintech ledger during a regional split

Scenario
You run a payments ledger across two regions with synchronous replication. At 14:02 the inter-region link degrades: packets still flow but 40% are lost and latency spikes to 8 seconds. Do you let each region keep accepting writes, or pause half the world?

What's happening. Your two regions can no longer reliably coordinate. If both keep accepting writes, user alice could withdraw her full $500 balance in eu-west while another session withdraws the same $500 in us-east. When the link heals, the ledger has a $500 deficit — a real-money bug and a regulatory problem. This is the textbook CAP dilemma: during a network partition a distributed system must pick between Consistency (no wrong answers) and Availability (every region answers). You cannot have both while the regions are disconnected.

Terms to know
Synchronous replication
A write isn't acknowledged to the client until it has been persisted on more than one replica. Gives strong durability; slows writes by a network round-trip.
Network partition
A failure where nodes can't reliably talk to each other even though they're all individually alive. The cluster splits into groups that can't agree on state.
CAP theorem
Conjectured by Eric Brewer (2000), proved by Gilbert and Lynch (2002). During a partition, a distributed system must sacrifice either Consistency or Availability. P (partition tolerance) is mandatory on any real network.
Linearizability
The strongest form of consistency: every read returns the most recent committed write, as if there were a single copy of the data. What money demands.
Consensus protocol
An algorithm (Raft, Paxos, Zab) that lets a set of replicas agree on a linear sequence of commands despite failures. Commits require a majority of replicas to agree.
CP system
A system that, under partition, preserves Consistency at the cost of Availability — the minority side returns errors instead of risking wrong answers.
Reveal the solution Why money demands CP, and how many regions you actually need.

Solution. Choose CP — consistency, even if it means some users see an error. Run the ledger on a consensus-backed store (Raft-based like etcd or CockroachDB, Paxos-based like Spanner). These databases enforce one rule: a write is not committed until a majority of replicas agree. "Majority" is strictly more than half — for 3 replicas that's 2; for 5 replicas that's 3. Because only one group of replicas can hold a majority at any given time, only one side of a partition can accept writes. The other side returns HTTP 503.

Replay the scenario: when the inter-region link degrades, at most one region still has a majority of replicas reachable. That region keeps processing Alice's withdrawals. The region without a majority refuses them — Alice sees "payments temporarily unavailable" there, but she cannot drain $500 twice. When the link heals, there's no divergence to reconcile; the ledger was only ever changing on one side.

For money, a wrong answer is worse than no answer.

This is why region count matters. Two equal regions + partition = neither has a majority (both sides see 1 of 2), so both stop accepting writes. Real-world systems run three regions (or two regions plus a small tiebreaker node in a third location) so the surviving majority is always well-defined.

04A shopping cart that must never fail

Scenario
Same network failure as above, but now for the shopping-cart service of a big-box retailer. Black Friday is in 4 hours. Product leadership: "Under no circumstances do we show an error when a customer tries to add to cart."

What's happening. Here the trade-off inverts. A cart that rejects "Add to cart" during a partition directly burns revenue and trust — a rejection you saw is worse than a stale answer you didn't notice. But if both regions accept writes independently, the replicas will disagree when the partition heals. A naive "last write wins" resolver can delete items the user added if the clocks don't line up perfectly. Divergence is cheap; data loss is not.

Terms to know
AP system
Under partition, chooses Availability over Consistency: every healthy node answers, possibly with stale or divergent data, and conflicts are reconciled later.
Eventual consistency
A consistency model where replicas are allowed to diverge temporarily but are guaranteed to converge once updates stop flowing. No freshness guarantee at read time.
Last-write-wins (LWW)
A conflict-resolution rule that picks the update with the latest timestamp when replicas disagree. Simple, but loses concurrent updates and is clock-skew sensitive.
Vector clock
A data structure that records "who saw what, when" across replicas. Lets you detect when two versions were updated concurrently (neither happened-before the other) rather than silently picking one.
CRDT (Conflict-free Replicated Data Type)
A data type designed so concurrent updates always merge deterministically without loss. Examples: counters, sets, maps with well-defined union semantics.
OR-Set (Observed-Remove Set)
A specific CRDT for sets where adds always win over concurrent removes that haven't "seen" them. Used to model carts, todo lists, and other collections.
Reveal the solution AP done carefully — the conflict resolution is the whole game.

Solution. Choose AP — keep every region answering — but replace last-write-wins with a data model whose merge rules cannot lose information. The goal is to make the cart behave like a math operation: no matter which updates each side saw in which order, when the replicas finally compare notes they combine their states deterministically and never drop an add.

Walk through with an OR-Set (observed-remove set). Alice's cart starts as {shoes}. The partition begins. In eu-west she adds hat; in us-east she removes shoes. Each operation is tagged with a unique ID so later we can tell what "saw" what. On heal, the merge rule is: keep every add whose ID was not explicitly observed by any remove. Result: {hat}. The hat add happened on the other side and no remove ever saw it, so it survives. The shoes remove did see the original shoes add (that's why it could target it), so shoes is gone. No lost hats, no timestamp games, no clock-skew bugs.

Union-merge (CRDT)

Model the cart as an OR-Set, as above. The database library does the merge for you. Amazon's original Dynamo paper recommended this exact pattern for carts — it's the canonical AP example.

Per-replica vector clocks

Attach a vector clock (a "who saw what" counter) to each cart version. On heal, if two versions are genuinely concurrent, the database returns both as "siblings" and the app (or user) picks. Cassandra and Riak both support this.

The key insight: "AP" doesn't mean "accept lies." It means "accept divergence, then reconcile with rules that preserve user intent." Last-write-wins breaks that promise; CRDTs and vector clocks keep it.

05Picking R and W for a tunable store

Scenario
You're deploying a Cassandra-style cluster with replication factor N = 3. The client lets you pick R (read quorum) and W (write quorum) per query. Three teams want different things: the analytics team wants cheap reads, the user-profile team wants "normal" reads, the billing team wants safety. What do you give them?

What's happening. Consistency in a replicated store isn't a single cluster-wide knob — it's a pair you choose per query. You decide how many replicas a write must reach (W) before it's acknowledged, and how many a read must consult (R) before it can return. The interaction between these two, and the replication factor N, determines whether reads can possibly return stale data.

Terms to know
Replication factor (N)
How many replicas of each piece of data the cluster stores. Higher N means more redundancy and read options, but more write cost and storage.
Write quorum (W)
The number of replicas that must acknowledge a write before the client gets a success. Lower W is faster; higher W is safer.
Read quorum (R)
The number of replicas the coordinator consults on a read, reconciling the responses (usually by picking the version with the latest timestamp).
Linearizable read
A read guaranteed to see the latest committed write. Achieved when R + W > N: the read and write sets of replicas must overlap, so at least one replica in every read has seen every write.
Quorum / QUORUM
A majority: ⌊N/2⌋ + 1. With N=3 that's 2. A quorum read + quorum write satisfies the linearizability inequality.
PACELC
A refinement of CAP. Partition → choose Availability or Consistency. Else (normal operation) → choose Latency or Consistency. Captures that there's also a trade-off when the network is healthy.
Reveal the solution One cluster, three different (R, W) profiles per workload.

Solution. Give each team their own (R, W) profile on the same cluster — nobody needs a separate database. The rule that makes this work is one inequality:

R + W > N ⇒ linearizable reads

Why the inequality holds. Suppose N = 3. A write with W = 2 has to reach at least 2 of the 3 replicas before it's acknowledged. A subsequent read with R = 2 has to consult at least 2 replicas before returning. Since 2 + 2 = 4 is greater than N = 3, the write's two replicas and the read's two replicas must overlap in at least one node — by pigeonhole, there's no way to pick two disjoint pairs out of three. That overlapping replica has the latest write, the coordinator picks the latest-timestamped version it sees, and the read returns fresh data. If R + W ≤ N, the two sets might not overlap, and the read can return a stale value from a replica that missed the write.

Team R, W Behaviour Lands at
Analytics R=1, W=1 Fast, may be stale; survives 2 replica failures AP
User profile R=2, W=2 (QUORUM) R+W=4 > 3: linearizable; tolerates 1 replica loss balanced
Billing R=3, W=3 (ALL) Fully consistent; any replica outage blocks writes CP

One cluster, three consistency profiles. This is PACELC in practice: during a partition you're forced to pick availability vs consistency, but even during normal operation you're picking latency (R=1) vs consistency (R=3). Tune per query, not per cluster.

06Rolling upgrade of a live database cluster

Scenario
A 50-node Cassandra cluster runs on an OS that reached end-of-life. You need to upgrade every node. Total downtime allowed: zero. How do consistent hashing and replication let you do this without anyone noticing?

What's happening. To upgrade a node you have to take it down. Taking a node down deletes its vnodes from the ring. Any key whose primary replica was on that node is suddenly one replica short — the surviving replicas serve reads, but the redundancy is reduced. If you tried to do this to 10 nodes at once, some keys might lose all 3 replicas simultaneously and that's real data loss. A careful rolling upgrade has to keep the invariant "at most one replica missing per key" true for every moment of the rollout.

Terms to know
Drain / cordon
Drain: gracefully finish in-flight requests and stop accepting new ones. Cordon: mark the node so the cluster's routing avoids it without thinking it's failed.
Gossip protocol
A peer-to-peer heartbeat mechanism where each node periodically exchanges cluster state with a few random peers. Membership changes propagate to everyone within seconds.
Hinted handoff
When a target replica is temporarily down, a healthy replica stores the pending write as a "hint" and delivers it when the target recovers. Cassandra's first line of defence against short outages.
Anti-entropy repair
A background process that compares replicas of the same data (usually using Merkle trees) and fixes any divergence. Catches writes that slipped through hinted handoff.
Rack awareness
A replica-placement rule that forces the N replicas of each key onto N different failure domains (racks, AZs). Means a whole-rack outage costs only one replica per key.
Reveal the solution Three design properties conspire to make one-at-a-time rollouts invisible.

Solution. The trick isn't any single feature — it's that three independent design choices already in the cluster conspire to make single-node downtime invisible. Walk through one node's upgrade and watch them compose:

  1. Vnodes scatter the pain. You drain node A. Its 150 virtual positions disappear from the ring, and the keys they owned shift to 150 different neighbors — not one unlucky neighbor absorbing everything. No one peer gets a 2× load spike.
  2. RF ≥ 3 and rack-awareness survive the gap. Every key on node A still has 2 other replicas, placed on different racks (or AZs) by design. Reads with R = QUORUM (i.e. 2 out of 3) keep returning correct answers the whole time. Writes with W = QUORUM keep committing.
  3. Gossip + hinted handoff catch up the latecomer. While A is down, peers that receive writes for A's keys stash them locally as "hints." When the upgraded A rejoins (gossip notices within seconds), peers replay those hints to A. A background anti-entropy repair then sweeps for anything the hints missed.

The operational recipe, node by node: drain → cordon → upgrade the OS → restart Cassandra → wait for gossip to mark it UP → wait for repair to finish → move to the next node. At every single moment of the rollout, each key has at least 2 of 3 replicas live, so quorum reads and writes never fail. Do all 50 nodes sequentially and users see zero downtime.

The beautiful part
None of the routing logic changes during the upgrade. Clients still hash keys and walk the ring clockwise. A vnode just has a temporarily different physical home — which is the same situation the cluster handles every time a node crashes unexpectedly. Rolling upgrades reuse the failure-recovery code path on purpose.

07Losing an entire availability zone

Scenario
Your RF=3 cluster is split across three AZs (a, b, c) with 20 nodes per zone. At 02:14, AZ-a's networking hardware fails and stays down for 3 hours. Which keys are safe, which are not, and what does the cluster do?

What's happening. Losing an AZ kills ~33% of the nodes at once. Whether this is survivable depends entirely on where the replicas were placed. If the replica-placement strategy is oblivious to AZs, it may randomly land 2 of 3 replicas for some keys in the same AZ. Those keys just lost 2 of 3 replicas — quorum reads (R=2) now fail for them. If the placement strategy is AZ-aware, each key has exactly one replica per AZ, and losing one AZ costs exactly one replica per key — everything still meets quorum.

Terms to know
Availability Zone (AZ)
A failure-isolated datacenter within a cloud region. Usually on independent power, cooling, and networking. A typical region has 2–6 AZs; they're the granularity at which cloud providers guarantee independent failure.
NetworkTopologyStrategy
Cassandra's rack-aware replica placement strategy. When walking the ring to pick replicas, it skips candidates in racks (AZs) where it already placed a replica.
Replication factor (RF)
Same as N in the quorum inequalities — how many copies of each piece of data the cluster stores.
Under-replicated
State where a key has fewer live replicas than the replication factor demands. Still readable while any replica survives, but durability is reduced until repair restores the missing copies.
Reveal the solution Rack-awareness turns "disaster" into "graceful degradation".

Solution. Turn on a rack-aware placement strategy (in Cassandra: NetworkTopologyStrategy). The ring-walk for picking replicas now has an extra rule: when looking for the 2nd and 3rd replicas, skip any candidate that lives in an AZ already used. The result: every key's 3 replicas land in 3 different AZs, one per zone, guaranteed.

Concretely, picture key k. With the default (AZ-oblivious) strategy, its 3 replicas might land on nodes 4, 5, 6 — which happen to all be in AZ-a. When AZ-a fails, key k has 0 live replicas: total outage for that key. With rack-awareness, the walk picks node 4 in AZ-a, skips node 5 (also AZ-a) and node 6 (also AZ-a), and lands on node 21 in AZ-b; skips candidates in AZ-a and AZ-b until it finds node 42 in AZ-c. Now key k's three replicas are one per zone. AZ-a failing costs exactly one replica per key — never two or three.

While AZ-a is down (hours 0–3)

Every key has 2 of 3 replicas live. Reads and writes at QUORUM (=2) keep succeeding. Only reads at R = ALL fail — so a small slice of strict-consistency traffic errors, and everything else serves normally.

When AZ-a returns

Gossip marks the AZ-a nodes UP. Anti-entropy repair compares replicas (using Merkle-tree diffs) and backfills the 3 hours of missed writes. Network and disk utilization spike during repair, but user-facing traffic is unaffected. Full RF=3 is restored in minutes to hours.

Consistent hashing tells you where data lives. Rack-awareness tells you where it doesn't — and the second rule is what makes you disaster-proof.

08The pattern across all seven

These scenarios look different but the reasoning has the same shape:

  • Name the failure mode first. Thundering herd, skew, divergence, data loss. The fix falls out of knowing which one you're fighting.
  • Ask the CAP question explicitly. "When the network breaks, is a wrong answer worse than no answer?" The answer is different for a ledger (CP) and a cart (AP) — and may even differ per endpoint within one app.
  • Use the routing layer for routing, the storage layer for safety. Consistent hashing solves placement; replication and quorum solve durability and consistency. Don't ask one to do the other's job.

Every distributed system you'll ever build or operate will hand you a variation of one of these seven. The names change; the questions don't.
