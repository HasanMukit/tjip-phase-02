Interfaces under load

Six production scenarios that exercise idempotency, pagination shape, picking the right API style for the right reason, the GraphQL N+1 trap, and the two ways TLS certificates take a service down. Read the problem, work the diagnosis, then reveal the fix.

An API is a contract — and every contract has a failure mode that only shows up under traffic. Each scenario below pairs a real-world symptom with the part of the contract that was wrong, and ends with the smallest change that fixes it. Try your own answer before you reveal the solution; the lesson lands harder when you've already committed to a guess.

How to read this page
Every case study has three blocks visible up front: Scenario (the setup), What's happening (the diagnosis), and Terms to know (the jargon defined in plain language). The solution is hidden behind a "Reveal" toggle. Drafting your own fix first is the point — the explanation makes more sense once you've already picked a direction.

01We added retries and started double-charging customers

Scenario
Your payments service exposes POST /payments. Last week the team added an automatic-retry policy in the mobile client because a small percentage of requests were timing out at the gateway. Support tickets start flooding in: customers see two charges on their card for the same order. Logs confirm it — the same logical purchase produced two distinct payment rows, seconds apart, both successful.

What's happening. The retry policy is correct in spirit but unsafe against this endpoint. The original request did reach the server, the charge did land, and the response was lost on the way back to the client (gateway timeout, dropped connection, mobile radio handoff — pick one). The client can't tell the difference between "request never landed" and "request landed but reply got lost," so it retries. POST /payments isn't idempotent, so the second call creates a second payment. The protocol gave you the right semantics — idempotent verbs are safe to retry, non-idempotent ones aren't — and the retry policy ignored them.

Terms to know
Idempotent operation
Calling once and calling five times leaves the server in the same state. GET, PUT, DELETE are idempotent by construction. POST is not — every call creates a new resource.
Safe operation
An operation that doesn't change server state at all (GET, HEAD, OPTIONS). Safety implies idempotency; the converse isn't true.
Idempotency-Key header
A client-generated unique token (typically a UUID) that the server uses to deduplicate retries. First call does the work and stores the response; subsequent calls with the same key replay the stored response.
Natural-key dedup
Using a unique field already in the request body (order_id, transaction_reference) plus a database unique constraint to make the write idempotent without a special header.
At-least-once delivery
The retry contract: the server may see the same request more than once. Building APIs that survive at-least-once requires either idempotent operations or a dedup mechanism.
Replay attack
An attacker resends a captured idempotency key with a different body. The fix: bind the key to the body (e.g. reject if a replay arrives with a hash that doesn't match the first call's body).
Reveal the solution Two patterns; payments use both.

Solution. Make the endpoint idempotent before you turn retries on. Two patterns, used together at most payment companies:

Idempotency-Key header

Client generates a UUID per logical purchase and sends it on every retry. Server stores the full response (status code + body) under that key with a 24h TTL. First call processes; retries replay the stored answer. Stripe popularised this; most card APIs follow it.

Natural-key dedup at the DB

Add a unique constraint on (order_id) in the payments table. Use INSERT ... ON CONFLICT (order_id) DO NOTHING RETURNING *. Even if the header layer fails, the database can't write a second row for the same order.

Three things to get right around the header:

  1. Store the entire response, not just success/fail. A retry after a network blip wants the same body the first attempt produced — same payment ID, same timestamps. Otherwise the client builds two views of "the charge."
  2. TTL exceeds the retry budget. If the client retries for up to 10 minutes, the key store keeps the response for at least that long. Lose the key and you lose idempotency.
  3. Reject mismatched bodies. Same key + different body = either a client bug or a replay attack. Return 422; never silently succeed.

Retries are only safe if the operation is idempotent. The retry policy on the client and the idempotency mechanism on the server are two halves of the same contract.

02Page 50,000 of search results takes 20 seconds

Scenario
Your product-search endpoint paginates with ?page=N&per_page=20, mapped server-side to LIMIT 20 OFFSET (N-1)*20. Page 1 returns in 30 ms. Page 50 returns in 80 ms. Page 5,000 takes 4 seconds. Page 50,000 — which a power-user crawler hits routinely — takes 20+ seconds and frequently times out. The product team asks if you "need a bigger database."

What's happening. OFFSET 1000000 doesn't mean "skip a million rows cheaply" — it means the database scans and discards the first million rows before returning yours. The cost is O(offset), not O(limit). An index on the sort column helps the ordering, but doesn't avoid the discard step. There's no amount of hardware that fixes this; the query plan is fundamentally wrong shape for the workload. Worse, even if you bought your way out of the latency, offset pagination has a second bug — inserts and deletes shift rows between pages, so a user paging through a live-updating result set sees duplicates and gaps. The product team is asking for the wrong fix because they're reading the symptom (slow), not the shape (linear in offset).

Terms to know
Offset/limit pagination
"Skip N rows, return the next M." Easy to reason about, supports random access ("Page 50 of 200"). Cost is O(offset) — the database has to scan past everything you skipped.
Cursor pagination
"Return rows that come after this opaque token." The token typically encodes the sort key of the last row served. Cost is O(log N) per page — an indexed seek to the cursor position.
Stable sort key
A monotonically increasing tuple (usually (created_at, id)) that totally orders the rows. Required for cursors so two rows with the same timestamp don't ambiguate the boundary.
Result-set drift
What happens to offset pagination when rows are inserted or deleted between page fetches. A row that was at position 20 is now at 21, so "page 2" repeats it. Or a deletion shifts row 21 to 20 and you skip a result entirely.
Opaque cursor
A base64-encoded blob the client treats as a black box. The server can sign or encrypt it so clients can't synthesise their own and lock you into a specific encoding forever.
Random-access pagination
The "jump to page 50" UX. Cursors don't support it (only forward/back-from-here). Offset does — which is the only thing it's actually good for.
Reveal the solution Switch the pagination shape, not the database.

Solution. Replace offset with a cursor. The query stops scanning past discarded rows and starts seeking directly to the boundary.

Server runs:

SELECT * FROM products
WHERE (created_at, id) < ($cursor_ts, $cursor_id)
ORDER BY created_at DESC, id DESC
LIMIT 20;

The tuple comparison handles ties on created_at — without it, two products with identical timestamps could either both appear or both be skipped at the boundary. The response includes a next_cursor the client passes back unmodified for the next page. Cost drops from O(50,000 × 20) to O(log N + 20); page 50,000 returns in single-digit milliseconds.

What you give up
Cursors don't support "jump to page 50" — only "next from here" and (sometimes) "previous from here." For a search-results UI, that's almost always fine: users either scroll forward or refine the query. If a UI genuinely needs random-access pagination (an admin dashboard with "page N of M"), keep offset for that view but cap the maximum offset hard.

Two implementation notes:

  • Make the cursor opaque. Base64-encode the tuple, optionally sign it. If you expose cursor=12345, clients will start crafting their own — and you'll never be able to change the encoding.
  • Hard-cap the maximum page. Even after you ship cursor pagination, leave a defensive cap (e.g. 1000 pages forward) so a misbehaving crawler can't walk a billion-row table to its end.

The general lesson: "slow at high pages" is a shape problem, not a hardware problem. If your dataset can have a million rows and users routinely scroll past the first thousand, you want cursor pagination — and you want it from v1, because adding it later is a breaking API change.

03"Our REST API is slow — can we move it to gRPC?"

Scenario
Your mobile team is unhappy with the public REST API's performance. The home screen needs the user, their last 5 orders, each order's line items, and the user's saved addresses. Total wall-clock time on a good 4G connection: 2.4 seconds. An engineer pitches a rewrite: "JSON is verbose, parsing is slow, let's move to gRPC." A six-month migration is on the table.

What's happening. The pitch is optimising the wrong axis. JSON is verbose and gRPC's binary framing is genuinely cheaper — but the home-screen render isn't bottlenecked on bytes. Profile the request: 2.4 s breaks down as 4 sequential round-trips (user → orders → items → addresses), each ~500–600 ms on mobile, plus ~50 ms of parse and decode. Switching wire formats might shave the 50 ms. The 2,400 ms is round-trip count × RTT, and gRPC over the same network has the same RTT. Worse, the public web doesn't speak gRPC natively — you'd add a gRPC-Web proxy and re-implement the caching, debugging, and tooling stories you already get free with HTTP. The fix isn't a different protocol; it's fewer requests.

Terms to know
Round-trip time (RTT)
How long a packet takes to reach the server and a response to return. Same-DC: ~500 µs. Mobile 4G: 50–200 ms. Cross-region: 60–150 ms. Set by physics and routing, not by your wire format.
Over-fetching
The endpoint returns more fields than the client needs. GET /users/123 returns 30 fields when the client wants 3. Cheap to fix in REST with sparse fieldsets (?fields=id,name,email); GraphQL collapses it for free.
Under-fetching
The endpoint returns less than the client needs in one call, forcing N+1 round-trips for related data. The thing actually killing the home screen here.
BFF (Backend-for-Frontend)
A thin server-side aggregator that fans out to multiple internal services and returns one shaped response per UI screen. One mobile round-trip; the fan-out happens inside the data centre where RTT is microseconds.
HTTP/2 multiplexing
Multiple concurrent requests over one TCP connection. Removes head-of-line blocking between requests but doesn't reduce the number of requests; doesn't help when each request depends on the previous one's response.
Wire format vs round-trip count
Two independent latency contributors. Wire format affects bytes-on-the-wire and parse time. Round-trip count affects how many sequential network hops the response waits on. Most "slow API" complaints are the latter.
Reveal the solution Diagnose before reaching for a new tool. The fix is one endpoint, not a new protocol.

Solution. Don't migrate. Add a BFF endpoint that returns the entire home-screen payload in one round-trip:

GET /home?user_id=123
→ {
    "user": { ... },
    "orders": [ ... last 5, with items embedded ... ],
    "addresses": [ ... ]
  }

Internal fan-out happens server-side over the data centre's microsecond-RTT network. The mobile client makes one 600 ms call instead of four sequential 600 ms calls. Total home-screen time drops from 2.4 s to ~0.7 s — a 70%+ improvement, no protocol change required. Same JSON, same fetch, same caching, same debugging.

Lever What it changes Verdict
Switch to gRPC Wire bytes (~30%), parse cost (~2× faster) Wrong axis — saves ~50 ms of a 2,400 ms problem.
BFF endpoint Round-trip count: 4 → 1 Right lever — saves ~1,800 ms.
GraphQL at the edge Same as BFF, plus client-shaped responses Equivalent for this case; more infrastructure.
Sparse fieldsets (?fields=...) Bytes per response Cheap REST-side fix for over-fetching; doesn't help here.

The general lesson: profile before you migrate. "REST is slow" usually means "we're making too many sequential requests" — and the fix is endpoint shape, not protocol. gRPC has its place (internal service-to-service where you control both ends and want speed plus types), but reaching for it because the public API feels slow on mobile is solving the wrong problem.

04We migrated to GraphQL and now the database is on fire

Scenario
Your team rolled out a GraphQL endpoint to fix over-fetching. A typical query asks for 50 users, each with their last 10 orders, each order with its line items. Latency is fine in dev with seed data. In production, the page that runs this query brings the database to its knees — DB CPU pinned at 100%, query log full of nearly-identical SELECTs, p95 on every endpoint blowing up.

Your monitoring shows the GraphQL server emitted 501 SQL queries for one incoming request: 1 to load the users, then 50 to load each user's orders, then 450 to load each order's items.

What's happening. Classic N+1 resolver. Each GraphQL field has its own resolver function, and the default implementation of "user.orders" is "given a user, run a query for that user's orders." When the query asks for 50 users, the framework calls the orders-resolver 50 times — each running its own DB round-trip, each independent of the others. Then for each of those orders, the items-resolver fires once per order. The query plan doesn't see the shape of the GraphQL request; it just sees 501 unrelated SELECTs arriving at once. The bug isn't in GraphQL; it's that the resolver layer didn't batch the per-parent fetches into a per-level fetch.

Terms to know
N+1 query problem
One initial query returns N rows; for each row, the code runs an additional query to fetch related data. Total: N+1 queries for what should have been 2. Endemic to ORMs and GraphQL resolvers; absent in carefully-written hand SQL.
Resolver
A function GraphQL calls to populate one field of one type. Composes recursively — the orders resolver runs once per user, the items resolver runs once per order. Naive resolvers are the source of N+1.
DataLoader
A per-request batching/caching layer (Facebook's open-source pattern). Collects keys requested within the same tick of the event loop and runs one bulk fetch (SELECT ... WHERE user_id IN (...)) instead of one fetch per key.
Query complexity
A weight assigned to each field in a GraphQL schema. The total complexity of a query is summed and rejected if it exceeds a per-client budget. The right unit for rate-limiting GraphQL — request count is meaningless when one query can ask for the universe.
Persisted queries
The client registers each query with the server up-front; runtime calls send only a query ID + variables. Limits the client surface to a known set of queries, restores HTTP-level caching, and lets the server pre-analyse complexity.
Resolver-level caching
Caching at the field resolver, not at the URL. Replaces the URL-based caching REST gets free from the CDN/browser. Apollo Cache, GraphQL Mesh, and DataLoader's per-request cache all play here.
Reveal the solution Batch the resolver fetches and budget the queries.

Solution. Two layers, applied together:

Wrap every per-parent fetch in a DataLoader

The orders-resolver no longer queries directly. It calls ordersByUserLoader.load(user.id). DataLoader collects every key requested in the same event-loop tick and issues one query: SELECT * FROM orders WHERE user_id IN (1, 2, ..., 50). The 501 queries collapse to 3 — one per level of the tree.

Budget queries by complexity

Every field gets a cost; the query gets a budget. users(first: 50) { orders(first: 10) { items }} is roughly 50 × 10 × items_cost. If a client tries users(first: 10000) they get rejected at parse time. Rate-limiting by request count doesn't work — one query can ask for nothing or everything.

Three operational rules that come with adopting GraphQL seriously:

  1. DataLoader is mandatory, not optional. If a junior engineer can write a resolver without it, your N+1 will reappear within a week. Lint for it; provide loaders centrally; make raw queries inside resolvers an exception.
  2. Persist queries from trusted clients. Your own apps don't need to send arbitrary GraphQL — register their queries up-front and reference by hash. Restores edge caching, eliminates whole classes of complexity attacks.
  3. Watch resolver-level metrics, not just endpoint metrics. Endpoint-level dashboards show POST /graphql at 200 ms; the right granularity is per-field — so you can see which resolver is slow.

GraphQL solves over-fetching at the wire and creates an N+1 problem at the resolver. The fix is in the resolver layer, not the schema.

05The cert expired and the whole API went down

Scenario
At 03:14 UTC, every client of api.example.com starts seeing SSL_ERROR_BAD_CERT_DOMAIN or NET::ERR_CERT_DATE_INVALID. Mobile apps fail to connect. Browsers show a full-screen interstitial. Status page goes red. Investigation: the public TLS certificate for api.example.com expired at 03:00 UTC. It had been renewed manually for years; the engineer who owned the calendar reminder left the company nine months ago.

What's happening. A TLS certificate is a signed document with a notBefore and notAfter date. Past the notAfter, every client that validates certificates rejects the connection — and almost every client validates by default. There's no graceful degradation, no warning period, no "expired but we'll allow it" mode. The certificate's job is to prove the server's identity; an expired one fails that proof. The system didn't break because of a code bug or a load spike; it broke because a piece of paper aged out and the renewal was a manual process tied to one person's calendar. This is an organizational failure dressed up as a technical one.

Terms to know
Certificate validity period
The time window (notBefore, notAfter) during which the certificate is valid. Public CAs cap leaf certificates at 13 months (will drop to 47 days in 2029). Past notAfter, every conforming client rejects the cert.
ACME / Let's Encrypt
The protocol and CA that made automated 90-day issuance the default. Tools like certbot, cert-manager, and Caddy renew certs every ~60 days without human involvement.
Certificate transparency
Public logs of every issued certificate. You can query CT logs (crt.sh) to find every cert issued for your domain — useful for inventory ("what certs are out there in our name?") and for spotting unauthorized issuance.
Lifecycle ownership
The organisational answer to "who renews this and what happens if they leave?" Paging is on a rotation, not a person. Calendar reminders attached to a team mailbox, not a single email address.
Multi-day pre-expiry alert
Monitoring that fires weeks before notAfter, not at expiry. Common thresholds: 30 days (warning), 14 days (page on-call), 7 days (incident).
Zero-downtime rotation
Reload the new cert into the running service without dropping connections. Most TLS-terminating servers (nginx, envoy, HAProxy) support a SIGHUP/admin-API reload that picks up the new cert atomically.
Reveal the solution Automate the renewal, alert before expiry, never let the lifecycle live in one person's head.

Solution. Three layers, in priority order:

  1. Automate issuance and renewal. Use ACME (Let's Encrypt) or your cloud's managed cert (ACM, Cloud Certificate Manager). cert-manager on Kubernetes; certbot with a systemd timer on bare metal. Cert lifetime drops to 90 days, renewal happens at 60 with a generous safety margin. The cert never reaches the expiry date because the system reissues weeks earlier — and if a renewal fails, you have a month of warning, not a 14-minute outage.
  2. Alert on remaining validity, not on expiry. Monitoring scrapes notAfter from the live cert (any TLS scanner does this — Prometheus blackbox-exporter, ssl_exporter, even openssl s_client in cron) and pages on-call when remaining lifetime drops below 14 days. The page goes to a rotation, not an individual.
  3. Inventory every certificate. Public CT logs plus internal scanning will surface certs you forgot about — wildcard, subdomain, internal mTLS, vendor integrations. The cert that takes you down is the one nobody owns.
What "manual renewal" actually costs
One person, one calendar, one annual reissue. Works fine for years until that person leaves, the calendar reminder fires into a defunct mailbox, the cert expires, and 100% of traffic stops in the same five minutes. The cost of automating ACME for one domain is a single certbot install. The cost of not automating it is exactly one outage of indeterminate length.

The general lesson: any operational task that depends on a human remembering will eventually fail. Certificates are the canonical version of this — invisible most of the time, catastrophic on the day they aren't. Automate the renewal, alert before the deadline, page a rotation not a person.

06Our internal certs failed verification after we reissued

Scenario
Your platform team rotates the internal CA that signs service-to-service mTLS certificates. The new root cert is published; the leaf certs on every service get reissued and deployed. Within minutes, half the internal traffic starts failing with x509: certificate signed by unknown authority. The new leaf certs are valid. The new root is trusted by the services that were redeployed. But anything that hasn't redeployed since the rotation can't verify the new chain.

What's happening. A certificate is verified by walking the signature chain back to a root that the verifier already trusts. The trust anchors live in the verifier's trust store: a static list of CA public keys baked into the OS, the browser, the language runtime, or — for service-to-service mTLS — a config file mounted into each service. When you reissue under a new root CA, every leaf cert chains back to that new root. But services running with the old trust store don't know about the new root, so the chain ends at "unknown authority." The leaf cert is valid — just not from a CA the verifier has agreed to trust. The bug isn't in the certs; it's that "we issued a new cert" and "the verifier learned about the new CA" are two separate operations and the team conflated them.

Terms to know
Trust store / trust anchor
The static set of CA public keys the verifier accepts as roots of trust. Public web: ~150 CAs shipped in browsers and operating systems. Private mTLS: usually one internal CA, distributed via config or a secret mount.
Certificate chain
Leaf cert → intermediate CA → root CA. The verifier walks the chain and rejects if it can't terminate at a root in its trust store. "Unknown authority" almost always means the chain is correct but the root isn't trusted — not that the cert is malformed.
Cert vs trust anchor
The cert proves "this key belongs to this name." The trust anchor decides whether you believe the proof. They're independent — issuing a new cert doesn't update anyone's trust store.
CA bundle
The file (often ca-certificates.crt or a Kubernetes secret) that holds the trust anchors. Adding a new CA means modifying this file and restarting the verifier — not just deploying a new leaf.
Cross-signed root
A new root that's also signed by an old, already-trusted root. Verifiers with the old trust store still accept the new chain via the cross-signature. The standard escape hatch for rolling root rotations.
Trust-then-rotate sequence
Operationally: distribute the new trust anchor first, wait until every verifier has it, then reissue leaves under the new root. The reverse order causes exactly this outage.
Reveal the solution Distribute trust before you cut over signatures.

Solution. Roll forward, in this order:

  1. Bundle both old and new roots. Update the trust store on every service to include both the old CA and the new CA. Deploy. Wait until every service is running with the dual-root bundle. Verify with a fleet-wide check.
  2. Then reissue leaves under the new root. Now both old leaves (signed by the old CA) and new leaves (signed by the new CA) verify against every service's trust store. Roll out new leaves at whatever cadence you like; nothing breaks during the transition.
  3. Then drop the old root. Once every leaf is reissued, remove the old CA from the trust bundle. The system is now fully on the new root; the old one is no longer accepted.

The key insight: distribute trust first, signatures second. If you swap the order, every verifier that hasn't rebooted is still on the old trust store, and every new leaf cert looks unsigned to it. Doing it the other way round is reversible (drop the new bundle, you're back where you started); doing it the wrong way around is an outage that lasts as long as it takes to roll the trust store everywhere — usually longer than your incident channel can stay calm.

Cross-signing as a shortcut
For very large fleets where dual-bundling is hard, the alternative is to have the new root also signed by the old root (a "cross-sign"). Verifiers that only know the old root still see a chain ending at a root they trust, via the cross-signature. Let's Encrypt's IdenTrust cross-sign worked exactly this way during the ISRG Root X1 rollout. Inside a single org, dual-bundling is usually simpler than getting cross-signs right.

The general lesson: certificates and trust anchors are separate concerns. "We issued new certs" and "the verifier accepts the new CA" are two deploys, not one. Treat them in sequence, with monitoring between, and the rotation is a non-event instead of an incident.

07The pattern across all six

Three habits come out of these six cases, and they're the working playbook for owning an API in production:

  1. Treat the contract as load-bearing. Idempotency, pagination shape, and the verbs you pick aren't cosmetic — they decide whether retries double-charge customers, whether deep pages cripple the database, and whether the next breaking change costs you a month of client migrations. Get them right at v1; retrofitting them is a contract change.
  2. Diagnose before you migrate. Most "REST is slow" complaints are round-trip count, not wire format. Most "GraphQL is fast" wins evaporate the moment a naive resolver hits a real-world database. Profile first; reach for the new tool only when the diagnosis points there.
  3. Operationalise the things that look "static." Certificates expire. Trust stores drift. CAs rotate. None of these break on the day you ship — they break months later, when the person who owned the calendar reminder has left. Automate the lifecycle, alert before the deadline, page a rotation not a person.

Every interface problem you'll chase in production is a variation of one of these. The names change; the shape doesn't.
