Resilience patterns

The performance numbers describe the happy path. These patterns are how systems stay on it when networks blip, downstreams slow down, and timers conspire.

Every page so far has assumed the happy path: requests arrive, latency is bounded, throughput is steady. Real systems run on flaky networks, restarting processes, full disks, and timing-out databases. Most of the engineering work in production isn't making the happy path faster — it's keeping the system upright when something downstream breaks. The patterns on this page are how that's done: a small vocabulary of techniques that, together, separate "a service that handles 100 K RPS in benchmarks" from "a service that's still standing on Tuesday morning."

Each pattern below is a response to a specific failure mode. Read them as a set — they compose. A retry without a circuit breaker amplifies failures. A circuit breaker without graceful degradation just turns slow errors into fast ones. A jitter strategy without throttling at the gateway is a band-aid. The whole stack is the answer.

01When the happy-path numbers lie

A transient failure is one that succeeds if you try again, with no human intervention: a dropped TCP packet, a brief network partition, a process restarting, a leader re-election in progress. Permanent failures (a malformed request, a missing record, a 401) won't get better — but transient ones often will, and they're vastly more common than permanent ones in any networked system.

The whole resilience stack is built around this asymmetry. If you can detect transient-vs-permanent and respond appropriately, the system stays up through most of what infrastructure throws at it. If you treat them the same way — by retrying everything, or by giving up on everything — you get either a retry storm or a brittle system. Neither is acceptable in production.

Idempotency is a precondition, not a pattern
Most patterns below assume the operation is idempotent — running it twice has the same effect as running it once. Reads are naturally idempotent; mutations need an explicit idempotency key (a UUID the client generates and the server dedupes by). Without idempotency, retries mean double-charges, duplicate emails, and corrupted state. Establish idempotency at the API boundary before reaching for any of the techniques below — see making POST safe to retry for the two patterns (Idempotency-Key header and natural-key dedup) that get you there.

02Retry safely — storm, capping, backoff, jitter, Retry-After

The naive instinct: a request failed, send it again. The problem: failure is often the signal that the downstream is already overloaded — and retrying is exactly the wrong response.

The retry storm

Imagine 10 K clients calling a service that just slowed down past its timeout. Every client times out and retries; some retry twice. The service now sees 20 K–30 K requests per second instead of 10 K — at the moment it could least handle them. Retries that succeed-eventually under light load become a self-inflicted denial of service under heavy load. The system has to reject more aggressively just to stay alive, and even then the downstream is still chasing its tail. This is the retry storm, and it's responsible for a sobering fraction of real outages.

Capping retries

The first line of defense is to put hard limits on how often retries happen. Three knobs:

  • Max attempts. Typically 2 or 3 total. Past that, fail. More retries don't recover failures; they just hide them.
  • Total deadline. Don't retry past the user's tolerance. If the call had a 1 s budget and 800 ms is gone, retrying an operation that takes 400 ms means the user sees a timeout anyway — better to fail now and let them retry the whole flow if they want.
  • Retry budget. A fleet-wide cap: "retries cannot exceed 10% of base traffic." Implemented as a token bucket per client. Past the budget, retries are dropped immediately even if individual policies would allow them. This is what really stops storms.

Exponential backoff

Each retry waits longer than the last: 100 ms, 200 ms, 400 ms, 800 ms. This gives the downstream room to recover and prevents busy-wait amplification. The doubling is canonical but not magic — any monotonic schedule with reasonable growth works. The point is not busy, not fixed, and not infinite.

Jitter — the unsung hero

Backoff alone leaves all clients synchronized. After a shared incident, every client retries at exactly +100 ms, then +300 ms, then +700 ms. The downstream sees pulses of load instead of a smooth recovery. The fix is to add randomness — jitter:

Equal jitter

delay = backoff/2 + random(0, backoff/2). Half of the backoff is fixed, half is random. Predictable lower bound; spread upper bound. Common default.

Full jitter

delay = random(0, backoff). Maximum smoothing. Used when the cost of synchronized retries is severe (e.g., AWS recommends this for SDK retries). Some calls come back fast, some slow — but the herd is broken.

Decorrelated jitter

delay = min(cap, random(base, prev × 3)). Each delay is randomized relative to the previous one. Avoids the case where consecutive retries land in the same window.

No jitter (don't)

Plain exponential backoff with no randomness. Common in code that hasn't seen a retry storm yet. The fix is one line and it's worth shipping before the first incident.

Retry-After — the server's voice

HTTP gives servers a way to tell clients exactly when to come back: the Retry-After header on 429 (Too Many Requests) and 503 (Service Unavailable) responses. The value is either a delay in seconds or an absolute date. Clients that respect it stop retrying until the server invites them back; servers that send it actively shape their own incoming load. This is the single best protocol-level retry primitive that exists, and it's underused.

03The circuit breaker

A circuit breaker wraps a remote call and tracks its failure rate. When failures cross a threshold, the breaker "opens" — subsequent calls fail immediately, without contacting the downstream at all. After a cooldown, it tries one trial request; if that succeeds, the breaker "closes" again. The pattern is named after household electrical breakers for the same reason: rather than melt, it disconnects.

Figure 01 — Circuit breaker as a three-state machine

Three states. CLOSED is normal: traffic flows, failures are counted in a sliding window. When the failure ratio crosses the threshold (e.g., 50% over the last 100 calls), the breaker OPENS — calls return an error instantly without touching the downstream. After a cooldown the breaker enters HALF-OPEN: a single trial request flows. Success returns to CLOSED; failure returns to OPEN.

The threshold isn't binary. Real implementations use a sliding-window failure ratio (e.g., "open if >50% of the last 100 calls failed"), with a minimum-volume gate ("don't trip on 1 of 2"). Good circuit breakers also propagate Retry-After hints from the downstream and emit events for dashboards. Hystrix popularized the pattern; Resilience4j, Polly, gobreaker are the modern incarnations.

Why fast failure helps everyone
Without a breaker, every upstream caller waits the full timeout (often seconds) for a doomed call. Threads pile up. Connection pools fill. The upstream's own latency tail explodes. With a breaker, the same calls fail in microseconds; threads and pools stay drained; the upstream's tail stays bounded. The breaker is upstream protection at least as much as it's downstream protection.

04The gateway — bulkheads and throttling

The gateway is the front door of your service: an API gateway, an Envoy or nginx layer, an edge proxy. Every request passes through it before reaching your application code. That makes it the natural place for two patterns whose whole purpose is to enforce protection at the boundary, before traffic touches anything fragile.

The two patterns answer different questions, and confusing them is the most common reason this topic feels muddled:

Bulkhead

"If something downstream fails, how do I keep that failure from spreading to the rest of the system?" Bulkheads contain a breach that already happened.

Throttle

"How do I keep too much traffic from arriving in the first place?" Throttles prevent overload by saying no at the door.

One contains; the other prevents. You almost always want both — they are complementary, not alternatives.

Bulkhead — partition resources so a failure stays in its lane

The name comes from shipbuilding. A bulkhead is a watertight wall inside a ship's hull. If the hull is breached, one compartment floods, but the wall stops the water from spreading and the ship stays afloat. Without bulkheads, one hole sinks the whole vessel.

Services have the same problem. A typical service has shared resources — a single thread pool, a single connection pool, a single CPU budget — and every request, no matter what work it's doing, pulls from those same shared pools. Sharing is efficient when everything is healthy. It becomes catastrophic the moment one downstream slows down: every thread gets stuck waiting on the slow downstream, the queue fills with calls for other downstreams, and the entire service stops responding.

The bulkhead fix is to partition the shared resource. Give payments its own pool of, say, 100 threads, and recommendations its own pool of 100 threads. Now when payments slows, it can saturate at most its own 100 threads. The other 100 keep serving recommendations. The breach is contained to one compartment of the ship.

Throttling — the hard cap at the door

A throttle is a hard limit on incoming rate or concurrency, enforced at the gateway. Past the limit, requests are rejected immediately with 429 Too Many Requests and a Retry-After header. The system never even attempts the work.

The instinct is "rejecting requests is bad — we should try to serve them." But once a system passes the knee of its capacity curve (see latency & throughput), accepting more work doesn't get more done — it just makes everything slower for everyone, until the system collapses. Rejecting 1% at the door beats degrading 100% behind it.

Token bucket

A bucket holds up to N tokens and refills at rate R per second. Each request consumes one token; if the bucket is empty, the request is rejected. Allows bursts up to N, sustains R over time.

Leaky bucket

Requests enter a fixed-size queue and drain out at a constant rate. Output is perfectly smooth no matter how spiky the input.

Fixed / sliding window

Count requests per N-second window — "max 100 per minute." Simple to implement and easy to explain to API customers.

Concurrency limiting

Cap the number of in-flight requests, not the rate. Maps better to actual capacity than rate does — by Little's Law, in-flight = QPS × latency, so a rate cap protects you only if latency stays constant.

Bulkhead vs throttle — the one-line distinction
The throttle decides how much traffic is allowed inside the system. The bulkhead decides where that traffic is allowed to do damage once it's in. Throttle keeps the door from being broken; bulkhead keeps a breach from spreading past one compartment. You install both because either alone leaves a failure mode the other would have caught.

05Cache stampede & thundering herd

Cache stampede

A popular cache key expires. The next 10 K requests all see a cache miss simultaneously. Every one of them goes to the database. The database falls over. This is the cache stampede (sometimes called dog-piling).

Single-flight / request coalescing

Only one request goes to the database for a given key; the others block until its answer is ready, then read it. Implementations: Go's singleflight, Java's CompletableFuture-based memoizers.

Probabilistic early refresh

Each read has a small chance of refreshing the cache before expiry, with the chance growing as expiry approaches. The cache gets refreshed once, by a "lucky" request, before the stampede window opens.

Stale-while-revalidate

Serve the stale value immediately while a single background task refreshes. No client ever waits for a cold rebuild; the cost of staleness is bounded to one refresh cycle.

TTL jitter

Don't set TTLs to round numbers everyone shares. Add per-key randomness — TTL = base + random(0, base × 0.1) — so caches don't all expire on the same second.

Thundering herd — the same idea, broader scope

Cache stampede is a special case of a wider phenomenon: any event that wakes many clients at the same instant becomes a load spike. Server restarts, config pushes, leader re-elections, expiring tokens. The cure is the same as for cache stampede: jitter. Spread the wakeup across an interval. The herd is broken in the time domain rather than coordinated through it.

06Synchronization spikes & coordinated events

Desync periodic events

The most insidious load spikes come from independent processes whose timers happen to align. Cron jobs at the top of the hour. Token refreshes that all expire at midnight. Health checks every 30 seconds, started at boot — and everyone boots at the same time after a deploy. Each timer alone is fine; aligned, they become a self-inflicted denial-of-service. The fix: when scheduling anything periodic, add jitter at startup.

Coordination as another spike source — ZooKeeper election

Some events are inherently coordinated. The clearest example: every node in a distributed system must agree on who the leader is. When the current leader dies, every other node finds out at roughly the same moment and rushes to choose a replacement — a guaranteed coordinated event. A naive implementation produces a thundering herd against the coordination service. So the protocol has to be designed to not wake everyone up at once.

ZooKeeper is the classic example of getting this right. A znode is one entry in ZooKeeper's filesystem-shaped tree. Two flags: ephemeral (auto-deleted when the client disconnects) and sequential (gets a globally-unique monotonic suffix). The third primitive is the watch: clients can ask "tell me when this specific znode changes."

The election: each candidate creates an ephemeral-sequential znode under /election/. The lowest sequence number wins. Each follower puts a watch on the znode directly below its own — not on the leader. When the leader dies, ZooKeeper auto-deletes its znode (because it was ephemeral), and exactly one watch fires — the next candidate in line. No herd. The chained-watch design guarantees that any single state change fires exactly one notification, no matter how many candidates are participating.

The lesson behind the recipe
Even when N processes must coordinate, the protocol can be designed so that any state change fires exactly one notification instead of N. Anything that broadcasts to all N watchers on every change will bite you at scale. When choosing a coordination tool, ask first: "what happens when the watched thing changes — who gets woken up?"

07Graceful degradation

When the patterns above can't restore a broken downstream, the last lever is graceful degradation — return a useful but degraded response instead of an error. The discipline is to design the fallback explicitly — not let it emerge by accident.

Fallbacks for missing data

Recommendation service times out → return the global popular list. User profile fetch fails → render with placeholder name and avatar. Personalization service down → serve the unpersonalized version.

Feature degradation

Pricing service unavailable → show "price unavailable" and disable the buy button instead of 500-erroring the whole page. Search index lagging → serve from a slightly older index with a "showing recent results" hint.

Batching as a degradation lever

When a downstream is overloaded, batching pending requests trades latency for throughput. Five 100 ms requests that each fail under load might succeed as one 200 ms batched request.

Well-formed errors

When you must return an error, do it well. Use the right status code (4xx for client, 5xx for server). Include a Retry-After when transient. Don't leak raw exception text. The error is a user experience, not a failure of imagination.

The test for any production endpoint is: what does it return when its primary dependency is down? If the answer is "500," the endpoint isn't done. The graceful path is part of the spec, and writing it forces you to decide which features are essential and which are decorative.

08The one-line takeaway

The performance numbers describe the happy path. The resilience patterns are how you stay on it.

Read the patterns as a stack. Idempotency is the precondition. Capped, jittered retries handle transient failures without amplifying them. Circuit breakers stop the amplification when retries aren't enough. Bulkheads keep the breach contained. Throttling shapes the door. Cache-stampede protections and jittered timers prevent self-inflicted spikes. Graceful degradation is the last lever when the system can't fully recover. None of these are exotic — they're the small, well-known moves that compose into a system that survives Tuesday morning.
