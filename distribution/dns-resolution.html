DNS resolution

What actually happens between hitting Enter on www.google.com and a packet leaving the machine — told as a walk through five layers of cache.

You type www.google.com and hit Enter. The address bar gave you a name, but the network only knows how to route IP addresses. Something has to translate one to the other before a single byte can leave the machine. That something is DNS — the largest distributed cache on the planet.

It's distributed because no single server could answer queries for every name on the internet. It's a cache because, even if such a server existed, asking it on every page load would melt it under traffic. So DNS is layered: every level keeps a copy, every level can answer if it has one, and only on a miss does the question travel to the next level up.

01Names go in, IPs come out

Computers route packets using IP addresses (142.250.190.36). Humans don't remember those. We remember names (google.com). DNS is the agreement that translates the human-readable thing into the machine-routable thing — and it does so without any single server having to know the answer for every name.

The clever part isn't the translation itself. It's the architecture: a tree of authoritative servers at the top (each owning some slice of names) and a forest of caches below them that remember answers so the top doesn't get overwhelmed. When the system works well, your laptop never asks the authoritative server at all — somebody nearby already knows.

02The five-layer cache hierarchy

A DNS query walks down a staircase of caches. The first layer that holds a non-expired copy answers immediately and the rest are never touched. Listed in the order the browser tries them:

Figure 01 — The cache hierarchy

A query falls through layers 1–4 (your local caches) until something hits or the recursive resolver has to do real work. Only on a full miss do the authoritative servers see traffic at all.

  1. Browser cache. Chrome, Firefox, Safari each keep an in-memory DNS cache scoped to the browser process. View Chrome's at chrome://net-internals/#dns. Lifetime is bounded by the record's TTL, with browser-imposed caps (Chrome historically clamps to ~60 s).
  2. OS stub resolver. Every OS ships a tiny resolver that consults /etc/hosts first (a static override file) and otherwise caches answers. macOS calls it mDNSResponder; Linux is typically systemd-resolved or nscd; Windows has the DNS Client service.
  3. Router cache. Your home or office router runs a tiny DNS forwarder/cache. The first device on the LAN to resolve a name pays the cost; every other device on the network reads it from the router for free.
  4. Recursive resolver. Your ISP's resolver, or a public one like Google's 8.8.8.8, Cloudflare's 1.1.1.1, Quad9's 9.9.9.9. This is the box that actually walks the authoritative chain on your behalf. It's also the deepest cache outside your machine.
  5. Authoritative servers. The nameservers that own the zone. Root (.) points at TLD servers; .com TLD points at the authoritative servers for google.com; those finally answer. Only this layer is the source of truth.

The hierarchy exists because if every page load did the full root → TLD → authoritative dance, the root servers would melt and every request would pay tens of milliseconds in network round-trips. Caching turns the common case (popular names, repeat visits) into a sub-millisecond local memory read.

03The recursive walk

When the query finally reaches a recursive resolver that doesn't have the answer cached, it does the actual work of finding it — and the structure of that work is its own little hierarchy.

Figure 02 — Recursive resolution of www.google.com

The resolver asks root "who runs .com?", asks the .com TLD "who runs google.com?", asks google.com's authoritative server "what's www.google.com?" — three hops to reach the answer, then caches every step on the way back.

Three queries, three answers, then the IP propagates back. The resolver caches all three intermediate results — the root NS, the .com NS, and the final A record — each with its own TTL. Next time someone asks for any other .com name, the resolver already knows the TLD; next time anyone asks for any other google.com subdomain, it already knows the authoritative server.

04Recursive vs authoritative — the distinction to nail

This is the single most important confusion to clear up about DNS.

Recursive resolver — the worker

Knows nothing on its own. Its job is to find the answer by asking root → TLD → authoritative in sequence, then cache the result so it can answer faster next time. Examples: 8.8.8.8, 1.1.1.1, your ISP's resolver.

Authoritative server — the source

Holds the actual zone file. Doesn't recurse on anyone's behalf; just answers "this is what I know about my zone" — definitively. Run by whoever owns the domain (or by their DNS provider, like Cloudflare or Route 53).

The common misconception
"Isn't 8.8.8.8 authoritative for the internet?" No. It's a recursive resolver with a huge cache. Authority is per-zone — every zone has its own authoritative servers, and only those servers can definitively answer about names in that zone. 8.8.8.8 is just the most popular cache between you and them.

05A and CNAME — the two records you'll set 99% of the time

DNS has many record types — NS (who's authoritative for this zone), MX (mail servers), TXT (free-form, used for SPF / DKIM / domain verification), SOA (zone metadata) — but the two you'll touch most often when configuring a service are A and CNAME.

Figure 03 — A vs CNAME

An A record terminates — name maps directly to IP. A CNAME hands you another name instead, which the resolver must now resolve. The chain ends only when something terminal (A or AAAA) appears.

A record — name → IP. The terminal answer. www.example.com → 93.184.216.34. The resolver hands this straight to the browser and the connection is made. (AAAA is the same idea for IPv6 — different record type because the address shape is different.)

CNAME — name → another name. An alias. blog.example.com → example.wordpress.com. The resolver doesn't get an IP back; it gets another name, which it must now resolve. Resolvers follow the chain (CNAME → CNAME → … → A) until they land on a terminal record.

The mental model: an A record is a street address; a CNAME is a "see also" pointer to a different listing. Phonebooks allow both — but with rules.

06Two CNAME rules that trip people up

Rule 1 — A name with a CNAME can have no other records

If blog.example.com has a CNAME, it cannot also have an A, MX, TXT, or anything else. RFC 1034 §3.6.2 makes the CNAME exclusive at its name (with a small DNSSEC carve-out).

The reason is semantic, not arbitrary. CNAME means "this name is that other name — go ask there for everything." If you also defined an MX record locally, the resolver wouldn't know whether to honor the local MX or follow the alias and ask the target's MX. The spec resolves the ambiguity by forbidding it.

Figure 04 — Why CNAME is exclusive

Which wins — follow the alias to wordpress.com's mail server, or use the locally-defined MX? The spec sidesteps the question by banning the second record entirely.

This is why "I added a TXT record for domain verification but my CNAME stopped working" is a recurring help-desk ticket. The TXT silently breaks the CNAME (or vice versa, depending on the provider's enforcement).

Rule 2 — You cannot CNAME the apex

The apex (or "root", or "naked domain") is the zone name itself — example.com, not www.example.com. By DNS rules, the apex must carry mandatory records: at minimum SOA (zone metadata) and NS (who's authoritative). These are required by the protocol — they're how the rest of DNS finds your zone at all.

Combine that with Rule 1 (CNAME excludes everything) and the contradiction is direct: a CNAME at the apex would have to coexist with the required SOA and NS records, which the spec forbids. So the apex cannot be a CNAME, full stop.

This is annoying in practice, because hosted services (Heroku, Netlify, Vercel, S3 static sites) typically tell you "point a CNAME at myapp.herokudns.com." That works for www.myapp.com but not for myapp.com itself.

The workaround providers invented
Route 53 ALIAS, Cloudflare CNAME flattening, DNSimple ALIAS, NS1 ANAME — all variants of the same trick. The provider resolves the target name at query time and serves the resulting IPs as a real A record. To the configurer it feels like "a CNAME at the apex"; on the wire it's a perfectly legal A record. The provider eats the indirection cost so the standard doesn't have to bend.

Short answer to "why can't I just CNAME my root domain?" — because SOA and NS are required at the apex, and CNAME refuses to share a name with anything else.

07The one-line takeaway

DNS is a giant tree of caches sitting in front of a small set of authoritative servers — every layer answers if it can, and only the truth at the top gets the last word.

Once you internalize this shape, every other DNS topic — TTL, propagation delay, the ALIAS workaround, why your /etc/hosts override worked instantly while your provider's record change took an hour — becomes a question of which cache layer is involved. The architecture is the explanation.
