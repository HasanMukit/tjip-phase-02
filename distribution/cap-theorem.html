The CAP theorem

Why every distributed system is forced to make a painful choice — and how understanding that choice changes the way you design everything.

You're building a distributed database. You want three things: every read returns the latest write, every request gets a response, and the system survives network failures. Sounds reasonable. Except in 2000, Eric Brewer stood up at a conference and said you can only have two.

This is the CAP theorem — arguably the most important idea in distributed systems. It was conjectured by Brewer and formally proved by Seth Gilbert and Nancy Lynch in 2002. It says that when a network partition happens, a distributed system must choose between consistency and availability. You cannot have both.

The theorem is often misunderstood, oversimplified, or hand-waved away. Let's actually understand what it means and why it matters for every system design decision you'll ever make.

01The impossible promise

Imagine you run two database servers, Node-1 in New York and Node-2 in London. A user writes balance = $100 to Node-1. One second later, another user reads from Node-2. What should they see?

If the network between New York and London is healthy, Node-1 can replicate the write to Node-2 before the read arrives. Everyone's happy. But what if the cable under the Atlantic gets cut? Now Node-2 has stale data, and you face a brutal decision:

Option A: Stay consistent

Node-2 refuses to answer reads until it can confirm it has the latest data. The user gets an error or a timeout. You preserved correctness but sacrificed availability.

Option B: Stay available

Node-2 answers with whatever data it has — possibly stale. The user gets a response, but it might be wrong. You preserved availability but sacrificed consistency.

There is no Option C. That's the CAP theorem.

02C, A, and P — what they actually mean

The three letters are precise terms, not vague goals. Getting them wrong is the #1 source of CAP confusion.

Figure 01 — The three guarantees

The CAP triangle. Each edge represents a pair of guarantees. Since partition tolerance is mandatory in any real distributed system, the practical choice is between CP and AP.

Consistency (C)

Every read receives the most recent write or an error. This is linearizability — the strictest form of consistency. If you write x = 5 and then read x, you must get 5, no matter which node you ask. It's as if there's a single copy of the data, even though there are multiple replicas.

Availability (A)

Every request to a non-failing node receives a response — no errors, no timeouts. The system always answers. Note: "non-failing" means the node itself is up. If a node is on fire, no one expects it to respond. But if it's healthy, it must answer every request.

Partition Tolerance (P)

The system continues to operate despite arbitrary message loss or delay between nodes. A network partition is when some nodes can't talk to each other — the network splits the cluster into isolated groups. Partition tolerance means the system doesn't just crash when this happens.

03Why partitions are not optional

Here's the key insight that clears up most CAP confusion: you don't get to choose whether partitions happen. They happen. Networks fail. Cables get cut. Switches die. Cloud regions lose connectivity. It's not a question of if, but when.

The "pick two out of three" framing is misleading. Since P is mandatory, the real question is: when a partition happens, do you choose C or A?

A system that doesn't tolerate partitions is just a single-node system. The moment you put data on more than one machine connected by a network, you've accepted that the network can fail. So P is always in the picture.

What about CA systems?
A CA system (consistent and available, but not partition-tolerant) is essentially a single-node database like a traditional PostgreSQL or MySQL instance. It's consistent and available — until the network (or the single node) fails. The moment you distribute it, you're in CP or AP territory.

04The real choice: CP or AP

Since partitions are inevitable, every distributed system makes a choice — sometimes explicitly, sometimes by accident — about what happens during a partition.

Figure 02 — What happens during a partition

During a network partition, a CP system refuses to answer rather than return stale data. An AP system always answers but may return outdated values. Neither is universally "better" — the right choice depends on what your application can tolerate.

05CP systems in practice

A CP system prioritizes correctness. During a partition, nodes that can't confirm they have the latest data will refuse requests rather than risk serving stale reads.

How it works: Most CP systems use a consensus protocol — like Raft or Paxos — where a majority of nodes must agree before a write is committed. If a partition isolates a minority of nodes, those nodes stop accepting writes. Reads from the minority may also be rejected to prevent stale results.

Real-world CP systems

ZooKeeper

Used for distributed coordination — leader election, config management, distributed locks. Correctness is everything here. A wrong answer from ZooKeeper could cause two nodes to both think they're the leader.

etcd / Consul

Service discovery and configuration stores. They use Raft consensus and will become unavailable to minority partitions rather than serve stale service registry data.

When to choose CP: when a wrong answer is worse than no answer. Financial transactions, distributed locks, leader election, inventory counts — anywhere that stale data leads to real damage.

06AP systems in practice

An AP system prioritizes responsiveness. Every healthy node answers every request, even if it might be serving slightly outdated data. When the partition heals, nodes reconcile their differences.

How it works: AP systems typically use eventual consistency. Writes are accepted by any node and asynchronously replicated. During a partition, different nodes may have different versions of the data. After the partition heals, a conflict resolution strategy — last-write-wins, vector clocks, or application-level merge — brings things back in sync.

Real-world AP systems

Cassandra

Distributed database designed for high availability. Every node can accept writes. Uses tunable consistency — you can dial between AP and CP on a per-query basis by adjusting quorum sizes.

DynamoDB

Amazon's key-value store, born from the Dynamo paper. Defaults to eventual consistency for reads (AP), but offers strongly consistent reads as an option (trading some availability).

When to choose AP: when no answer is worse than a slightly stale answer. Social media feeds, DNS lookups, shopping carts, caching layers, real-time analytics — places where showing slightly old data is far better than showing nothing.

07It's a spectrum, not a switch

The biggest misconception about CAP is that it's a permanent, binary choice. In reality, modern systems are far more nuanced:

Tunable consistency. Cassandra lets you set the consistency level per query. ONE means any single replica can answer (very AP). QUORUM means a majority must agree (more CP). ALL means every replica must respond (fully CP). You don't pick once — you pick per operation based on what that operation needs. The classic threshold: R + W > N guarantees linearizability, where R is the read quorum size, W the write quorum size, and N the replication factor.

Different guarantees for different data. A single application can use CP for payments and AP for user profile updates. The banking part of your app doesn't have to make the same trade-off as the notification feed.

Partitions are rare and temporary. CAP only kicks in during a partition. The rest of the time — which is the vast majority of the time — a well-designed system can be both consistent and available. The theorem constrains what happens during failures, not during normal operation.

Beyond CAP — the PACELC model
Daniel Abadi extended CAP with the PACELC model: if there's a Partition, choose A or C; Else (normal operation), choose Latency or Consistency. This captures the reality that even without partitions, there's a trade-off between how fast you respond and how consistent your data is. A system might be PA/EL (available during partitions, low latency normally — like Cassandra) or PC/EC (consistent always, even at the cost of latency — like ZooKeeper).

08The one-line takeaway

When the network splits, a distributed system must choose: refuse to answer (consistency) or answer with possibly stale data (availability). You can't have both at the same time.

That's CAP in one sentence. Everything else — tunable consistency, PACELC, eventual consistency, conflict resolution — is about navigating that fundamental constraint as gracefully as possible. The best systems don't blindly pick a side. They understand the trade-off and make different choices for different operations, giving you consistency where it matters and availability where it doesn't.

The next time someone says "we need a distributed database," the first question shouldn't be "which one?" It should be: "when the network fails, what's worse — a wrong answer or no answer?" That question, and how you answer it, shapes everything that follows.
