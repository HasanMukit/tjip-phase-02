The API gateway pipeline

What sits between a client and a server in any modern system: gateway → auth → rate limit → routing → load balancer → reverse proxy → server. Each step explained, and why the order matters.

DNS gave the client an IP. The packet leaves the laptop and arrives at — what, exactly? Almost never the application server itself. In any modern system the request first threads through a pipeline of edge components, each of which does one cross-cutting job: authenticate, throttle, route, balance, decrypt, compress, cache, then finally hand the request to a server. The server is the last thing the request touches, not the first.

This page walks the pipeline left-to-right, names each role, explains how the roles connect, and ends with the architectural punch line: the whole pipeline exists so the server can be small.

01Why a pipeline at all

A naive design puts the server at the public IP and lets the client talk to it directly. That works for a class project. It breaks for anything real, because every server then has to:

  • Hold its own TLS certificate and renew it.
  • Verify auth tokens on every request.
  • Count requests per user to enforce rate limits.
  • Decide whether the request belongs to it or to a different microservice.
  • Compress responses, cache hot results, log everything for observability.
  • And — separately — handle the actual business logic.

The pipeline takes those cross-cutting concerns off the server's plate and centralises them at the edge. Each stage does one thing; the server, at the end, can be a plain HTTP service that just answers questions.

02The shape of the pipeline

Figure 01 — The full pipeline

Top row: gates and dispatch. Bottom row: balancing, edge functions, then the server pool. The dashed box marks how much of the pipeline often runs in a single gateway process — the boundaries are conceptual, not physical.

Read the diagram top to bottom: a request enters the gateway, gets through auth and rate limit (the gates), gets routed to the right service, gets balanced across that service's instances, decrypted and possibly served from the proxy's cache, and only then arrives at a server.

These are roles, not necessarily separate boxes. In practice an API gateway is often one process that internally does auth + rate limit + routing, and the load balancer + reverse proxy are usually the same NGINX/Envoy/HAProxy instance. But conceptually each role is distinct, and large systems split them across separate tiers.

03API gateway — the front door

An API gateway is one public endpoint that fronts many backend services. It owns the cross-cutting concerns no individual service should re-implement: authentication, rate limiting, request shaping, observability, API versioning. Backends sit behind it as plain services.

The mental model: a hotel front desk. Every guest passes through it; the back-of-house rooms don't each need their own check-in. Common implementations: Kong, AWS API Gateway, Apigee, Envoy, Tyk, or NGINX with custom modules.

The gateway is the surface the API contract actually lives on — the URLs, verbs, headers, and status codes a client sees are the gateway's promises, even when individual services produce them. API design and API design best practices cover how that contract should be shaped; the pipeline below is how it's enforced.

04Auth and rate limit — the gates

Authentication and authorization

  • Authentication = "who are you?" Verifies identity. Mechanisms: API key, JWT, OAuth2 access token, mTLS client cert, session cookie.
  • Authorization = "what are you allowed to do?" Checks the authenticated identity against the requested action. Roles, scopes, ACLs, policy engines (OPA).

Token validation is usually stateless — verify a JWT signature locally, no database round-trip — which is part of why it's cheap to put at the edge. Anything that fails auth short-circuits here, before the rest of the pipeline costs anything.

Rate limiting

Rate limiting caps how many requests a given identity (user, API key, IP) can issue per window. Two reasons: fairness (one noisy tenant can't starve everyone else) and defense (cap the cost of abuse, scraping, DoS).

Token bucket

A bucket holds up to N tokens, refilling at rate R per second. Each request consumes one token; an empty bucket means rejection. Allows bursts up to N, sustains R over time.

Leaky bucket

Requests enter a fixed-size queue and drain out at a constant rate. Output is perfectly smooth no matter how spiky the input. Use when you need to protect a downstream that hates bursts.

Other algorithms: fixed-window (simple but has edge-of-window double-burst), sliding-window log (precise but expensive). Whatever the algorithm, the limit lives at the gateway because the gateway sees every request — putting it deeper means the limit only kicks in after you've already paid for routing.

Order matters
Auth first, rate limit second. Auth so that anonymous abusers can't even count against per-user limits; rate limit so that authenticated abusers are still bounded. Reversing the order means an attacker can trigger your rate-limit storage on every request without ever having to log in.

05Routing and load balancing — the dispatchers

These look similar from a distance and are often the same physical box, but they answer different questions.

Routing — which service?

Routing picks the right backend service based on the request: path (/api/users/* → users service), host (payments.example.com → payments service), header, method, even body shape. This is where a monolithic-looking external API decomposes into microservices internally.

Routing decisions can also be weighted — useful for canary deploys ("send 1% of traffic to v2").

Load balancing — which instance?

The load balancer spreads requests across the N instances of a single service chosen by routing. It comes in two main flavours.

Figure 02 — L4 vs L7 load balancing

L4 lives below the HTTP layer and is correspondingly faster and simpler. L7 understands HTTP, which is what makes URL-based routing, sticky sessions, and content-aware logic possible.

Algorithms within the LB: round-robin (rotate), least connections (pick the instance with the fewest active conns — adapts to real load), least response time, consistent hashing (sticky placement; ties back to the consistent-hashing page), weighted (give bigger boxes more traffic).

Health checks remove dead instances from the pool — without this the LB happily routes into a black hole.

06Reverse proxy — TLS, compression, cache

A reverse proxy sits in front of one or more backend servers and pretends to be the server to the client. The same physical box is often also the load balancer, but the role is different: load balancing is about spreading load; proxying is about wrapping the backend with edge functionality.

Wait — isn't LB before RP? Do I need one RP per server?
Common confusion. In the diagram, LB and RP are separate roles; in deployment, they're usually the same process (NGINX, Envoy, HAProxy all play both). One box receives the request, its LB function picks a backend, its proxy function wraps and forwards. So one RP fans out to N servers — no per-server proxy explosion. Multi-layer setups exist (a cloud L4 LB in front of an NGINX fleet; service-mesh sidecars next to each pod), but every layer still funnels-then-fans-out. The "LB before RP" arrow in the diagram is logical order, not physical sequence.

Three jobs always show up at the proxy layer:

TLS termination

The proxy holds the certificate and private key. It decrypts incoming HTTPS once, then talks plain HTTP (or re-encrypted HTTP) to the backend. (The handshake itself — how the certificate proves identity, how Diffie-Hellman delivers a shared secret, why TLS 1.3 is one round-trip — is laid out on the HTTPS & the TLS handshake page.)

  • Certificate management in one place — issue, renew, rotate at the edge instead of on every backend.
  • Backends don't pay TLS CPU — handshakes are expensive; do them once.
  • Routing by HTTP content becomes possible — you can't inspect or route by URL while the bytes are still encrypted.

Compression

Gzip / Brotli responses on the way out. Done at the edge so backends don't each implement it; saves bandwidth and tail latency for clients on slow links.

Caching

Repeat responses (especially GETs with cacheable headers) get served from the proxy without ever touching the backend. This is the same idea as DNS caching, one layer up: TTL-based pull, purge APIs for explicit invalidation, stale-while-revalidate to hide refresh latency. CDNs like Cloudflare, Fastly, and Akamai are reverse proxies turned global.

Other things the proxy commonly handles: HTTP/2 → HTTP/1 translation, request/response header rewriting, WebSocket upgrade handling, basic WAF rules.

07Forward proxy — the mirror image

The reverse proxy is one half of a symmetric pair. Its mirror is the forward proxy, which sits in front of clients rather than servers. Same idea — a middleman that wraps and translates traffic — but on the opposite side of the wire.

Where a reverse proxy stands in for the server (the client thinks it's talking to the server), a forward proxy stands in for the client (the server thinks it's talking to the proxy). What changes between the two is not what the box does — it's whose side it represents.

Figure 03 — Forward vs reverse proxy

The proxy is a middleman in both cases — what differs is which end it's pretending to be. A forward proxy speaks for the client; a reverse proxy speaks for the server.

Common reasons to run a forward proxy:

  • Corporate egress filtering. Every laptop on the network talks to the internet through a forward proxy that blocks known-bad domains, scans for malware, and enforces an acceptable-use policy. The classic on-premises web proxy (Squid, Blue Coat, Zscaler).
  • Egress caching. When 500 employees all download the same Ubuntu update, the proxy fetches it once and serves the rest from cache. Saves WAN bandwidth on networks where the bottleneck is the link, not the destination.
  • Audit and DLP. Logging every outbound request — and inspecting payloads for data-loss prevention — is much easier when one box sees everyone's traffic.
  • Anonymity / IP masking. The destination server sees the proxy's IP, not the client's. This is the entire premise of Tor, residential proxies, and VPNs in their proxy mode.
  • Geo-routing or unblocking. Routing through a proxy in another jurisdiction to reach content gated by IP geolocation.

The mnemonic that makes the names finally click:

Forward proxy hides the client. Reverse proxy hides the server.

In modern systems both kinds often coexist on the same request. Your laptop's traffic might leave a corporate network through a forward proxy, cross the public internet, and land at a destination's reverse proxy before reaching the actual server. The same software (NGINX, Envoy, HAProxy, Squid) can play either role — what makes it "forward" or "reverse" is whose side it stands on, not the binary itself.

08How the pieces interconnect

Worth pointing out the connections that aren't obvious from the picture:

  • Auth and rate limit are both gates. They cheap-fail before expensive work. Order: auth first, rate limit second (see callout above).
  • Routing and load balancing answer different questions. Routing: which service? Load balancing: which instance of that service? In tools like Envoy these are the same config language but semantically separate stages.
  • Reverse proxy and load balancer often live in the same process. NGINX, Envoy, HAProxy each play both roles. The terms describe what the box is doing for a given request, not where it runs.
  • Caching at the reverse proxy connects back to the resilience patterns. Single-flight, stale-while-revalidate, TTL jitter, and stampede protection all belong here — the proxy cache is exactly where Class 03's cache patterns get applied in real systems.
  • DNS sits outside the pipeline entirely. It ran before any of this — the pipeline starts only once the client has an IP. But the same TTL-and-pull idea recurs at the proxy cache layer, which is why we taught DNS first.

The whole pipeline exists so the server can be small.

By the time the request reaches the application, it's been authenticated, rate-checked, routed to the right service, sent to a healthy instance, decrypted, possibly compressed-on-the-way-out, and possibly served from cache before the server was even woken up. The server therefore doesn't need to know about TLS certs, JWT signing keys, gzip, or rate-limit storage. It just answers the question.

09The one-line takeaway

A modern request doesn't reach the server first — it threads through a pipeline of edge components that take cross-cutting concerns off the server's plate, and the server is the last thing to touch the bytes.

Once you can name the seven stages and what each one does, every architecture diagram in the industry becomes legible. "We put auth in front of rate limit," "we terminate TLS at the LB," "we cache at the CDN" — these are all moves on the same board. The pipeline is the board.
