Throughput

How many units of work a system finishes per second — and why raising that number is a fundamentally different game from making any single request faster.

Stand on an overpass and count cars for one minute. That's throughput — the rate at which a system finishes work. It says nothing about how long any single car's drive took. A system can have enormous throughput and terrible trip times (a clogged highway moving many cars slowly) or excellent trip times and trivial throughput (a single Ferrari on an empty road). Throughput is its own number, with its own rules.

This page is about that number alone: what it measures, the units it comes in, why it's gated by the slowest stage of your system, and the levers you have to raise it.

01What throughput is

Throughput is how many units of work the system finishes per unit of time. If a service handles 10,000 requests in 10 seconds, its throughput is 1,000 requests per second. That's it. It's a system-wide number, not a per-request one.

Figure 01 — Throughput is a counter, not a stopwatch

You don't time any one request. You pick a window, count what finished inside it, and divide. That's the number. It tells you nothing about what each individual request experienced — only the aggregate rate.

Notice what throughput doesn't tell you. Ten requests finished in one second could mean each took 100 ms and they ran one after the other — or each took 900 ms and ten ran in parallel — or anything in between. Latency and throughput are genuinely independent axes. You need both numbers before you understand a system.

02Units and variants

Throughput is reported in work-per-time, but "work" varies by context. Pick the unit that matches what the system produces:

Requests & queries

RPS (requests per second) for HTTP services, QPS for databases and search, TPS for transactional systems. One "unit" is one round-trip from the caller's point of view.

Messages & events

Messages per second for queues and streams (Kafka, SQS, Pub/Sub). Events per second for analytics pipelines. The unit is a single discrete record moving through the pipe.

Bytes

MB/s, GB/s for bulk data — disk I/O, network links, backup jobs, ETL. The unit is a byte, which matters when payloads vary wildly in size.

Business operations

Orders per minute, trades per second, payments per hour. Not always the same as RPS — one business operation may fan out to many internal calls, or many user actions may collapse into one.

Sustained vs peak

A system can handle a burst of 50,000 RPS for three seconds but only 5,000 RPS indefinitely. Both are throughput numbers, and confusing them kills systems at launch. Peak throughput is what you can briefly absorb — usually buffered by queues, caches, or slack capacity. Sustained throughput is what you can keep up with without falling behind. Capacity-plan on sustained; survive spikes with peak.

Goodput — the honest number

Raw throughput counts every response, including errors, retries, and duplicates. Goodput counts only useful work — successful, unique, non-retried responses. A service reporting "12,000 RPS" while 30% of those are client retries after timeouts is actually doing 8,400 useful RPS and drowning in its own failure amplification. When something gets slow, throughput and goodput diverge; that gap is a signal.

The first question to ask
When someone says "we handle X throughput," ask three follow-ups: what unit? sustained or peak? throughput or goodput? The answers usually reveal whether the number is load-bearing or cosmetic.

03The bottleneck rule

A pipeline's throughput is set by its slowest stage. Not the average. Not the sum. The slowest. If request processing flows through three stages — say parse → compute → write — and each can individually handle 1,000 / 500 / 2,000 RPS, the whole system tops out at 500 RPS. Upstream work piles up in front of the 500-RPS stage; downstream capacity sits idle.

Figure 02 — Throughput is gated by the slowest stage

Three stages, three capacities — but only one governs. You can double the parse stage or triple the write stage and system throughput doesn't move. The only lever that matters is the narrow stage.

This has a useful corollary: optimizing anywhere except the bottleneck produces zero throughput gain. Speeding up the already-fast parse stage is work that never touches the output rate. It's the single most common wasted effort in performance engineering — optimizing the part of the system that's easy to see rather than the part that's actually throttling.

Finding the bottleneck is usually not subtle once you measure. Watch the utilization of each stage. The bottleneck is saturated (CPU near 100%, queue full, connections exhausted) while everything downstream sits bored. Fix that one. Then repeat — because as soon as you do, a new stage becomes the bottleneck, and that's the next target.

Theory of Constraints, in one line
An improvement that doesn't move the bottleneck doesn't move the system. Find the slowest stage; invest everything there; when it's no longer the slowest, find the new one. Repeat until the system meets its goal or you run out of money.

04Scaling throughput

Once you know where the bottleneck is, there are a handful of levers for raising it. Each has its own cost, and each has failure modes when abused.

Parallelism — do more at once

Add more workers, threads, or processes at the bottleneck stage. If a request takes 200 ms of work and you run 50 of them in parallel, you can finish 250 RPS on one machine even though each individual request still takes 200 ms. This is the main lever in practice, and it's why most server frameworks are built around worker pools.

The catch: parallelism is only free until workers start contending for something shared — a database connection pool, a lock, a cache line. Once they do, adding more workers decreases throughput. You're paying coordination overhead for zero new work.

Horizontal scale — add more machines

If one box can do 500 RPS, ten boxes behind a load balancer can do ~5,000 RPS. Works cleanly for stateless services. Gets harder the more shared state is involved — eventually you need to shard: partition the data itself so each machine owns a slice of requests and they rarely talk to each other.

Batching — amortize overhead

If every request pays 1 ms of per-call overhead and you bundle 100 requests into one call, you've paid the overhead once for 100 units of work. Throughput jumps an order of magnitude. The trade-off is that each individual request now waits longer — batching trades latency for throughput, and that trade-off is covered in detail on the combined page.

Async and pipelining — overlap the waiting

If a request spends 180 of its 200 ms waiting on I/O, you don't need 50 threads to run 50 in parallel — you need one thread that can juggle them. Asynchronous I/O, event loops, and pipelining let a small pool of workers handle many in-flight requests by overlapping their waits. Same machine, same CPU; dramatically higher throughput for I/O-bound workloads.

Shed load before you drown

The last lever isn't about raising throughput — it's about protecting it. When offered load exceeds capacity, the worst response is to accept everything and slow down for all of it. Better: reject excess requests quickly (429 Too Many Requests, circuit breakers, token buckets) and keep serving the rest at full speed. A refused request is cheap; a queued-then-timed-out request wastes capacity twice.

Scaling throughput without sharding is a temporary fix
Parallelism and horizontal scale buy you orders of magnitude, but eventually shared state becomes the bottleneck. The big jumps — Netflix, Stripe, YouTube scale — come from partitioning data so most requests never leave their own shard. Every system that claims to "scale linearly" is really claiming to have found a good sharding key.

05The one-line takeaway

Throughput is the rate work completes. It's gated by the slowest stage, and the only lever that matters is the one pointed at that stage.

The two habits worth forming: always name the unit (RPS, QPS, MB/s, orders per minute — they're not interchangeable), and always find the bottleneck before you optimize. Nine times out of ten, the first instinct ("let's speed up the part I know") targets the wrong stage, and the tenth time you discover the bottleneck was never in the code at all — it was the database, the network, or the disk.

Throughput on its own is half the picture. The other half is latency — how long each individual request takes. And the two interact in ways that are their own subject: latency ⇄ throughput covers the trade-off, Little's Law, and why averages hide the tail.
